Knuth's Optimization

Reduces interval DP from O(n³) to O(n²) by exploiting the monotonicity of optimal split points.

Core Idea

In standard interval DP, we compute dp[i][j] = min over k in [i, j-1] of (dp[i][k] + dp[k+1][j] + cost(i, j)). This requires O(n) choices for k per state, giving O(n³) total.

Key Insight: If the cost function satisfies the quadrangle inequality, then the optimal split point opt[i][j] is monotone: opt[i][j-1] ≤ opt[i][j] ≤ opt[i+1][j]. This means as the interval grows, the optimal split point can only move right.

Conditions (Both Required)

  1. Quadrangle Inequality: For all a ≤ b ≤ c ≤ d: cost(a,c) + cost(b,d) ≤ cost(a,d) + cost(b,c)
  2. Monotonicity of opt: opt[i][j-1] ≤ opt[i][j] ≤ opt[i+1][j]

The second condition follows from the first in most practical cases, but you should verify it.

Intuition

Instead of trying all k in [i, j-1] for each (i, j), we only try k in [opt[i][j-1], opt[i+1][j]]. Summing over all states, the total work becomes O(n²) because the ranges telescope.

Standard O(n³) Interval DP

// Standard interval DP: O(n^3)
public int intervalDP(int[] arr) {
    int n = arr.length;
    int[][] dp = new int[n][n];
    int[] prefix = new int[n + 1];
    for (int i = 0; i < n; i++)
        prefix[i + 1] = prefix[i] + arr[i];

    // cost(i, j) = prefix[j+1] - prefix[i] (sum of elements)
    for (int len = 2; len <= n; len++) {
        for (int i = 0; i + len - 1 < n; i++) {
            int j = i + len - 1;
            dp[i][j] = Integer.MAX_VALUE;
            for (int k = i; k < j; k++) { // O(n) choices
                dp[i][j] = Math.min(dp[i][j],
                    dp[i][k] + dp[k + 1][j] + prefix[j + 1] - prefix[i]);
            }
        }
    }
    return dp[0][n - 1];
}

Knuth-Optimized O(n²) Version

// Knuth's Optimization: O(n^2)
public int knuthOptimized(int[] arr) {
    int n = arr.length;
    int[][] dp = new int[n][n];
    int[][] opt = new int[n][n]; // optimal split point
    int[] prefix = new int[n + 1];
    for (int i = 0; i < n; i++) {
        prefix[i + 1] = prefix[i] + arr[i];
        opt[i][i] = i; // base: single element, split at itself
    }

    for (int len = 2; len <= n; len++) {
        for (int i = 0; i + len - 1 < n; i++) {
            int j = i + len - 1;
            dp[i][j] = Integer.MAX_VALUE;

            // KEY: only search k in [opt[i][j-1], opt[i+1][j]]
            int lo = opt[i][j - 1];
            int hi = (j < n - 1) ? opt[i + 1][j] : j - 1;

            for (int k = lo; k <= Math.min(hi, j - 1); k++) {
                int val = dp[i][k] + dp[k + 1][j] + prefix[j + 1] - prefix[i];
                if (val < dp[i][j]) {
                    dp[i][j] = val;
                    opt[i][j] = k;
                }
            }
        }
    }
    return dp[0][n - 1];
}

How to Verify the Condition

Practical check: Run the O(n³) version on small inputs and record opt[i][j]. Print the opt table and visually verify monotonicity: each row should be non-decreasing left to right, each column non-decreasing top to bottom.

Network Flow (Max Flow / Min Cut)

Core Concept

Max-Flow Min-Cut Theorem: The maximum flow from source s to sink t equals the minimum capacity of any s-t cut.

Intuition: Imagine water flowing through pipes. The max flow is limited by the "bottleneck" — the minimum capacity cut that separates source from sink. No matter how you route the water, you cannot exceed this bottleneck.

Ford-Fulkerson Method

  1. Start with zero flow
  2. While there exists an augmenting path from s to t in the residual graph:
  3.    Find the path (BFS for Edmonds-Karp variant)
  4.    Push maximum possible flow along this path
  5.    Update residual capacities (forward edges decrease, backward edges increase)

Edmonds-Karp (BFS-based Ford-Fulkerson): O(VE²) — using BFS guarantees shortest augmenting paths, giving polynomial time.

Edmonds-Karp Implementation (Java)

class MaxFlow {
    int[][] capacity;  // capacity[u][v] = edge capacity
    int[][] flow;      // flow[u][v] = current flow
    int n;

    MaxFlow(int n) {
        this.n = n;
        capacity = new int[n][n];
        flow = new int[n][n];
    }

    void addEdge(int u, int v, int cap) {
        capacity[u][v] += cap; // support multiple edges
    }

    int maxflow(int s, int t) {
        int totalFlow = 0;
        while (true) {
            // BFS to find augmenting path
            int[] parent = new int[n];
            Arrays.fill(parent, -1);
            parent[s] = s;
            Queue<Integer> queue = new LinkedList<>();
            queue.offer(s);

            while (!queue.isEmpty() && parent[t] == -1) {
                int u = queue.poll();
                for (int v = 0; v < n; v++) {
                    // residual capacity = capacity - current flow
                    if (parent[v] == -1 && capacity[u][v] - flow[u][v] > 0) {
                        parent[v] = u;
                        queue.offer(v);
                    }
                }
            }

            if (parent[t] == -1) break; // no augmenting path

            // Find bottleneck
            int pathFlow = Integer.MAX_VALUE;
            for (int v = t; v != s; v = parent[v]) {
                int u = parent[v];
                pathFlow = Math.min(pathFlow, capacity[u][v] - flow[u][v]);
            }

            // Update flow along the path
            for (int v = t; v != s; v = parent[v]) {
                int u = parent[v];
                flow[u][v] += pathFlow;  // forward edge: add flow
                flow[v][u] -= pathFlow;  // backward edge: allow "undo"
            }

            totalFlow += pathFlow;
        }
        return totalFlow;
    }
}

Finding the Min Cut

After computing max flow: Run BFS from source on the residual graph (edges with remaining capacity > 0). All reachable nodes form set S, unreachable nodes form set T. The edges crossing from S to T with full capacity constitute the minimum cut.

// Find min cut after running maxflow
boolean[] minCut(int s) {
    boolean[] reachable = new boolean[n];
    Queue<Integer> queue = new LinkedList<>();
    queue.offer(s);
    reachable[s] = true;
    while (!queue.isEmpty()) {
        int u = queue.poll();
        for (int v = 0; v < n; v++) {
            if (!reachable[v] && capacity[u][v] - flow[u][v] > 0) {
                reachable[v] = true;
                queue.offer(v);
            }
        }
    }
    return reachable; // reachable = S side of min cut
}

Bipartite Matching

Find the maximum number of edges in a bipartite graph such that no vertex is matched more than once.

Hopcroft-Karp Algorithm

Faster than naive augmenting path approach: O(E√V) vs O(VE).

Key Idea: Instead of finding augmenting paths one at a time, Hopcroft-Karp finds a maximal set of shortest augmenting paths simultaneously using BFS layering, then augments all of them via DFS. Each phase takes O(E) and there are at most O(√V) phases.

Algorithm Steps

  1. BFS Phase: Build a layered graph from all free (unmatched) left vertices. Stop when we reach free right vertices or no more progress.
  2. DFS Phase: For each free left vertex, find an augmenting path through the layered graph. Flip matched/unmatched edges along the path.
  3. Repeat until BFS finds no augmenting path.

Full Implementation (Java)

class HopcroftKarp {
    int leftSize, rightSize;
    List<List<Integer>> adj; // adj[left] = list of right neighbors
    int[] matchL, matchR;    // matchL[l] = matched right, matchR[r] = matched left
    int[] dist;              // BFS layer distance for left nodes
    static final int INF = Integer.MAX_VALUE;

    HopcroftKarp(int left, int right) {
        leftSize = left;
        rightSize = right;
        adj = new ArrayList<>();
        for (int i = 0; i < left; i++) adj.add(new ArrayList<>());
        matchL = new int[left];
        matchR = new int[right];
        dist = new int[left];
        Arrays.fill(matchL, -1);
        Arrays.fill(matchR, -1);
    }

    void addEdge(int l, int r) {
        adj.get(l).add(r);
    }

    // BFS: build layered graph from free left vertices
    boolean bfs() {
        Queue<Integer> queue = new LinkedList<>();
        for (int l = 0; l < leftSize; l++) {
            if (matchL[l] == -1) {
                dist[l] = 0;
                queue.offer(l);
            } else {
                dist[l] = INF;
            }
        }
        boolean found = false;
        while (!queue.isEmpty()) {
            int l = queue.poll();
            for (int r : adj.get(l)) {
                int nextL = matchR[r]; // who is r matched to?
                if (nextL == -1) {
                    found = true; // found free right vertex = augmenting path exists
                } else if (dist[nextL] == INF) {
                    dist[nextL] = dist[l] + 1;
                    queue.offer(nextL);
                }
            }
        }
        return found;
    }

    // DFS: find augmenting path through layered graph
    boolean dfs(int l) {
        for (int r : adj.get(l)) {
            int nextL = matchR[r];
            if (nextL == -1 || (dist[nextL] == dist[l] + 1 && dfs(nextL))) {
                matchL[l] = r;
                matchR[r] = l;
                return true;
            }
        }
        dist[l] = INF; // remove from layered graph (optimization)
        return false;
    }

    int maxMatching() {
        int result = 0;
        while (bfs()) { // each BFS phase = O(E)
            for (int l = 0; l < leftSize; l++) {
                if (matchL[l] == -1 && dfs(l)) {
                    result++;
                }
            }
        }
        return result;
    }
}

König's Theorem

In any bipartite graph: Maximum Matching = Minimum Vertex Cover. This means the minimum number of vertices needed to "cover" all edges equals the size of the maximum matching.

Corollary: Maximum Independent Set = Total Vertices - Maximum Matching (in bipartite graphs).

Applications

  • Task Assignment: Assign workers to tasks (each worker does at most one task)
  • Scheduling: Schedule jobs on machines without conflicts
  • Tiling: Domino tiling of a grid (each cell = node, adjacent cells = edge)
  • Minimum Vertex Cover: Via König's theorem

Hungarian Algorithm (Weighted Bipartite Matching)

Finds the minimum cost (or maximum profit) perfect matching in a weighted bipartite graph. O(n³) for an n×n assignment.

Problem Statement

Given n workers and n jobs with a cost matrix cost[i][j] = cost of assigning worker i to job j, find the assignment that minimizes total cost.

Building Intuition Step by Step

Step 1: Labels (Dual Variables)

Assign a "label" to each worker and each job:

  • labelWorker[x] = potential of worker x
  • labelJob[y] = potential of job y

Invariant (feasibility): For all x, y: labelWorker[x] + labelJob[y] ≤ cost[x][y]

This means labels are always a lower bound on the true cost of any assignment.

Step 2: Equality Graph

The equality graph contains only "tight" edges where labelWorker[x] + labelJob[y] == cost[x][y].

Key Theorem: If we can find a perfect matching using ONLY edges in the equality graph, that matching is optimal! Because the sum of labels = sum of costs along those edges, and labels are a lower bound for any other assignment.

Step 3: Growing the Equality Graph

If the current equality graph doesn't have a perfect matching, we adjust labels to add new tight edges:

  • Find delta = minimum slack among edges not yet in the equality graph
  • Decrease worker labels in the alternating tree by delta
  • Increase job labels in the alternating tree by delta
  • This makes at least one new edge tight (added to equality graph)

Step 4: The Sentinel Trick (Index 0)

Why does the code use index 0 as a sentinel?

jobMatch[0] = i means "we are currently trying to match worker i." Index 0 is a virtual "unmatched" state. When we trace the augmenting path backward, hitting job 0 means we've reached the starting point. The final answer accumulates into -labelJob[0] because every time we adjust labels, the sentinel absorbs the delta that doesn't get applied to a real job.

The Main Loop Explained

// For each worker i:
//   Start by "assigning" worker i to virtual job 0
//   Then repeatedly:
//     1. Find the job with minimum slack (closest to being tight)
//     2. If slack > 0: adjust labels (dual update) to make it tight
//     3. If that job is unmatched: augment! (flip the alternating path)
//        If that job is matched: add its worker to tree, continue

Full Annotated Implementation (Minimum Cost)

/**
 * Hungarian Algorithm for minimum cost assignment.
 * Workers: 1..n, Jobs: 1..n (0 is sentinel)
 * cost[i][j] = cost of assigning worker i to job j (1-indexed)
 * Returns minimum total assignment cost.
 */
public static long hungarian(int[][] cost, int n) {
    // Labels (dual variables): labelWorker[x] + labelJob[y] <= cost[x][y]
    long[] labelWorker = new long[n + 1]; // potential for each worker
    long[] labelJob = new long[n + 1];    // potential for each job

    // jobMatch[j] = worker assigned to job j (0 = unassigned)
    int[] jobMatch = new int[n + 1];

    for (int i = 1; i <= n; i++) { // for each worker
        // === Start augmenting from worker i ===
        // Trick: "assign" worker i to sentinel job 0
        jobMatch[0] = i;

        // inSetT[j] = true if job j is already in the alternating tree
        boolean[] inSetT = new boolean[n + 1];

        // minSlack[j] = minimum (labelWorker[x] + labelJob[j] - cost[x][j])
        //               over all workers x currently in the tree
        // This tracks how close job j is to becoming a tight edge
        long[] minSlack = new long[n + 1];
        Arrays.fill(minSlack, Long.MAX_VALUE);

        // replaceBy[j] = the worker that achieves minSlack[j]
        // Used to trace back the augmenting path
        int[] replaceBy = new int[n + 1];

        int currentJob = 0; // start from sentinel

        do {
            inSetT[currentJob] = true;
            int currentWorker = jobMatch[currentJob];
            long delta = Long.MAX_VALUE;
            int nextJob = -1;

            // Phase 1: Update minimum slack for all jobs not in tree
            for (int j = 1; j <= n; j++) {
                if (inSetT[j]) continue;
                // Slack = how far this edge is from being tight
                long slack = (long)cost[currentWorker][j]
                           - labelWorker[currentWorker] - labelJob[j];
                if (slack < minSlack[j]) {
                    minSlack[j] = slack;
                    replaceBy[j] = currentJob; // trace: came from currentJob
                }
                if (minSlack[j] < delta) {
                    delta = minSlack[j];
                    nextJob = j; // job with smallest slack
                }
            }

            // Phase 2: Adjust labels (dual update)
            // For jobs IN tree: increase label (tighter constraint)
            // For workers IN tree (via their jobs): decrease label
            // Net effect: makes nextJob's edge tight
            for (int j = 0; j <= n; j++) {
                if (inSetT[j]) {
                    // Jobs in tree: increase their label
                    labelWorker[jobMatch[j]] -= delta;
                    labelJob[j] += delta;
                } else {
                    // Jobs not in tree: reduce their slack
                    minSlack[j] -= delta;
                }
            }

            // Phase 3: Move to nextJob
            currentJob = nextJob;

        } while (jobMatch[currentJob] != 0);
        // Loop ends when nextJob is unmatched (augmenting path found!)

        // Trace back and flip the augmenting path
        while (currentJob != 0) {
            int prevJob = replaceBy[currentJob];
            jobMatch[currentJob] = jobMatch[prevJob];
            currentJob = prevJob;
        }
    }

    // The answer accumulates into -labelJob[0]
    // Why? Every dual update adds delta to labelJob[0] (sentinel is always in tree)
    // The sum of all deltas = total assignment cost
    return -labelJob[0];
}

Why the Answer is -labelJob[0]

The sentinel absorbs cost: Job 0 is always in the tree (inSetT[0] = true from the start). Every dual update adds delta to labelJob[0]. Since labelJob[0] starts at 0 and only increases, the total accumulated value equals the negative of the minimum assignment cost. Hence the answer is -labelJob[0].

Maximum Profit Version

For maximum profit assignment, negate all costs: cost'[i][j] = -cost[i][j], then run min-cost Hungarian. Or equivalently, flip the inequality direction and sign conventions:

/**
 * Hungarian Algorithm for MAXIMUM profit assignment.
 * profit[i][j] = profit of assigning worker i to job j (1-indexed)
 * Returns maximum total assignment profit.
 */
public static long hungarianMaxProfit(int[][] profit, int n) {
    // For max profit: labels satisfy labelWorker[x] + labelJob[y] >= profit[x][y]
    // Initialize labelWorker[x] = max(profit[x][*]), labelJob[y] = 0
    long[] labelWorker = new long[n + 1];
    long[] labelJob = new long[n + 1];
    int[] jobMatch = new int[n + 1];

    // Initialize: each worker's label = max profit they can achieve
    for (int i = 1; i <= n; i++) {
        labelWorker[i] = Long.MIN_VALUE;
        for (int j = 1; j <= n; j++)
            labelWorker[i] = Math.max(labelWorker[i], profit[i][j]);
    }

    for (int i = 1; i <= n; i++) {
        jobMatch[0] = i;
        boolean[] inSetT = new boolean[n + 1];
        long[] minSlack = new long[n + 1];
        Arrays.fill(minSlack, Long.MAX_VALUE);
        int[] replaceBy = new int[n + 1];
        int currentJob = 0;

        do {
            inSetT[currentJob] = true;
            int w = jobMatch[currentJob];
            long delta = Long.MAX_VALUE;
            int nextJob = -1;

            for (int j = 1; j <= n; j++) {
                if (inSetT[j]) continue;
                // Slack for max: labelWorker[w] + labelJob[j] - profit[w][j]
                long slack = labelWorker[w] + labelJob[j] - profit[w][j];
                if (slack < minSlack[j]) {
                    minSlack[j] = slack;
                    replaceBy[j] = currentJob;
                }
                if (minSlack[j] < delta) {
                    delta = minSlack[j];
                    nextJob = j;
                }
            }

            // Dual update: decrease worker labels, increase job labels
            for (int j = 0; j <= n; j++) {
                if (inSetT[j]) {
                    labelWorker[jobMatch[j]] -= delta;
                    labelJob[j] += delta;
                } else {
                    minSlack[j] -= delta;
                }
            }

            currentJob = nextJob;
        } while (jobMatch[currentJob] != 0);

        while (currentJob != 0) {
            int prevJob = replaceBy[currentJob];
            jobMatch[currentJob] = jobMatch[prevJob];
            currentJob = prevJob;
        }
    }

    // For max profit, answer is labelJob[0] (positive accumulation)
    return labelJob[0];
}

Complexity Analysis

O(n³): n workers × (at most n label adjustments per worker × n jobs scanned per adjustment). The inner do-while loop runs at most n times because each iteration either adds a job to the tree or finds an augmenting path.

Minimum Cost Maximum Flow (MCMF)

Combines shortest path algorithms with flow augmentation to find the maximum flow with minimum total cost.

When to Use

  • Assignment problems with capacities (one worker can do multiple jobs up to a limit)
  • Problems where you need both maximum throughput AND minimum cost
  • Complex matching with constraints that don't fit pure bipartite matching

Core Idea: Repeatedly find the shortest path (minimum cost augmenting path) from source to sink using SPFA (or Bellman-Ford), then push flow along it. Each augmentation increases flow by at least 1 while keeping total cost minimal.

Algorithm

  1. Build residual graph with costs (forward edge has cost c, backward edge has cost -c)
  2. Find shortest path from s to t using SPFA (handles negative edges from backward edges)
  3. Push maximum flow along this path
  4. Update residual graph
  5. Repeat until no path exists

Simplified SPFA-based MCMF (Java)

class MCMF {
    static final int INF = Integer.MAX_VALUE;
    int n;
    List<int[]>[] graph; // each edge: [to, cap, cost, reverseIdx]

    MCMF(int n) {
        this.n = n;
        graph = new ArrayList[n];
        for (int i = 0; i < n; i++) graph[i] = new ArrayList<>();
    }

    void addEdge(int u, int v, int cap, int cost) {
        graph[u].add(new int[]{v, cap, cost, graph[v].size()});
        graph[v].add(new int[]{u, 0, -cost, graph[u].size() - 1});
    }

    // Returns [maxFlow, minCost]
    int[] solve(int s, int t) {
        int totalFlow = 0, totalCost = 0;
        while (true) {
            // SPFA to find shortest path (min cost)
            int[] dist = new int[n];
            Arrays.fill(dist, INF);
            dist[s] = 0;
            boolean[] inQueue = new boolean[n];
            int[] prevNode = new int[n];
            int[] prevEdge = new int[n];
            Arrays.fill(prevNode, -1);

            Queue<Integer> queue = new LinkedList<>();
            queue.offer(s);
            inQueue[s] = true;

            while (!queue.isEmpty()) {
                int u = queue.poll();
                inQueue[u] = false;
                for (int idx = 0; idx < graph[u].size(); idx++) {
                    int[] e = graph[u].get(idx);
                    int v = e[0], cap = e[1], cost = e[2];
                    if (cap > 0 && dist[u] + cost < dist[v]) {
                        dist[v] = dist[u] + cost;
                        prevNode[v] = u;
                        prevEdge[v] = idx;
                        if (!inQueue[v]) {
                            queue.offer(v);
                            inQueue[v] = true;
                        }
                    }
                }
            }

            if (dist[t] == INF) break; // no more augmenting paths

            // Find bottleneck along shortest path
            int pathFlow = INF;
            for (int v = t; v != s; v = prevNode[v])
                pathFlow = Math.min(pathFlow, graph[prevNode[v]].get(prevEdge[v])[1]);

            // Update flow and residual
            for (int v = t; v != s; v = prevNode[v]) {
                int[] e = graph[prevNode[v]].get(prevEdge[v]);
                e[1] -= pathFlow; // reduce forward capacity
                graph[v].get(e[3])[1] += pathFlow; // increase backward capacity
            }

            totalFlow += pathFlow;
            totalCost += pathFlow * dist[t];
        }
        return new int[]{totalFlow, totalCost};
    }
}

  2. "What's the theoretical best approach?" — For assignment-type problems, mentioning Hungarian shows you know the optimal algorithm exists.

Name-Dropping Guide

Safe to mention: "This is essentially a bipartite matching problem. The optimal algorithm is Hopcroft-Karp at O(E√V), but for these constraints, a simple DFS-based approach works."

Don't over-engineer: If the interviewer gives you n ≤ 15 and it's an assignment problem, just use bitmask DP. Saying "I'll implement Hungarian" for a tiny problem signals poor judgment about time management.

The Decision Framework

Is it an assignment/matching problem? └— n ≤ 20? → Bitmask DP (interview standard) └— n ≤ 1000, unweighted? → Augmenting path DFS └— n large, unweighted? → Hopcroft-Karp └— n large, weighted? → Hungarian └— Capacities > 1? → MCMF or Flow Is it an interval DP that's too slow? └— Cost satisfies quadrangle inequality? → Knuth's Optimization └— DP is convex? → Divide & Conquer optimization └— Transitions linear? → Convex Hull Trick

What Competitive Programmers Know That Interviewees Don't

  • Modeling skill: Reducing a problem to max-flow or bipartite matching is the hard part, not the implementation.
  • Duality: Max matching = Min vertex cover (König), Max flow = Min cut. These let you solve "minimum" problems by solving "maximum" problems.
  • Template library: In competitive programming, you pre-write these algorithms. In interviews, you shouldn't need them.

Ford-Johnson (Merge-Insertion Sort)

Minimizes the number of comparisons to sort n elements. Achieves the information-theoretic lower bound ceil(log2(n!)) for many values of n. Originally motivated by AtCoder Library Practice Contest comparison-counting problems.

Core Idea

  1. Pair elements and compare within each pair (n/2 comparisons)
  2. Recursively sort the "winners" (larger of each pair)
  3. Insert "losers" into the sorted sequence using binary search, following Jacobsthal number order to minimize worst-case binary search depth

Why Jacobsthal order? Inserting in order 1, 3, 2, 5, 4, 11, 10, 9, 8, 7, 6, ... ensures each insertion's binary search range is at most 2^k - 1 (a power of 2 minus 1), which is the optimal range for binary search to use exactly k comparisons.

Java Implementation

import java.util.*;

public class FordJohnson {
    private int compareCount;

    public static int sort(int[] nums) {
        FordJohnson fj = new FordJohnson();
        fj.mergeInsertionSort(nums, 1);
        return fj.compareCount;
    }

    private void mergeInsertionSort(int[] nums, int level) {
        if (level << 1 > nums.length) return;
        int n = nums.length, pair = level << 1;

        // Step 1: Compare pairs, put larger in first half
        for (int i = 0; i + pair <= n; i += pair) {
            int j = i + level;
            if (lessThan(nums[j + level - 1], nums[i + level - 1]))
                swap(nums, i, j, level);
        }

        // Step 2: Recursively sort winners
        mergeInsertionSort(nums, level << 1);

        // Step 3: Build main chain + pending, insert in Jacobsthal order
        List<int[]> mainChain = new ArrayList<>(), pending = new ArrayList<>();
        for (int i = 0, label = 1; i + level <= n; i += pair, label++) {
            int[] loser = Arrays.copyOfRange(nums, i, i + level);
            (mainChain.isEmpty() ? mainChain : pending).add(loser);
            if (i + pair <= n)
                mainChain.add(Arrays.copyOfRange(nums, i + level, i + pair));
        }
        jacobsthalInsert(mainChain, pending);

        // Write back
        int pos = 0;
        for (int[] elem : mainChain)
            for (int v : elem) nums[pos++] = v;
    }

    private void jacobsthalInsert(List<int[]> chain, List<int[]> pending) {
        // Jacobsthal sequence: 1, 3, 5, 11, 21, 43, ...
        // Insert pending elements in groups bounded by Jacobsthal numbers
        Deque<int[]> group = new LinkedList<>();
        int jacobsthal = 3, sign = -1;
        for (int i = 0, label = 1; i < pending.size(); ) {
            while (i < pending.size() && label <= jacobsthal) {
                group.add(pending.get(i)); i++; label++;
            }
            // Insert group in reverse order (binary search into chain)
            while (!group.isEmpty()) {
                int[] current = group.pollLast();
                int lo = 0, hi = chain.size();
                while (lo < hi) {
                    int mid = (lo + hi) >>> 1;
                    if (lessThan(chain.get(mid)[level(current) - 1], current[current.length - 1]))
                        lo = mid + 1;
                    else hi = mid;
                }
                chain.add(lo, current);
            }
            jacobsthal = (jacobsthal << 1) + sign;
            sign *= -1;
        }
    }

    private boolean lessThan(int a, int b) {
        compareCount++;
        return a < b;
    }

    private void swap(int[] nums, int i, int j, int len) {
        for (int k = 0; k < len; k++) {
            int tmp = nums[i + k]; nums[i + k] = nums[j + k]; nums[j + k] = tmp;
        }
    }
}
