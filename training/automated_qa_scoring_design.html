Automated QA Scoring Overview
What it does, why it matters, and what interviewers evaluate.

The Old Way vs The New Way

Traditionally, a QA manager listens to ~20 random calls per agent per month and scores them on a rubric. This means only 2-5% of conversations are ever reviewed — riddled with sampling bias, expensive in human hours, and feedback arrives weeks late.

Automated QA scores every single conversation within minutes of it ending, across multiple quality dimensions. Every agent gets complete, timely, objective feedback.

The Core Idea

This is test automation for human conversations. Just as CI runs assertions against every code commit, this system runs quality checks against every conversation — some deterministic (rules), some judgment-based (LLM-as-judge).

What Gets Scored

Quality Dimensions

  • Compliance: did the agent verify identity, read required disclosures?
  • Tone: was the agent empathetic and professional?
  • Resolution: was the customer's issue actually solved?
  • Efficiency: unnecessary transfers, excessive hold time?
  • Knowledge accuracy: did the agent give correct information?

The Cost Bomb

$3.6M/year in raw LLM cost is a non-starter. The entire scaling story of this system is cost optimization — batching, caching similar conversations, and tiered scoring (cheap model for obvious cases, expensive only for ambiguous ones). Bring this up early; it's the defining constraint.

"At 500K conversations a day, naive LLM scoring costs about $3.6M a year. So cost isn't an afterthought — it's the central design constraint. I'll batch requests, cache scores for near-duplicate conversations, and use a tiered approach where an expensive model only handles the genuinely ambiguous cases."

Why Rules and ML Run in Parallel

Rules are fast (<50ms) and local; ML is slow (~1s) and remote. Run them concurrently, not sequentially — the aggregator waits for both, so total latency is max(rule, ML), not sum. And if ML is down, rules still produce a compliance verdict immediately.

Scoring Dimensions Design
The critical distinction: deterministic rules vs probabilistic ML.

Two Fundamentally Different Score Types

Rule-Based (Deterministic)

  • Binary pass/fail
  • High precision, zero false positives
  • Auditable — "you didn't verify ID at 0:45"
  • Used for compliance (must be defensible)

ML-Based (Probabilistic)

  • Score 1-5, nuanced
  • Captures tone, empathy, clarity
  • Tolerates some noise
  • Used for coaching (improvement, not punishment)

The Design Decision That Wins Points

Don't use ML for everything. Compliance MUST be a rule — a probabilistic "87% likely the agent verified identity" is legally useless. You need a defensible binary answer with evidence. Reserve ML for the subjective dimensions where nuance matters and some noise is acceptable.

Rule Engine Deep Dive
Config-driven rules that non-engineers can change without a deploy.

Why Configurable Matters

Compliance requirements change constantly — new regulations, new products, new scripts. If every rule change needs an engineer and a deploy, the system can't keep up. Rules must be data, not code.

Rule Format (Config-Driven)

{
  "rule_id": "identity_verification",
  "name": "Customer Identity Verification",
  "description": "Agent must verify identity within first 2 minutes",
  "condition": {
    "type": "pattern_match",
    "speaker": "agent",
    "patterns": ["verify your identity", "confirm your name",
                 "date of birth", "last four digits"],
    "time_window": { "from_sec": 0, "to_sec": 120 }
  },
  "severity": "critical",
  "score_impact": { "pass": 0, "fail": -100 },
  "applies_to": ["billing", "account_changes"],
  "version": 3,
  "active": true
}

Rule Versioning & A/B Testing

Rules are immutable versions, never edited in place. This gives you a full audit trail (which rule version scored this call?) and lets you A/B test a new rule against the old one before full rollout. A non-engineer edits rules through an admin UI with dropdowns — never raw JSON.

Evidence Is Non-Negotiable

Every rule result stores the exact transcript excerpt and timestamp. When an agent disputes "you didn't verify identity," you show them 0:45 in the transcript. Without evidence, the score is unauditable and agents won't trust it.

LLMs Fail Confidently, Not Gracefully

A subtly-wrong transcript produces a confidently-wrong score, which produces wrong coaching. Never trust a single LLM judgment blindly. Mitigate with confidence scoring, human review of edge cases, and the calibration loop (Section 9).

Cost Optimization (The Whole Game)

Three Levers to Kill the $3.6M Bill

  • Batch: score 10 conversations in one API call — amortizes fixed overhead
  • Cache: near-duplicate conversations (same topic, same flow) get cached scores — contact centers have huge repetition
  • Tiered: a cheap model handles obvious cases (clear 1 or clear 5); the expensive model only judges ambiguous middle (2-4)
Tiered Scoring in Practice

Run a cheap, fast classifier first. If it's highly confident the call is clearly great or clearly bad, accept that score. Only escalate the uncertain 20-30% to the expensive model. This alone can cut LLM cost by 60-70%.

Score Aggregation & Scorecards
From per-conversation scores to per-agent scorecards.

Per-Conversation Score

Combine rule results and ML scores into one number with a weighted formula:

conversation_score = 0.40 * compliance_score   (from rules)
                   + 0.30 * quality_score      (from ML)
                   + 0.30 * efficiency_score   (hybrid)

// CRITICAL OVERRIDE: any failed critical rule caps the total.
if anyCriticalRuleFailed {
    conversation_score = min(conversation_score, 50)  // cap at 50%
}
The Critical-Rule Cap

A conversation where the agent was warm, efficient, and helpful but forgot to verify identity cannot score well — that's a compliance failure regardless of everything else. The cap enforces this: critical rules are gates, not just weighted factors.

Statistical Significance

Don't Rank on Tiny Samples

An agent with 5 conversations shouldn't be ranked against one with 200 — the small sample is noise. Require a minimum (say 30 conversations) before generating a reliable scorecard or percentile ranking. Otherwise a single bad call tanks a new agent unfairly.

Coaching Recommendations

Auto-generate from pattern analysis: "Agent X scores low on empathy specifically on billing calls — recommend empathy training for billing scenarios." The value isn't the score; it's the actionable, specific next step.

Alert Fatigue Prevention

Too Many Alerts = No Alerts

If supervisors get 200 alerts a day, they ignore all of them. Group related alerts (5 agents missed the same new disclaimer → one alert about the disclaimer, not five). Support snoozing and de-duplication. An alert system nobody reads is worse than none.

Compliance Audit Trail

Every compliance alert and its resolution is logged immutably. When a regulator asks "how do you ensure agents verify identity?", you show the rule, the 100% coverage, the violation alerts, and the remediation — a complete, defensible record.

Calibration & Drift Detection
Keeping LLM judges accurate as they drift over time.

The Drift Problem

LLM judges drift — model updates, prompt tweaks, or subtle distribution shifts change how they score. A "4/5 empathy" today might be a "3/5" next month for identical behavior. Undetected, this corrupts every downstream scorecard and coaching decision.

The Human Calibration Loop

Ongoing
Maintain a golden set
A fixed set of conversations with expert human-assigned scores — the ground truth.
Periodic
Re-score the golden set
Run the LLM judge against the golden set regularly and compare to human scores.
On divergence
Humans win, re-tune the judge
When agreement drops below threshold, the human scores are truth — re-tune the prompt/model until agreement stabilizes.
"Humans Win, The Judge Gets Re-Tuned"

This one line captures the whole philosophy. The LLM is a scalable approximation of human judgment, continuously validated against real human judgment. When they disagree, humans are the ground truth and the judge is corrected — never the reverse.

A/B Testing Scoring Models

Before rolling out a new judge model, run it in shadow mode against the current one on live traffic. Compare their scores and their agreement with the golden set. Only promote the new model if it matches humans better than the incumbent.

Score Schema

CREATE TABLE conversation_scores (
    id                UUID PRIMARY KEY,
    conversation_id   UUID NOT NULL,
    tenant_id         UUID NOT NULL,
    agent_id          UUID NOT NULL,
    scored_at         TIMESTAMPTZ NOT NULL,
    overall_score     REAL,
    compliance_passed BOOLEAN,       -- did all critical rules pass?
    dimensions        JSONB,          -- per-dimension scores + evidence
    rule_version_set  JSONB,          -- which rule versions were applied
    model_version     VARCHAR(50)     -- which judge model scored it
) PARTITION BY RANGE (scored_at);
CREATE INDEX idx_scores_agent_time ON conversation_scores(agent_id, scored_at DESC);
Store the Version Set

Recording rule_version_set and model_version on every score is what makes the system auditable and reproducible. Six months later you can answer "why did this call score 45?" by replaying the exact rules and model that scored it.

Partitioning

Partition scores by time (monthly) and cluster by tenant_id. Recent partitions stay hot for dashboards; old partitions move to cheaper storage but remain queryable for audits.

Scaling & Cost
Handling 500K/day and keeping the LLM bill sane.

Scaling the Pipeline

  • Rule engine: trivially parallel, stateless workers — scale horizontally, cheap
  • ML scoring: the bottleneck. Batch, cache, and tier to control both latency and cost
  • Queue-based: completed conversations land in a queue; workers pull and score. Backpressure is natural — if scoring falls behind, the queue absorbs it (score-later, never drop)

The Cost Optimization Stack

TechniqueSavingsTrade-off
Batching (10/call)~20-30%Slight latency increase
Caching near-duplicates~30-40%Cache staleness risk
Tiered (cheap→expensive)~60-70%Cheap model must be well-calibrated
Stack Them

These compound. Tiered scoring on cached, batched requests can take the naive $3.6M/year down toward $500K-800K/year — the difference between a viable product and a money pit. This is the single most important scaling conversation for this system.

Failure Handling

LLM service down

  • Rules still produce compliance verdict immediately
  • ML scores marked pending, retried later
  • Compliance never blocks on ML

Scoring falls behind

  • Queue absorbs backlog
  • Prioritize compliance-critical conversations first
  • Score-later beats score-never

Pitfalls & Follow-ups
Mistakes to avoid and how to answer the hard questions.

Common Pitfalls

Pitfall 1 — LLM for everything

Using an LLM to check compliance. A probabilistic "87% likely verified" is legally useless. Compliance MUST be deterministic rules with evidence. LLM is for subjective coaching dimensions only.

Pitfall 2 — Ignoring cost

Designing the scoring pipeline without addressing the $3.6M/year LLM bill. Cost IS the design here — batch, cache, tier. If you don't bring it up, the interviewer will, and you'll look naive.

Pitfall 3 — Ignoring drift

Assuming the LLM judge scores consistently forever. It drifts. Without a human calibration loop, every scorecard silently corrupts over time.

Pitfall 4 — Ranking on tiny samples

Generating a percentile ranking for an agent with 5 calls. Statistical noise. Require a minimum sample (~30) before ranking or the scores are unfair and untrusted.

Likely Follow-up Questions

Q: How do you keep the LLM cost under control?▶

Three stacked techniques: batch 10 conversations per call (~25% savings), cache scores for near-duplicate conversations (~35%), and tiered scoring where a cheap model handles obvious cases and the expensive model only judges the ambiguous 20-30% (~65%). Stacked, this takes $3.6M/year toward ~$600K.

Q: Why not use ML for compliance too?▶

Compliance needs a defensible, auditable binary answer. "87% likely the agent verified identity" is useless to a regulator or in a dispute. A deterministic rule with a cited transcript excerpt is defensible. Reserve ML for subjective dimensions where nuance beats precision.

Q: How do you know the LLM judge is still accurate?▶

A golden set of human-scored conversations. Periodically re-run the judge against it and measure agreement. When agreement drops below threshold, humans are truth — re-tune the judge until it stabilizes. New models run in shadow mode against the golden set before promotion.

Q: An agent disputes their score. What happens?▶

Every score stores evidence — the exact transcript excerpt and timestamp — plus the rule versions and model version used. You show the agent "at 0:45 you discussed the account before verifying identity." Reproducible and auditable, which is why agents trust it.

Q: What if the ML scoring service goes down?▶

Rules and ML run in parallel, so rules still produce the compliance verdict immediately — the critical path is unaffected. ML scores get marked pending and retried when the service recovers. Compliance never blocks on the availability of the coaching-oriented ML.
