How to Excel in Code Review Interviews

1. Don't pretend to know everything — say: "I'm not an expert in this exact syntax, but I see a potential issue here..."

2. Be collaborative — "What do you think about this approach?" or "Would you prefer X or Y?"

3. Focus on 2-3 most impactful issues rather than nitpicking everything.

4. Always explain WHY something is a problem and suggest a concrete fix.

5. Show trade-off thinking — "This is simpler but less scalable. Depending on expected load, we might want to..."

6. Prioritize by severity — Security > Correctness > Performance > Maintainability > Style.

Challenge 1: Race Condition
Java · Hard · Bug Hunting
Context: This Java class manages a shared counter for a web application's rate limiter. Multiple threads call tryAcquire() simultaneously. The developer claims it's thread-safe.
1public class RateLimiter { 2 private int tokens; 3 private final int maxTokens; 4 private long lastRefillTime; 5 private final int refillRate; 6 7 public RateLimiter(int maxTokens, int refillRate) { 8 this.maxTokens = maxTokens; 9 this.tokens = maxTokens; 10 this.refillRate = refillRate; 11 this.lastRefillTime = System.currentTimeMillis(); 12 } 13 14 public synchronized boolean tryAcquire() { 15 refill(); 16 if (tokens > 0) { 17 tokens--; 18 return true; 19 } 20 return false; 21 } 22 23 private void refill() { 24 long now = System.currentTimeMillis(); 25 long elapsed = now - lastRefillTime; 26 int newTokens = (int)(elapsed / 1000) * refillRate; 27 if (newTokens > 0) { 28 tokens = Math.min(tokens + newTokens, maxTokens); 29 lastRefillTime = now; 30 } 31 } 32 33 public int getAvailableTokens() { 34 return tokens; 35 } 36 37 public void reset() { 38 tokens = maxTokens; 39 lastRefillTime = System.currentTimeMillis(); 40 } 41}
What issues do you see? Select all that apply:
getAvailableTokens() not synchronized — reads stale value
reset() not synchronized — race with tryAcquire
Line 29: lastRefillTime = now loses fractional seconds
tryAcquire() has a race condition inside synchronized block
Integer overflow: elapsed / 1000 * refillRate could overflow for long uptimes
tokens should be AtomicInteger for thread safety
Constructor should validate maxTokens > 0
Line 26: integer division truncates — tokens accumulate slower than expected
Detailed Analysis
Bug — Unsynchronized Methods
getAvailableTokens() and reset() access shared mutable state without synchronization. Another thread could be mid-tryAcquire(), causing torn reads or lost updates.
Fix: Add synchronized keyword to both methods, or use volatile/AtomicInteger for tokens.
Bug — Time Precision Loss
Line 29 sets lastRefillTime = now, discarding any remainder from the integer division on line 26. If 1500ms elapsed, you get 1 token but reset the clock by the full 1500ms — losing 500ms of credit.
Fix: lastRefillTime += (elapsed / 1000) * 1000; // only advance by whole seconds consumed
Performance — Integer Division Truncation
(int)(elapsed / 1000) * refillRate truncates. If refillRate=100 and elapsed=999ms, you get 0 tokens instead of ~99. This makes the rate limiter slower than configured.
Fix: Use elapsed * refillRate / 1000 (multiply first to preserve precision), or track sub-second credits.
Bug — Potential Overflow
For very long-running systems, elapsed can be very large. (int)(elapsed / 1000) * refillRate can overflow int range, producing negative token counts.
Fix: Cap newTokens at maxTokens before addition, or use long arithmetic.
Challenge 2: Off-by-One & Null Handling
Python · Medium · Bug Hunting
Context: A function that implements pagination for a search API. It retrieves results from a database and returns a page of items with metadata.
1def get_paginated_results(query, page, page_size=20): 2 """Fetch paginated search results.""" 3 results = db.search(query) 4 total = len(results) 5 total_pages = total // page_size 6 7 start = (page - 1) * page_size 8 end = start + page_size 9 page_results = results[start:end] 10 11 return { 12 "items": [item.to_dict() for item in page_results], 13 "page": page, 14 "total_pages": total_pages, 15 "total_items": total, 16 "has_next": page < total_pages, 17 "has_prev": page > 1, 18 }
What issues do you see? Select all that apply:
Line 5: total // page_size loses remainder — last partial page unreachable
Line 3: loads ALL results into memory — doesn't use DB-level LIMIT/OFFSET
No validation: page <= 0 or page_size <= 0 will produce wrong results
Line 9: results[start:end] will throw IndexError if out of bounds
Line 12: item.to_dict() will crash if any item in results is None
Line 16: has_next logic is wrong due to the total_pages bug on line 5
Should use enumerate() instead of list comprehension
Missing return None if query is empty
Detailed Analysis
Bug — Off-by-One in total_pages
total // page_size uses floor division. If total=21 and page_size=20, total_pages=1, but there should be 2 pages. The last item is unreachable.
Fix: total_pages = (total + page_size - 1) // page_size # or math.ceil(total / page_size)
Performance — Loading All Results
db.search(query) loads the entire result set into memory, then slices. For large datasets, this is catastrophically slow and memory-intensive.
Fix: Use db.search(query, offset=start, limit=page_size) — push pagination to the database.
Bug — No Input Validation
If page=0 or page=-1, start becomes negative. Python slicing handles this silently but returns wrong data. page_size=0 causes ZeroDivisionError.
Fix: Validate page >= 1 and page_size >= 1 at the start. Return 400 for invalid inputs.
Bug — None Items
If the database returns None entries in the result list, item.to_dict() raises AttributeError. This is common with soft-deleted records.
Fix: Filter None items: [item.to_dict() for item in page_results if item is not None]
Bug — Cascading has_next Error
Since total_pages is wrong, has_next will be False on what should be the second-to-last page, making the last partial page unreachable via pagination UI.
Fix: Fixing total_pages fixes has_next automatically.
Challenge 3: Concurrency Bug
JavaScript · Hard · Bug Hunting
Context: A Node.js service that processes payment webhooks. Multiple webhooks for the same order can arrive simultaneously. The handler is supposed to be idempotent.
1async function handlePaymentWebhook(event) { 2 const { orderId, amount, status } = event.body; 3 4 // Check if already processed 5 const existing = await db.findPayment({ orderId }); 6 if (existing) { 7 return { statusCode: 200, body: 'Already processed' }; 8 } 9 10 // Process payment 11 const order = await db.getOrder(orderId); 12 order.paymentStatus = status; 13 order.paidAmount = amount; 14 15 await db.saveOrder(order); 16 await db.createPayment({ orderId, amount, status }); 17 await sendConfirmationEmail(order.userEmail, order); 18 await updateInventory(order.items); 19 20 return { statusCode: 200, body: 'Payment processed' }; 21}
What issues do you see? Select all that apply:
TOCTOU race: check-then-act on lines 5-16 allows duplicate processing
No transaction: if line 17 fails, order is updated but payment record exists without email
Line 11: order could be null if orderId doesn't exist in DB
No webhook signature verification — anyone can send fake events
Should use Promise.all() for lines 15-18 to improve performance
Line 15-16 ordering: saveOrder before createPayment — partial failure leaves inconsistent state
event.body destructuring will throw if body is undefined
No retry/dead-letter handling — transient failures lose the event permanently
Detailed Analysis
Bug — TOCTOU Race Condition
Between the findPayment check (line 5) and createPayment (line 16), another concurrent webhook can pass the same check. Both proceed to process the same payment, causing double charges, duplicate emails, and inventory over-decrement.
Fix: Use a database unique constraint on orderId + INSERT ... ON CONFLICT DO NOTHING, or use a distributed lock (Redis SETNX) before processing.
Bug — No Transaction Boundary
Lines 15-18 are independent awaits. If sendConfirmationEmail throws, the order and payment are saved but the user never gets notified. The system is in an inconsistent state.
Fix: Wrap DB operations in a transaction. Move side effects (email, inventory) to an outbox pattern or a separate async queue.
Security — No Webhook Verification
No signature verification means anyone who knows the endpoint URL can send fake payment success events, granting free products.
Fix: Verify the webhook signature (e.g., Stripe's stripe-signature header) before processing.
Bug — Null Order
If orderId doesn't match any order (stale data, wrong environment), db.getOrder returns null. Line 12 then throws "Cannot set property 'paymentStatus' of null".
Fix: Check for null and return a 404 or log + retry later.
Reliability — No Dead-Letter Queue
If any await throws an unhandled error, the event is lost. Payment processors typically retry, but without idempotency protection (see TOCTOU above), retries cause duplicates.
Fix: Catch errors, store failed events in a dead-letter queue, alert for manual investigation.
Challenge 4: N+1 & Caching
Python · Medium · Performance
Context: An API endpoint that returns a user's dashboard data including their posts and the number of likes on each post. Called on every page load.
1def get_dashboard(user_id): 2 user = User.objects.get(id=user_id) 3 posts = Post.objects.filter(author=user) 4 5 dashboard_data = { 6 "username": user.username, 7 "avatar": user.profile.avatar_url, 8 "posts": [] 9 } 10 11 for post in posts: 12 likes_count = Like.objects.filter(post=post).count() 13 comments = Comment.objects.filter(post=post) 14 comment_authors = [c.author.username for c in comments] 15 16 dashboard_data["posts"].append({ 17 "title": post.title, 18 "created": post.created_at.isoformat(), 19 "likes": likes_count, 20 "tags": [t.name for t in post.tags.all()], 21 "recent_commenters": comment_authors[:5], 22 }) 23 24 return dashboard_data
What issues do you see? Select all that apply:
N+1 queries: each post triggers separate queries for likes, comments, and tags
Line 14: c.author.username triggers another N+1 — each comment loads its author
Line 7: user.profile.avatar_url triggers extra query — not prefetched
Line 13: loads ALL comments but only uses first 5 — wasteful for popular posts
No caching — dashboard is called on every page load but data changes infrequently
Should use async/await for the database calls
No pagination on posts — user with 1000 posts returns all of them
isoformat() is slow and should use timestamp instead
Detailed Analysis
Performance — Multiple N+1 Query Problems
For N posts, this executes: 1 (user) + 1 (posts) + N (likes) + N (comments) + N (tags) + M (comment authors) queries. For 50 posts with 10 comments each, that's 250+ queries per request.
Fix: Use select_related('author') and prefetch_related('tags', 'comments__author'). Use annotate(likes_count=Count('likes')) to get counts in one query.
Performance — Over-fetching Comments
Line 13 loads ALL comments for each post, but only displays the first 5. A viral post with 10,000 comments wastes massive memory and DB bandwidth.
Fix: Use Comment.objects.filter(post=post).select_related('author')[:5] or Prefetch with queryset slicing.
Performance — No Caching
Dashboard data changes infrequently (only on new posts/likes/comments) but is fetched fresh on every page load. This creates unnecessary database load.
Fix: Cache the result with a TTL (e.g., 60s) or invalidate on write. Consider separate cache keys for frequently-changing data (like counts) vs. static data.
Performance — No Pagination
Users with many posts return all of them. Combined with the N+1 issues above, a prolific user's dashboard could take 10+ seconds.
Fix: Add pagination (e.g., limit to most recent 20 posts) or implement cursor-based pagination.
Challenge 5: Memory Leak
JavaScript · Hard · Performance
Context: A Node.js WebSocket server that manages real-time chat rooms. It's been running in production but ops reports memory usage grows linearly over days until OOM kill.
1const rooms = new Map(); 2const messageHistory = new Map(); 3const userSessions = new Map(); 4 5function onConnection(socket) { 6 const userId = socket.handshake.auth.userId; 7 userSessions.set(userId, { socket, connectedAt: Date.now() }); 8 9 socket.on('join', (roomId) => { 10 if (!rooms.has(roomId)) { 11 rooms.set(roomId, new Set()); 12 messageHistory.set(roomId, []); 13 } 14 rooms.get(roomId).add(userId); 15 socket.emit('history', messageHistory.get(roomId)); 16 }); 17 18 socket.on('message', (roomId, content) => { 19 const msg = { userId, content, timestamp: Date.now() }; 20 messageHistory.get(roomId).push(msg); 21 22 rooms.get(roomId).forEach(uid => { 23 const session = userSessions.get(uid); 24 if (session) session.socket.emit('message', msg); 25 }); 26 }); 27 28 socket.on('disconnect', () => { 29 userSessions.delete(userId); 30 }); 31}
What issues do you see? Select all that apply:
Line 20: messageHistory grows unboundedly — never trimmed or evicted
Line 14: user added to room Set but never removed on disconnect
Rooms/messageHistory Maps never cleaned up even when all users leave
Line 7: if same userId reconnects, old socket reference kept in room Sets — stale broadcasts
Should use WeakMap instead of Map for rooms
Line 20: no null check — if roomId was never joined, .push() crashes
EventEmitter memory leak — too many listeners on socket
Line 15: sends ENTIRE history on join — for active rooms this can be megabytes
Detailed Analysis
Memory Leak — Unbounded Message History
Every message ever sent is stored in memory. A busy chat room accumulating 1000 msgs/hour will consume ~100MB per room per day. This is the primary cause of the OOM kills.
Fix: Cap history (e.g., keep last 100 messages per room). Use a ring buffer or splice: if (history.length > MAX) history.splice(0, history.length - MAX). Better: persist to Redis/DB.
Memory Leak — Ghost Users in Rooms
On disconnect, userId is removed from userSessions but NOT from any room's Set. Over time, room Sets accumulate thousands of disconnected userIds. The forEach on line 22 iterates all of them.
Fix: Track which rooms a user joined. On disconnect, remove from all rooms. Clean up empty rooms.
Memory Leak — Immortal Rooms
Rooms and their messageHistory are created but never destroyed. Even abandoned rooms persist forever.
Fix: When a room's Set becomes empty, delete both the room and its messageHistory from the Maps. Or use TTL-based eviction.
Bug — No Null Check on messageHistory
If a user sends a message to a roomId they haven't joined (or that doesn't exist), messageHistory.get(roomId) returns undefined, and .push() throws TypeError.
Fix: Validate that the user has joined the room before processing messages.
Challenge 6: Algorithm Choice
Java · Medium · Performance
Context: A method in a search service that finds the top-K most relevant documents from a large corpus. Called thousands of times per second. The corpus has ~1M documents.
1public List<Document> findTopK(String query, int k) { 2 List<ScoredDoc> scored = new ArrayList<>(); 3 4 for (Document doc : corpus) { 5 double score = computeRelevance(query, doc); 6 scored.add(new ScoredDoc(doc, score)); 7 } 8 9 scored.sort((a, b) -> Double.compare(b.score, a.score)); 10 11 List<Document> result = new ArrayList<>(); 12 for (int i = 0; i < k && i < scored.size(); i++) { 13 result.add(scored.get(i).doc); 14 } 15 return result; 16}
What issues do you see? Select all that apply:
O(n log n) sort when only top-K needed — should use min-heap for O(n log k)
Allocates 1M ScoredDoc objects per call — massive GC pressure at high QPS
No early termination — scores every document even if clearly irrelevant
Should use parallelStream() for the scoring loop
No pre-filtering — should use inverted index to narrow candidates first
Double.compare has precision issues — should use epsilon comparison
ArrayList initial capacity not set — causes multiple array copies during growth to 1M
Line 12 should use Math.min(k, scored.size()) for clarity
Detailed Analysis
Performance — Wrong Algorithm for Top-K
Sorting 1M items (O(n log n)) when you only need top K (typically 10-20) is wasteful. A min-heap of size K gives O(n log k), which for K=10 is ~20x faster.
Fix: Use PriorityQueue(k, comparator) — add each score, evict minimum when size > k. Final heap contains top-K.
Performance — Excessive Allocation
Creating 1M ScoredDoc objects per query at 1000 QPS = 1 billion short-lived objects per second. This causes constant GC pauses and high memory churn.
Fix: With a heap approach, only K objects live at once. Or use a primitive double[] for scores and track indices.
Performance — No Pre-filtering
Scoring every document against every query is brute-force. Real search systems use inverted indices to narrow to candidate documents first.
Fix: Use an inverted index to find documents containing query terms (typically <1% of corpus), then score only candidates.
Performance — ArrayList Growth
Default ArrayList capacity is 10. Growing to 1M requires ~17 resize operations, each copying all elements. For known size, pre-allocate.
Fix: new ArrayList<>(corpus.size()) — or better, eliminate the list entirely with the heap approach.

Python · Medium · Security
Context: A Flask API endpoint for searching users by various criteria. The developer says they added input validation.
1@app.route('/api/users/search') 2def search_users(): 3 name = request.args.get('name', '') 4 sort_by = request.args.get('sort', 'created_at') 5 role = request.args.get('role', '') 6 7 # Input validation 8 if len(name) > 100: 9 return jsonify(error="Name too long"), 400 10 name = name.replace("'", "''") # escape quotes 11 12 query = f"SELECT id, name, email, role FROM users WHERE name LIKE '%{name}%'" 13 14 if role: 15 query += f" AND role = '{role}'" 16 17 query += f" ORDER BY {sort_by} LIMIT 50" 18 19 results = db.execute(query) 20 return jsonify(users=[dict(r) for r in results])
What issues do you see? Select all that apply:
Line 10: quote escaping is bypassable — doesn't prevent all injection vectors
Line 15: role parameter has NO sanitization — direct injection
Line 17: sort_by is user-controlled column name — allows ORDER BY injection
Should use parameterized queries instead of string formatting
Line 20: exposes email addresses — potential data leak if role filter is bypassed
Missing CSRF protection on this endpoint
No authentication check — endpoint appears publicly accessible

Detailed Analysis
Security — SQL Injection via role parameter
The role parameter is concatenated directly into SQL with no escaping at all. An attacker can use role=' OR 1=1 -- to dump all users.
Fix: Use parameterized queries: db.execute("... AND role = ?", [role])
Security — ORDER BY Injection
sort_by is user-controlled and directly interpolated. An attacker can inject: sort=(CASE WHEN (SELECT...) THEN name ELSE email END) to extract data via blind injection.

Replacing single quotes with doubled quotes is a known-broken approach. It fails with backslash escaping on MySQL, multi-byte character attacks, and doesn't prevent numeric injection vectors.
Fix: NEVER try to manually escape SQL. Always use parameterized queries with your DB driver.
Security — No Authentication
This endpoint has no @login_required decorator or auth check. Anyone can search all users including their email addresses.
Fix: Add authentication and authorization. Restrict email visibility to admin roles.
Challenge 8: XSS & Auth Bypass
