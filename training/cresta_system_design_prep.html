Overview: What Interviewers Want

What They Evaluate
  • Scope management: Can you ask clarifying questions and narrow scope in 5 minutes?
  • Architecture clarity: Clean separation of concerns, clear data flow
  • ML integration awareness: You do NOT design ML models — you design the infrastructure around them
  • Scale reasoning: Back-of-envelope math, bottleneck identification
  • Trade-off articulation: Latency vs consistency, cost vs performance, build vs buy

Key insight: In an AI platform company, the system design interview tests whether you can build reliable, low-latency infrastructure AROUND ML models. The models themselves are black boxes. Your job is plumbing, scaling, and reliability.

General Tips for AI Platform System Design
  1. Draw the boundary: "The ML model is a black box that takes X and returns Y in Z ms. I'm designing everything else."
  2. Start with latency budget: If end-to-end is 200ms, allocate: network 20ms + preprocessing 30ms + model inference 100ms + postprocessing 20ms + delivery 30ms
  3. Identify the hot path: What happens on every single request? Optimize ruthlessly there. What's async? Be lazy there.
  4. Show graceful degradation: If the ML model is slow/down, what fallback exists? Never let AI failures break the core product.
  5. Mention observability early: Latency percentiles, model accuracy drift, error rates — shows production maturity.

Interview structure (45 min): 5 min requirements + 5 min estimation + 15 min high-level design + 15 min deep dive + 5 min scaling/trade-offs. Don't spend 20 min on requirements.

Real-time Agent Assist

Design a system that listens to live conversations and provides real-time suggestions to human agents.

Requirements & Scale

Functional Requirements

  • Process live audio/chat streams and deliver context-aware suggestions to agents
  • Support multiple suggestion types: next-best-action, knowledge articles, compliance alerts
  • Maintain conversation context across the full session (up to 30 min)
  • Allow supervisors to configure suggestion rules per team/campaign

Non-Functional Requirements

MetricTargetRationale
End-to-end latency<200ms (from utterance end to suggestion)Agent must see suggestion while context is fresh
Availability99.95%Failure = agent flies blind during live call
Concurrent sessions10,000Large enterprise contact center at peak

Scale Estimates

  • QPS: 10K sessions x ~1 utterance/3s = ~3,300 utterances/sec to process
  • WebSocket connections: 10K persistent connections (one per agent)
  • Context storage: ~50KB per session x 10K = 500MB active memory
  • Daily transcripts: ~200K conversations/day x 5KB avg = 1GB/day raw text
High-Level Architecture
Audio Stream → [STT Service - BLACK BOX] → Transcript Chunks
    ↓
Transcript Chunks → Session Manager (context accumulation) → NLP Preprocessor
    ↓
NLP Preprocessor → [ML Suggestion Engine - BLACK BOX] → Raw Suggestions
    ↓
Raw Suggestions → Post-processor (filtering, ranking, dedup) → WebSocket Gateway → Agent UI

Component Responsibilities

  • Audio Gateway: Accepts WebRTC/telephony streams, forwards to STT (you don't build this)
  • Session Manager: Maintains per-conversation state: full transcript, speaker turns, detected intents. Key-value store (Redis) keyed by session_id
  • NLP Preprocessor: Extracts features from transcript: intent, entities, sentiment. Lightweight — rule-based + small models
  • ML Suggestion Engine: BLACK BOX. Takes context JSON, returns ranked suggestions. ~80ms P95 latency
  • Post-processor: Applies business rules (suppress duplicates, enforce rate limiting on suggestions), ranks by confidence
  • WebSocket Gateway: Manages persistent connections to agent UIs, handles reconnection, message ordering

Critical path is ~3 hops you control: Session Manager (10ms) + Post-processor (15ms) + WebSocket delivery (5ms) = 30ms budget for your code. The rest is STT + ML model. Design accordingly — keep your services stateless and fast.

Deep Dive: Session Manager

Why This Component

The Session Manager is the most interesting backend challenge: it must accumulate state at high throughput, serve reads with sub-5ms latency, and handle failover without losing conversation context mid-call.

API Schema

// Append new utterance to session
POST /sessions/{session_id}/utterances
{
  "speaker": "customer" | "agent",
  "text": "I need to cancel my subscription",
  "timestamp_ms": 1704067200000,
  "confidence": 0.95
}

// Get current session context (called by ML engine)
GET /sessions/{session_id}/context
Response:
{
  "session_id": "sess_abc123",
  "started_at": "2024-01-01T10:00:00Z",
  "duration_sec": 180,
  "utterances": [...],
  "detected_intents": ["cancellation", "billing_dispute"],
  "customer_sentiment": "frustrated",
  "suggestions_delivered": 3
}

Data Model (Redis)

Key: session:{session_id}
Type: Hash
Fields:
  meta        → JSON (start_time, agent_id, customer_id, campaign)
  utterances  → JSON array (append-only, ordered)
  intents     → Set of detected intents
  last_update → timestamp (for TTL/cleanup)

TTL: 2 hours (auto-cleanup after call ends)
Backup: Async write to PostgreSQL every 30s for recovery

Key Design Decisions

  • Redis over DB for hot path: Sub-ms reads, atomic appends. PostgreSQL as async backup only.
  • Consistent hashing for session affinity: Same session always hits same Redis node — avoids cross-node reads
  • Sliding window context: ML engine only needs last 10 utterances (not full history) — reduces payload size
  • Graceful failover: If Redis node dies, rebuild state from PostgreSQL backup (stale by max 30s)

Key Trade-offs

  • Latency vs accuracy: Could wait for more context (3 utterances) before suggesting → better accuracy but 9s delay. Chose: suggest after each utterance, accept lower precision for faster feedback.
  • Consistency vs availability: If Redis failover takes 30s, agent loses suggestions for 30s. Acceptable — prefer availability over perfect consistency.
  • Push vs pull: Push suggestions via WebSocket (chosen) vs agent UI polling. Push = lower latency, more complex connection management.
Industry Insight: Voice Integration Patterns

How real AI contact center platforms handle audio:

  • Protocols: SIP TRUNK, SIPREC, WebSocket streaming, gRPC streams, raw RTP packets. AI Agent needs bidirectional; Agent Assist only needs listen-only dual-channel.
  • Latency target: Sub-300ms for natural conversation flow. Above 800ms, conversational quality "degrades sharply."
  • Correlation challenge: Audio streams and call metadata arrive via SEPARATE channels with mismatched identifiers. Must correlate before agent answers.
  • Vendor state machines: Each CCaaS vendor (Cisco, Five9, Genesys) fires different event sequences for identical actions (e.g., consultative transfer). Each requires its own state machine implementation.
  • Design pattern: Treat AI agents and human agents as equivalent entities on the same platform — same tools, same monitoring, same orchestration.

Common Follow-ups

Q: "What happens if the ML model latency spikes to 500ms?"

Implement a timeout + fallback. After 150ms, return cached suggestions from similar contexts (last successful suggestion for this intent). Show stale data rather than nothing. Alert on P99 latency drift.

Q: "How do you handle agent disconnection mid-call?"

WebSocket gateway detects disconnect, keeps session alive in Redis. On reconnect (same session_id cookie), replay last 3 suggestions. Session state is independent of connection state.

Q: "How do you A/B test different suggestion models?"

Route at session level (not request level — consistency within a call). Session Manager tags each session with experiment_variant. Post-processor routes to model A or B based on tag. Track acceptance rate per variant.

Knowledge Retrieval (RAG)

Design an enterprise knowledge search system that proactively delivers answers during conversations.

Requirements & Scale

Functional Requirements

  • Ingest enterprise documents (PDFs, wikis, FAQs, product manuals) and keep them searchable
  • Given a conversation context, retrieve the most relevant knowledge passages
  • Generate a natural language answer from retrieved passages (LLM, black box)
  • Support incremental updates — new/modified docs indexed within 15 minutes
  • Track which documents are used, surface staleness and gaps

Non-Functional Requirements

MetricTargetRationale
Retrieval latency<300ms (search + rerank)Budget: 300ms retrieval + 200ms generation = 500ms total
Indexing freshness<15 min from doc updateStale answers erode agent trust
Retrieval recall@10>85%Relevant doc must be in top-10 candidates

Scale Estimates

  • Document corpus: 500K documents, 50M chunks (~200 tokens each)
  • Embedding storage: 50M vectors x 1536 dims x 4 bytes = ~300GB
  • Query QPS: ~2,000 searches/sec (one per agent utterance that triggers search)
  • Ingestion: ~1,000 doc updates/day (batch + real-time)
High-Level Architecture

Ingestion Pipeline

Doc Source (wiki, CMS, S3) → Crawler/Webhook → Chunker (split by section, ~200 tokens)
    ↓
Chunks → [Embedding Model - BLACK BOX] → Vectors
    ↓
Vectors → Vector DB (Pinecone/Weaviate) + Chunks → Elasticsearch (BM25 index)

Query Pipeline

Conversation Context → Query Formulator (extract search query from transcript)
    ↓
Query → Hybrid Search: [ES BM25 top-20] + [Vector DB ANN top-20]
    ↓
Candidates (merged, deduped) → [Reranker Model - BLACK BOX] → Top-5 passages
    ↓
Top-5 + Conversation Context → [LLM Answer Generator - BLACK BOX] → Answer to Agent

Components

  • Crawler: Polls doc sources on schedule + listens for webhooks. Detects changes via content hash.
  • Chunker: Splits documents intelligently (by heading, paragraph). Preserves metadata (source, section, last_modified).
  • Vector DB: Stores embeddings, supports ANN search. Handles ~50M vectors.
  • Elasticsearch: Traditional BM25 keyword search. Handles exact term matches that embeddings miss.
  • Query Formulator: Converts conversation transcript into a concise search query. (Small model or heuristic.)
  • Reranker: BLACK BOX. Takes query + 40 candidate passages, returns top-5 ranked by relevance.
Deep Dive: Hybrid Search & Freshness

Why Hybrid Search

BM25 alone misses semantic similarity ("cancel my plan" vs "subscription termination"). Vectors alone miss exact terms (product IDs, error codes). Combine both.

Merge Strategy: Reciprocal Rank Fusion (RRF)

// Reciprocal Rank Fusion
score(doc) = sum over each ranker R:
    1 / (k + rank_R(doc))

// where k = 60 (standard constant)
// Example: doc ranked #2 in BM25, #5 in vector
// score = 1/(60+2) + 1/(60+5) = 0.016 + 0.015 = 0.031

Data Model

// Document metadata (PostgreSQL)
Table: documents
  id          UUID PRIMARY KEY
  source_url  TEXT
  title       TEXT
  content_hash TEXT  -- detect changes
  last_crawled TIMESTAMP
  status      ENUM('active', 'stale', 'deleted')

// Chunk (stored in both ES and Vector DB)
Table: chunks
  id          UUID PRIMARY KEY
  document_id UUID REFERENCES documents
  chunk_index INT
  text        TEXT
  embedding   VECTOR(1536)  -- in vector DB only
  metadata    JSONB  -- section_title, page_num, etc.

Freshness Strategy

  • Change detection: Crawler computes content_hash. If hash changed, re-chunk and re-embed only affected chunks.
  • Staleness scoring: Boost recently-updated docs in retrieval. Docs not updated in 90 days get staleness flag.
  • Incremental indexing: New chunks indexed into a "hot" partition. Merged into main index in background.
  • Deletion handling: Soft-delete with 24h grace period. Tombstone in vector DB prevents serving deleted content.
Scaling & Trade-offs

Bottlenecks

ComponentCurrentAt 10xSolution
Vector DB50M vectors, 300GB500M vectors, 3TBSharding by tenant/domain; tiered storage (hot/warm)

Trade-offs

  • Freshness vs cost: Real-time embedding on every doc change is expensive (GPU). Chose: 15-min batch window balances freshness and cost.
  • Recall vs latency: Searching more candidates improves recall but adds latency. Chose: top-20 from each source (40 total to reranker) as sweet spot.
  • Chunk size: Smaller chunks = better precision but more vectors (cost). Larger chunks = fewer vectors but may include irrelevant text. Chose: ~200 tokens with overlap.

Common Follow-ups

Q: "How do you handle conflicting information across documents?"

Use document metadata (last_modified, source authority ranking). If two docs conflict, prefer the more recent one from a higher-authority source. Surface the conflict to the agent with both answers + sources.

Q: "How do you evaluate retrieval quality?"

Offline: human-labeled relevance judgments, compute NDCG@10. Online: track agent feedback (thumbs up/down on suggested answers), click-through on source links. A/B test retrieval changes.

Q: "What if the corpus grows to 1B chunks?"

Hierarchical retrieval: first narrow by domain/category (metadata filter), then search within subset. Pre-filtering reduces vector search space by 10-100x. Also consider: do all chunks need to be active? Archive old versions.

Process and analyze millions of conversations for business insights — topics, sentiment, resolution rates.

Functional Requirements

  • Support both near-real-time (within 5 min) and batch (daily aggregation) processing

  • ML processing: Each conversation needs 3-4 model calls (topic, sentiment, intent, quality)

Components

  • Event Bus (Kafka): Decouples conversation completion from processing. Guarantees at-least-once delivery. Partitioned by tenant_id.
  • Stream Processor: Consumes events, orchestrates ML calls, writes enriched records. Stateless workers with auto-scaling.
  • ML Pipeline (BLACK BOX): Topic extraction, sentiment analysis, intent classification, language detection. ~200ms per conversation.
  • Data Lake (S3/GCS): Raw transcripts + enriched metadata in Parquet format. Source of truth.
  • Aggregation Layer (BigQuery): Pre-computed rollups: hourly/daily topic counts, sentiment averages, resolution rates by team.
  • Dashboard Service (Grafana): Real-time + historical charts. Queries aggregation tables, not raw data.
  • Alert Engine: Monitors streaming aggregates. Fires alerts on anomalies (e.g., "cancellation intent up 40% in last hour").

Streaming gives you speed (5-min visibility). Batch gives you accuracy (reprocess with updated models, fix errors). Classic Lambda architecture.

Common Follow-ups

Q: "How do you handle late-arriving data?"

Watermark-based processing. Stream processor waits 2 minutes for stragglers. Late arrivals beyond watermark go to a "late" topic, processed separately and merged into batch results. Dashboard shows "data completeness" indicator.

Q: "How do you add a new analytics dimension without reprocessing everything?"

Q: "How do you ensure data privacy across tenants?"

Tenant isolation at every layer: Kafka partitions by tenant, separate BigQuery datasets or row-level security, encryption at rest with per-tenant keys. Audit log for all data access.

AI-to-Human Handoff

Seamless context transfer when an AI agent escalates to a human agent — zero context loss, minimal wait time.

Requirements & Scale

Functional Requirements

  • Detect when AI agent should escalate (confidence drop, customer request, policy rule)
  • Serialize full conversation context: history, intent, sentiment, attempted resolutions
  • Route to optimal human agent based on skill, availability, and load
  • Deliver context to agent desktop before/during connection (warm transfer)
  • Track SLA: time from escalation trigger to human agent connected

Non-Functional Requirements

MetricTargetRationale
Handoff latency<2s (trigger to agent connected)Customer shouldn't feel "stuck"
Context delivery0% data lossAgent re-asking questions = terrible CX
Agent match accuracy>90% first-route successAvoid re-routing (adds 30s+)

Scale Estimates

  • Escalation rate: ~20% of AI conversations escalate = 100K/day of 500K total
  • Peak escalations/sec: 100K / 8hr / 3600 = ~3.5/sec (bursty — peaks at 10x during incidents)
  • Agent pool: 5,000 human agents across multiple skill groups
  • Context payload: ~10KB per handoff (transcript + metadata + suggestions)
High-Level Architecture
AI Agent Session → Escalation Trigger (confidence < threshold OR customer request)
    ↓
Context Serializer → Handoff Payload (structured summary + full transcript)
    ↓
Agent Router → Skill Matching + Availability Check + Load Balancer
    ↓
Queue Manager → Priority Queue (VIP, SLA urgency, wait time)
    ↓
Agent Desktop Push (WebSocket) → Context Card + Conversation Transfer

Components

  • Escalation Trigger: Rule engine evaluating: AI confidence score, customer explicit request ("talk to a human"), compliance rules (financial advice requires human), max AI turns reached.
  • Context Serializer: Packages conversation state into a structured handoff payload. Includes: summary, full transcript, detected intent, sentiment history, AI actions taken, suggested next steps.
  • Agent Router: Matches escalation to best agent. Inputs: required skills, language, VIP status. Considers: agent availability, current load, historical performance on this topic.
  • Queue Manager: Priority queue per skill group. Manages SLA tracking, overflow routing (if primary skill group is full, expand to secondary).
  • Agent Desktop Push: Delivers context to agent UI via WebSocket before voice/chat is connected. Agent sees customer history while phone rings.
Deep Dive: Agent Router

Why This Component

Routing quality directly impacts customer experience AND agent efficiency. Bad routing = rerouting = +30s wait + frustrated customer. This is where backend engineering matters most.

Routing Algorithm

// Agent scoring function
func ScoreAgent(escalation Escalation, agent Agent) float64 {
    score := 0.0

    // Skill match (must-have)
    if !agent.HasSkill(escalation.RequiredSkill) {
        return -1  // disqualify
    }

    // Language match
    if agent.Language == escalation.Language {
        score += 30
    }

    // Current load (prefer less busy)
    score += (1 - agent.CurrentLoad) * 25  // 0-25 points

    // Historical performance on this topic
    score += agent.TopicSuccessRate(escalation.Topic) * 20  // 0-20 points

    // Time since last break (freshness)
    score += min(agent.MinutesSinceBreak / 60, 1.0) * 10  // 0-10 points

    // VIP handling (experienced agents for VIPs)
    if escalation.IsVIP && agent.Tier >= 2 {
        score += 15
    }

    return score
}

Industry Insight: Real-time Supervision & Handoff

How real platforms handle AI supervision and handoff:

  • Grid view: Supervisors monitor 100+ concurrent AI conversations. Visual hierarchy (color, motion) encodes priority — scan, then drill into specific conversations.
  • Demand/supply routing: Works like ride-hailing — AI agent requests (demand) matched to supervisor skills/availability (supply). Handles overflow, timeouts, and re-surfacing.
  • Multi-turn takeover: Supervisor pauses AI → engages customer directly for multiple turns → stabilizes → hands back to AI. Not just a one-shot transfer.
  • Transfer types: Cold (direct handoff), Warm (AI introduces human first), Conference (AI stays to brief human, then exits).
  • Unified entity model: AI and human agents treated as equivalent entities on same platform — same monitoring tools, same orchestration, same escalation patterns.

Tiered fallback: (1) Expand skill search to secondary skills, (2) Offer callback option with estimated wait time, (3) AI continues with disclaimer "agent will join shortly." Never leave customer in silent queue. Track queue abandonment rate.

Q: "How do you prevent the same customer from being escalated repeatedly?"

Track escalation history per customer. If 3+ escalations in 7 days for same intent, auto-route to supervisor or specialized retention team. Flag the AI model that this customer pattern isn't being resolved by standard agents.

Q: "How do you measure handoff quality?"

Metrics: (1) Did agent ask a question already answered by AI? (repeat-question rate), (2) Time-to-resolution after handoff, (3) Customer satisfaction post-handoff vs post-AI-only. A/B test different context summary formats.

Trade-offs

  • Strict rules vs ML flexibility: Rules catch exact violations (high precision, lower recall). ML catches nuance (higher recall, lower precision). Chose: rules for compliance (must be precise), ML for coaching (can tolerate noise).
  • Real-time vs batch scoring: Could batch-score hourly for cost efficiency. Chose: near-real-time for compliance (5 min), batch for coaching scores (daily refresh acceptable).
  • Transparency vs complexity: Agents need to understand why they scored low. Rule results are transparent ("you didn't verify identity at 0:45"). ML scores are opaque. Chose: always show rule evidence; for ML, show example phrases that influenced the score.
Industry Insight: STT Evaluation & QA Framework

How real platforms evaluate speech and conversation quality:

  • 4-layer evaluation: Lexical (WER/CER) → Entity (keyword recall, entity accuracy with 3-5x weight multiplier) → Semantic (LLM-as-judge) → Task (end-to-end success rate). Task layer is the ultimate gate.
  • LLM-as-judge patterns: Three modes — (1) Judge: reference vs hypothesis, rated 1-5; (2) Extractor: independently extract facts from both, compare; (3) Consistency-checker: single-transcript contradiction detection.
  • Critical insight: "LLMs do not fail gracefully on STT errors — they fail confidently." One digit swap triggers wrong tool calls. A missed negation flips intent entirely.
  • Test coverage from real data: Mine historical conversations for test cases. 3 types — happy path (early), critical deviations (pre-launch), post-launch feedback (continuous regression).
  • Evaluator drift: Human-rated samples required to catch LLM judge drift. When disagreements occur, "humans win and the judge gets re-tuned." Loop until agreement rates stabilize.
  • Slice-based evaluation: Segment by language × noise condition × speaker role. Worst-slice behavior matters more than averages.

Common Follow-ups

Q: "How do you handle disagreements between ML score and human QA?"

Human QA score always wins for that specific conversation. But log the disagreement and feed it back as training data. Track disagreement rate per dimension — if empathy model disagrees with humans >20% of the time, retrain. Show "confidence" alongside ML scores.

Q: "How do you prevent gaming — agents optimizing for the score vs actual quality?"

Goodhart's Law defense: (1) Don't make individual rules visible to agents (only dimension-level scores), (2) Rotate rule weights periodically, (3) Include outcome metrics (customer satisfaction, resolution rate) that can't be gamed by transcript manipulation, (4) Human QA samples 5% as calibration.

Q: "How do you onboard a new client with different QA criteria?"

Template-based rule sets: start with industry template (financial, healthcare, retail), then customize. New client runs in "calibration mode" for 2 weeks — human QA scores 200 conversations, system trains thresholds to match human judgment, then goes live with monitoring.

Q: "What if the conversation is in a language the rule engine doesn't support?"

Two approaches: (1) Translate to English first, then apply rules (adds ~100ms, some nuance lost), (2) Build language-specific rule patterns for top languages. Chose: translate for long-tail languages, native rules for top-5 (80% of volume). ML models are already multilingual (black box).
