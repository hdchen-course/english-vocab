System Design: Device Test Lab

Design a system that manages a pool of physical devices for automated testing. Engineers submit test requests, the system schedules them on available devices, executes tests, and reports results.

Requirements

Functional

  • Submit test job (specify: test suite, device type, OS version, timeout)
  • Queue management (priority, FIFO within priority)
  • Device allocation (match device capabilities to test requirements)
  • Test execution (flash firmware if needed → run suite → collect results)
  • Result reporting (pass/fail, logs, screenshots, video)
  • Retry on flaky failures

Non-functional

  • 100+ devices across multiple device types
  • 1000+ test jobs per day
  • Fair scheduling (no team starves)
  • Device health monitoring (detect broken/offline devices)
  • 99.9% scheduler uptime

Architecture

Components

1. Job Queue (Priority Queue + Persistence)

  • Store: Database (PostgreSQL or Spanner)
  • Schema: job_id, submitter, priority, device_requirements, test_suite, status, created_at
  • Priority: P0 (release blocking) > P1 (presubmit) > P2 (nightly)

2. Scheduler

  • Matches job requirements to available devices
  • Considers: device type, OS version, hardware revision, health status
  • Fair scheduling: round-robin across teams at same priority
  • Trigger: device becomes available → pull next matching job

3. Device Manager

  • Device states: AVAILABLE, IN_USE, FLASHING, OFFLINE, MAINTENANCE
  • Health checks: periodic heartbeat, detect unresponsive
  • Auto-quarantine: 3 consecutive test failures → MAINTENANCE

4. Test Executor (per device host)

  • Flash firmware if job requires specific version
  • Execute test suite with timeout
  • Collect artifacts: logs, logcat, screenshots, crash dumps
  • Watchdog: kill stuck tests → report timeout

5. Result Store + Dashboard

  • All results with full artifacts
  • Aggregations: pass rate by suite/device/date
  • Flaky detection: same test, same code, different results
  • Regression detection: pass rate drop after specific commit

Detailed Design: Scheduler Algorithm

while (true):
    device = waitForAvailableDevice()           // event-driven, not polling
    job = findBestMatchingJob(device)           // query DB: match device caps + priority
    if job == null: continue

    markDevice(device, IN_USE)
    markJob(job, RUNNING, assigned_device=device)
    notifyExecutor(device.host, job)            // RPC or queue message

Matching Logic

  • Hard constraints: device_type must match, OS version must match (or "any")
  • Soft preferences: prefer device with most recent successful test (less likely broken)
  • Priority tiebreaker: within same priority, FIFO by creation time
  • Starvation prevention: if a job waits > 30 min, boost priority by 1 level

Detailed Design: Flaky Test Detection

SignalDetectionAction
Same test, same code, different resultsTrack (test_id, commit_hash) → pass/fail ratioIf ratio between 20%-80% over 5+ runs → mark flaky
Fails only on specific deviceCompare pass rate per device for same testQuarantine device, not test
Fails only at certain timeCorrelate with time-of-day, other running jobsLikely resource contention → isolate or add capacity

Design Decisions (Talking Points)

"Why not use VMs/emulators?" — Hardware-specific sensors (cameras, IMUs, displays, payment chip readers) can't be virtualized. Timing-sensitive tests (Bluetooth, NFC) behave differently on emulators.

"How to handle flaky tests?" — Auto-retry up to 2x. If still flaky, quarantine and notify owner. Track flake rate per test. Distinguish "test flake" from "device flake" from "infra flake."

"How to scale?" — Devices + hosts scale horizontally. Scheduler is stateless (reads from DB). No single point of failure. Can add devices from new geo locations (multi-site lab).

"Scheduler goes down?" — Jobs stay in DB queue. Scheduler restarts, picks up. Running jobs continue on their hosts — executor reports result directly to DB. No coordinator dependency during execution.

"Fair scheduling?" — Weighted fair queuing: each team gets a proportional share. Burst allowed if others idle. Age-based priority boosting prevents starvation.

"Device breaks mid-test?" — Executor has watchdog timer. If no heartbeat from device for 60s, mark test as DEVICE_ERROR (not test failure). Auto-quarantine device. Requeue job for another device.

System Design: AI Bug Triage Agent

Design an AI agent that automatically triages incoming bugs: reads the bug report + logs, identifies likely root cause, assigns to the right team/person.

Architecture

How It Works

  1. Bug filed → trigger agent
  2. Extract context: stack trace, affected component, device type, OS version
  3. RAG lookup: similar past bugs, code ownership for affected files
  4. LLM reasoning: "Given this crash in libdisplay.so at RenderFrame(), and past bugs in this file owned by Display team → assign to Display team"
  5. Confidence check: if confidence < threshold → human triage queue
  6. Feedback loop: if reassigned by human → learn from correction

Key Design Decisions

QuestionAnswer
Why RAG, not just LLM?Code ownership changes weekly. LLM training data stale. RAG gives fresh context from OWNERS files + recent git blame.
How to measure success?% of bugs correctly auto-assigned (no human reassignment). Target: 70%+
What about hallucination?Always include confidence score. Low confidence → human triage. Never auto-close.
Cold start?Bootstrap from historical bug assignments (last 6 months). Fine-tune on reassignment corrections.

AI Agent Execution Model

From studying real AI agent platforms: The key primitives are: Agent (configured with persona + tools + model), Skill (reusable expertise: RAG knowledge base, tool access, sub-agents), Job (task definition + trigger), JobRun (single execution with lifecycle: QUEUED → STARTED → IN_PROGRESS → COMPLETED/FAILED). This decomposition enables non-engineers to compose agents from pre-built skills.

System Design: CI/CD for Device Firmware

Design CI/CD pipeline for device firmware. Must build for multiple hardware targets, flash onto real devices for testing, and gate releases on test results.

Pipeline Stages

Key Challenges

ChallengeSolution
Multi-target build (3+ hardware variants)Parallel build matrix, shared source with target-specific configs
Physical device dependencyDevice Test Lab integration (previous design)
Long build times (30-60 min)Incremental builds, cache object files, only rebuild changed modules
Bricked device riskRecovery bootloader that can reflash even if main OS is broken
Sidecar/container image stalenessVersion-pinned images per environment, pipeline gates on image freshness

Release Strategy

  • Canary: Flash 5% of test devices first, wait 1hr, then rest
  • Environment promotion: dev → gamma → staging → prod (gated by test pass rate)
  • Rollback: Auto-rollback if error rate > threshold within 30 min of deploy
  • Image tagging: Environment-specific tags (e.g., :gamma, :prod) to prevent cross-environment contamination

Real-world lesson (from experience): A pipeline's "publish-to-live" step was blocked for 2 months due to a cross-platform build failure. All downstream services ran stale images. New code merged but never deployed. Lesson: pipeline health monitoring must alert on promotion failures, not just build failures.

Monitoring Must-Haves

  • Alert on pipeline stage failure (not just final result)
  • Alert on image age > threshold per environment
  • Dashboard: last successful deploy date per service per environment
  • Compare image digests across environments (detect staleness)
