System Design: Device Test Lab

Design a system that manages a pool of physical devices for automated testing. Engineers submit test requests, the system schedules them on available devices, executes tests, and reports results.

Requirements

Functional

  • Submit test job (specify: test suite, device type, OS version, timeout)
  • Queue management (priority, FIFO within priority)
  • Device allocation (match device capabilities to test requirements)
  • Test execution (flash firmware if needed → run suite → collect results)
  • Result reporting (pass/fail, logs, screenshots, video)
  • Retry on flaky failures

Non-functional

  • 100+ devices across multiple device types
  • 1000+ test jobs per day
  • Fair scheduling (no team starves)
  • Device health monitoring (detect broken/offline devices)
  • 99.9% scheduler uptime

Architecture

Components

1. Job Queue (Priority Queue + Persistence)

  • Store: Database (PostgreSQL or Spanner)
  • Schema: job_id, submitter, priority, device_requirements, test_suite, status, created_at
  • Priority: P0 (release blocking) > P1 (presubmit) > P2 (nightly)

2. Scheduler

  • Matches job requirements to available devices
  • Considers: device type, OS version, hardware revision, health status
  • Fair scheduling: round-robin across teams at same priority
  • Trigger: device becomes available → pull next matching job

3. Device Manager

  • Device states: AVAILABLE, IN_USE, FLASHING, OFFLINE, MAINTENANCE
  • Health checks: periodic heartbeat, detect unresponsive
  • Auto-quarantine: 3 consecutive test failures → MAINTENANCE

4. Test Executor (per device host)

  • Flash firmware if job requires specific version
  • Execute test suite with timeout
  • Collect artifacts: logs, logcat, screenshots, crash dumps
  • Watchdog: kill stuck tests → report timeout

5. Result Store + Dashboard

  • All results with full artifacts
  • Aggregations: pass rate by suite/device/date
  • Flaky detection: same test, same code, different results
  • Regression detection: pass rate drop after specific commit

Detailed Design: Scheduler Algorithm

while (true):
    device = waitForAvailableDevice()           // event-driven, not polling
    job = findBestMatchingJob(device)           // query DB: match device caps + priority
    if job == null: continue

    markDevice(device, IN_USE)
    markJob(job, RUNNING, assigned_device=device)
    notifyExecutor(device.host, job)            // RPC or queue message

Matching Logic

  • Hard constraints: device_type must match, OS version must match (or "any")
  • Soft preferences: prefer device with most recent successful test (less likely broken)
  • Priority tiebreaker: within same priority, FIFO by creation time
  • Starvation prevention: if a job waits > 30 min, boost priority by 1 level

Detailed Design: Flaky Test Detection

SignalDetectionAction
Same test, same code, different resultsTrack (test_id, commit_hash) → pass/fail ratioIf ratio between 20%-80% over 5+ runs → mark flaky
Fails only on specific deviceCompare pass rate per device for same testQuarantine device, not test
Fails only at certain timeCorrelate with time-of-day, other running jobsLikely resource contention → isolate or add capacity

Design Decisions (Talking Points)

"Why not use VMs/emulators?" — Hardware-specific sensors (cameras, IMUs, displays, payment chip readers) can't be virtualized. Timing-sensitive tests (Bluetooth, NFC) behave differently on emulators.

"How to handle flaky tests?" — Auto-retry up to 2x. If still flaky, quarantine and notify owner. Track flake rate per test. Distinguish "test flake" from "device flake" from "infra flake."

"How to scale?" — Devices + hosts scale horizontally. Scheduler is stateless (reads from DB). No single point of failure. Can add devices from new geo locations (multi-site lab).

"Scheduler goes down?" — Jobs stay in DB queue. Scheduler restarts, picks up. Running jobs continue on their hosts — executor reports result directly to DB. No coordinator dependency during execution.

"Fair scheduling?" — Weighted fair queuing: each team gets a proportional share. Burst allowed if others idle. Age-based priority boosting prevents starvation.

"Device breaks mid-test?" — Executor has watchdog timer. If no heartbeat from device for 60s, mark test as DEVICE_ERROR (not test failure). Auto-quarantine device. Requeue job for another device.

System Design: AI Bug Triage Agent

Design an AI agent that automatically triages incoming bugs: reads the bug report + logs, identifies likely root cause, assigns to the right team/person.

Architecture

How It Works

  1. Bug filed → trigger agent
  2. Extract context: stack trace, affected component, device type, OS version
  3. RAG lookup: similar past bugs, code ownership for affected files
  4. LLM reasoning: "Given this crash in libdisplay.so at RenderFrame(), and past bugs in this file owned by Display team → assign to Display team"
  5. Confidence check: if confidence < threshold → human triage queue
  6. Feedback loop: if reassigned by human → learn from correction

Key Design Decisions

QuestionAnswer
Why RAG, not just LLM?Code ownership changes weekly. LLM training data stale. RAG gives fresh context from OWNERS files + recent git blame.
How to measure success?% of bugs correctly auto-assigned (no human reassignment). Target: 70%+
What about hallucination?Always include confidence score. Low confidence → human triage. Never auto-close.
Cold start?Bootstrap from historical bug assignments (last 6 months). Fine-tune on reassignment corrections.

AI Agent Execution Model

From studying real AI agent platforms: The key primitives are: Agent (configured with persona + tools + model), Skill (reusable expertise: RAG knowledge base, tool access, sub-agents), Job (task definition + trigger), JobRun (single execution with lifecycle: QUEUED → STARTED → IN_PROGRESS → COMPLETED/FAILED). This decomposition enables non-engineers to compose agents from pre-built skills.

System Design: CI/CD for Device Firmware

Design CI/CD pipeline for device firmware. Must build for multiple hardware targets, flash onto real devices for testing, and gate releases on test results.

Pipeline Stages

Key Challenges

ChallengeSolution
Multi-target build (3+ hardware variants)Parallel build matrix, shared source with target-specific configs
Physical device dependencyDevice Test Lab integration (previous design)
Long build times (30-60 min)Incremental builds, cache object files, only rebuild changed modules
Bricked device riskRecovery bootloader that can reflash even if main OS is broken
Sidecar/container image stalenessVersion-pinned images per environment, pipeline gates on image freshness

Release Strategy

  • Canary: Flash 5% of test devices first, wait 1hr, then rest
  • Environment promotion: dev → gamma → staging → prod (gated by test pass rate)
  • Rollback: Auto-rollback if error rate > threshold within 30 min of deploy
  • Image tagging: Environment-specific tags (e.g., :gamma, :prod) to prevent cross-environment contamination

Real-world lesson (from experience): A pipeline's "publish-to-live" step was blocked for 2 months due to a cross-platform build failure. All downstream services ran stale images. New code merged but never deployed. Lesson: pipeline health monitoring must alert on promotion failures, not just build failures.

Monitoring Must-Haves

  • Alert on pipeline stage failure (not just final result)
  • Alert on image age > threshold per environment
  • Dashboard: last successful deploy date per service per environment
  • Compare image digests across environments (detect staleness)

Likely Interview Questions + How to Answer

Technical Leadership

"How would you prioritize: fixing flaky tests vs building new AI tools vs improving build speed?"

What they're really asking: Can you make trade-offs? Do you think about team impact vs effort?

Answer framework: "I'd prioritize based on (1) how many engineers are blocked right now, (2) severity of pain, and (3) effort to fix. Flaky tests usually win because they erode trust in CI — engineers start ignoring failures and real bugs slip through. Build speed is next because it affects every engineer every day. AI tools are force multipliers but only after the foundation is solid."

"A team is blocked because the device lab has no available devices. What do you do?"

Answer: "Immediate: check if any devices are stuck (health check failures, zombie jobs). Often 20% of devices are in MAINTENANCE state that nobody cleared. Medium-term: add priority lanes so release-blocking tests can preempt nightly runs. Long-term: track utilization metrics to justify budget for more devices."

"Your AI bug triage tool has 60% accuracy. Ship it or wait for 80%?"

Answer: "Ship it with a confidence threshold — only auto-assign when confidence > 90%. Below that, suggest but require human confirmation. 60% overall means the top-confidence predictions are probably 85%+. Let humans handle the uncertain ones. Measure: did we reduce average triage time? If yes, it's already providing value."

"How do you measure engineering productivity? What metrics?"

Answer: "I focus on cycle time (code push → test result), not lines of code. Key metrics: (1) P50/P90 time from push to green CI, (2) flaky test rate, (3) device utilization, (4) MTTR for regressions. I'd avoid vanity metrics like 'number of tests' — a team can write 1000 trivial tests that catch nothing."

People Management

"If you don't have domain expertise in this area, how do you manage the team?"

This exact question was asked in a past coffee chat. Key question for EM roles where you're new to the domain.

Answer: "I don't need to be the deepest domain expert on my team — that's what my senior engineers are for. My value is: (1) defining process and quality bar, (2) unblocking cross-team dependencies, (3) growing people, and (4) making sure we ship on time. Domain knowledge I can ramp up on — I've done it multiple times. At my current company I took over an IoT test system with zero context and used AI tools to build a full architectural understanding in half a day. The engineering judgment and people management skills are what take years to build and are portable across domains."

"How do you grow engineers on your team?"

Answer: "I match stretch assignments to their growth areas. For a junior engineer who's technically strong but struggles with ambiguity, I give them a project with unclear requirements and coach them through the scoping process. I had a junior engineer I coached from IC to engineering manager in 2 years — by progressively expanding their scope from owning a feature, to owning a service, to leading a small sub-team."

"How do you handle an underperformer?"

Answer: "First, diagnose whether it's a skill issue, motivation issue, or clarity issue. Most 'underperformance' is actually unclear expectations. I start with a direct conversation: 'here's what I expect at your level, here's where I see a gap.' Set specific, measurable goals for 2-4 weeks. Meet weekly to check progress. If it's a skill gap, provide resources and pairing. If after 4-6 weeks there's no improvement despite clear support, then we have a harder conversation about fit."

"Tell me about a time you gave difficult feedback"

Answer: "At a previous company, I had an engineer who was technically brilliant but was dismissive in code reviews — leaving comments like 'this is wrong' without explanation. Other engineers started avoiding his reviews. I pulled him aside and showed him specific examples: 'when you write this, here's how the other person feels. Your technical insight is valuable but the delivery is blocking collaboration.' He was surprised — he thought he was being efficient. We agreed on a format: always explain why, always suggest an alternative. Within a month his review style changed and the team started actively requesting his reviews again."

"How do you manage across multiple teams/locations/time zones?"

Answer: "Async-first communication: decisions documented in writing, not just in meetings. Clear ownership boundaries: each team owns a service end-to-end. Overlap hours for sync: find the 2-3 hour window where all locations are online and use it for decisions that need real-time discussion. Everything else goes through async channels with clear SLAs for response time."

AI/LLM

"How would you use LLMs to generate unit tests?"

Answer: "Feed the LLM: (1) the function signature + docstring, (2) 2-3 example tests from the same codebase for style reference, (3) coverage report showing untested branches. Ask it to generate tests for uncovered paths. Human reviews generated tests for: correct assertions (not just 'it doesn't crash'), meaningful edge cases, no false positives. Measure: did coverage increase? Did we catch any real bugs?"

"What are the risks of AI-generated test code?"

Answer: "Three main risks: (1) Tests that always pass — assert nothing meaningful, just exercise code without checking correctness. (2) Tests that encode current behavior as 'correct' — if the code has a bug, the AI generates a test that expects the buggy behavior. (3) Maintenance burden — generated tests may be brittle, breaking on any refactor. Mitigation: require human review, mutation testing to verify tests actually catch bugs, and prefer generated tests that test contracts/interfaces not implementation details."

"How do you handle LLM hallucination in an automated system?"

Answer: "Never trust LLM output for consequential actions without verification. Three layers: (1) Constrain output format — use function calling / structured output so the LLM can only emit valid actions. (2) Confidence gating — only auto-execute above threshold, human review below. (3) Reversibility — if the action is reversible (e.g., assign a bug), auto-execute. If irreversible (e.g., close a bug, deploy code), always require human approval."

Debugging / RCA

"Tests pass locally but fail in CI. How do you debug?"

Answer: "Classic categories: (1) Environment difference — different OS, different dependencies, different timezone/locale. (2) Ordering dependency — locally tests run in one order, CI shuffles. (3) Resource constraint — CI has less memory/CPU, causes timeouts. (4) Shared state — test pollutes DB/filesystem, locally it's fresh each run. My approach: reproduce in CI environment first (run in Docker matching CI image), then bisect which category it is."

"A pipeline has been silently broken for weeks. How do you detect and prevent this?"

Answer: "I've seen this firsthand — a container image pipeline was blocked for 2 months and nobody noticed because the running service used a cached image. Prevention: (1) Alert on pipeline stage failures, not just final build status. (2) Monitor artifact freshness — alert if deployed image is older than X days. (3) Compare artifact digests across environments — catch staleness early. (4) Weekly pipeline health dashboard reviewed in team standup. The key insight: monitor deployments, not just builds. A passing build that never deploys is invisible."

"How do you debug a device test that fails intermittently?"

Answer: "Three suspects: (1) Device hardware — run same test on multiple devices, if it only fails on one device → device issue. (2) Timing/race condition — add retries, check if failure correlates with concurrent tests on same device or network traffic. (3) Environmental — correlate with time of day, other running jobs, or infrastructure state. Collect: full device logs, host logs, network state at time of failure. The hardest bugs are 'once a day' failures — for those I instrument heavily and wait."

Scaling & Strategy

"How would you build a team from scratch for this role?"

Answer: "First hire: a senior engineer who knows the device/platform well (compensates for my ramp-up time). Second: an infra engineer who can build CI/CD and test frameworks. Third: an AI/ML engineer for the tooling roadmap. First 90 days: understand current pain points by embedding with the teams we serve. Don't build anything until I've talked to 10+ engineers about what slows them down."

"What would your first 30/60/90 days look like?"

Answer:

  • 30 days: Listen. 1:1 with every team member + key stakeholders. Understand current systems, pain points, ongoing projects. Build relationship with partner teams (firmware, app, infra).
  • 60 days: Identify top 3 pain points. Propose roadmap. Quick win: fix the most-complained-about issue (usually flaky tests or slow builds). Start hiring if headcount available.
  • 90 days: Ship first improvement. Establish metrics dashboard. Set team goals for next quarter aligned with org priorities. Have regular syncs with partner teams.

"How do you balance platform work vs feature requests from other teams?"

Answer: "70/20/10 rule: 70% on committed roadmap items (reduce build time, improve test reliability). 20% on high-priority requests from partner teams (unblock a launch). 10% on tech debt and exploration (AI tools, new approaches). Requests from other teams go through a lightweight intake process — prevents drive-by interruptions while keeping us responsive. If a request is truly blocking a launch, we interrupt our roadmap. If it's 'nice to have,' it goes in the backlog and gets prioritized next quarter."
