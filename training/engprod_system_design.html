System Design: Device Test Lab

Design a system that manages a pool of physical devices for automated testing. Engineers submit test requests, the system schedules them on available devices, executes tests, and reports results.

Requirements

Functional

  • Submit test job (specify: test suite, device type, OS version, timeout)
  • Queue management (priority, FIFO within priority)
  • Device allocation (match device capabilities to test requirements)
  • Test execution (flash firmware if needed → run suite → collect results)
  • Result reporting (pass/fail, logs, screenshots, video)
  • Retry on flaky failures

Non-functional

  • 100+ devices across multiple device types
  • 1000+ test jobs per day
  • Fair scheduling (no team starves)
  • Device health monitoring (detect broken/offline devices)
  • 99.9% scheduler uptime

Architecture

Components

1. Job Queue (Priority Queue + Persistence)

  • Store: Database (PostgreSQL or Spanner)
  • Schema: job_id, submitter, priority, device_requirements, test_suite, status, created_at
  • Priority: P0 (release blocking) > P1 (presubmit) > P2 (nightly)

2. Scheduler

  • Matches job requirements to available devices
  • Considers: device type, OS version, hardware revision, health status
  • Fair scheduling: round-robin across teams at same priority
  • Trigger: device becomes available → pull next matching job

3. Device Manager

  • Device states: AVAILABLE, IN_USE, FLASHING, OFFLINE, MAINTENANCE
  • Health checks: periodic heartbeat, detect unresponsive
  • Auto-quarantine: 3 consecutive test failures → MAINTENANCE

4. Test Executor (per device host)

  • Flash firmware if job requires specific version
  • Execute test suite with timeout
  • Collect artifacts: logs, logcat, screenshots, crash dumps
  • Watchdog: kill stuck tests → report timeout

5. Result Store + Dashboard

  • All results with full artifacts
  • Aggregations: pass rate by suite/device/date
  • Flaky detection: same test, same code, different results
  • Regression detection: pass rate drop after specific commit

Detailed Design: Scheduler Algorithm

while (true):
    device = waitForAvailableDevice()           // event-driven, not polling
    job = findBestMatchingJob(device)           // query DB: match device caps + priority
    if job == null: continue

    markDevice(device, IN_USE)
    markJob(job, RUNNING, assigned_device=device)
    notifyExecutor(device.host, job)            // RPC or queue message

Matching Logic

  • Hard constraints: device_type must match, OS version must match (or "any")
  • Soft preferences: prefer device with most recent successful test (less likely broken)
  • Priority tiebreaker: within same priority, FIFO by creation time
  • Starvation prevention: if a job waits > 30 min, boost priority by 1 level

Detailed Design: Flaky Test Detection

SignalDetectionAction

Design Decisions (Talking Points)

"Why not use VMs/emulators?" — Hardware-specific sensors (cameras, IMUs, displays, payment chip readers) can't be virtualized. Timing-sensitive tests (Bluetooth, NFC) behave differently on emulators.

"How to handle flaky tests?" — Auto-retry up to 2x. If still flaky, quarantine and notify owner. Track flake rate per test. Distinguish "test flake" from "device flake" from "infra flake."

"How to scale?" — Devices + hosts scale horizontally. Scheduler is stateless (reads from DB). No single point of failure. Can add devices from new geo locations (multi-site lab).

"Scheduler goes down?" — Jobs stay in DB queue. Scheduler restarts, picks up. Running jobs continue on their hosts — executor reports result directly to DB. No coordinator dependency during execution.

"Fair scheduling?" — Weighted fair queuing: each team gets a proportional share. Burst allowed if others idle. Age-based priority boosting prevents starvation.

"Device breaks mid-test?" — Executor has watchdog timer. If no heartbeat from device for 60s, mark test as DEVICE_ERROR (not test failure). Auto-quarantine device. Requeue job for another device.

System Design: AI Bug Triage Agent

Design an AI agent that automatically triages incoming bugs: reads the bug report + logs, identifies likely root cause, assigns to the right team/person.

Architecture

How It Works

  1. Bug filed → trigger agent
  2. Extract context: stack trace, affected component, device type, OS version
  3. RAG lookup: similar past bugs, code ownership for affected files
  4. LLM reasoning: "Given this crash in libdisplay.so at RenderFrame(), and past bugs in this file owned by Display team → assign to Display team"
  5. Confidence check: if confidence < threshold → human triage queue
  6. Feedback loop: if reassigned by human → learn from correction

Key Design Decisions

QuestionAnswer
Why RAG, not just LLM?Code ownership changes weekly. LLM training data stale. RAG gives fresh context from OWNERS files + recent git blame.
How to measure success?% of bugs correctly auto-assigned (no human reassignment). Target: 70%+
What about hallucination?Always include confidence score. Low confidence → human triage. Never auto-close.
Cold start?Bootstrap from historical bug assignments (last 6 months). Fine-tune on reassignment corrections.

AI Agent Execution Model

From studying real AI agent platforms: The key primitives are: Agent (configured with persona + tools + model), Skill (reusable expertise: RAG knowledge base, tool access, sub-agents), Job (task definition + trigger), JobRun (single execution with lifecycle: QUEUED → STARTED → IN_PROGRESS → COMPLETED/FAILED). This decomposition enables non-engineers to compose agents from pre-built skills.

System Design: CI/CD for Device Firmware

Design CI/CD pipeline for device firmware. Must build for multiple hardware targets, flash onto real devices for testing, and gate releases on test results.

Pipeline Stages

Key Challenges

Release Strategy

  • Canary: Flash 5% of test devices first, wait 1hr, then rest
  • Environment promotion: dev → gamma → staging → prod (gated by test pass rate)
  • Rollback: Auto-rollback if error rate > threshold within 30 min of deploy
  • Image tagging: Environment-specific tags (e.g., :gamma, :prod) to prevent cross-environment contamination

Real-world lesson (from experience): A pipeline's "publish-to-live" step was blocked for 2 months due to a cross-platform build failure. All downstream services ran stale images. New code merged but never deployed. Lesson: pipeline health monitoring must alert on promotion failures, not just build failures.

Monitoring Must-Haves

  • Alert on pipeline stage failure (not just final result)
  • Alert on image age > threshold per environment
  • Dashboard: last successful deploy date per service per environment
  • Compare image digests across environments (detect staleness)

Likely Interview Questions + How to Answer

Technical Leadership

"How would you prioritize: fixing flaky tests vs building new AI tools vs improving build speed?"

What they're really asking: Can you make trade-offs? Do you think about team impact vs effort?

Answer framework: "I'd prioritize based on (1) how many engineers are blocked right now, (2) severity of pain, and (3) effort to fix. Flaky tests usually win because they erode trust in CI — engineers start ignoring failures and real bugs slip through. Build speed is next because it affects every engineer every day. AI tools are force multipliers but only after the foundation is solid."

"A team is blocked because the device lab has no available devices. What do you do?"

Answer: "Immediate: check if any devices are stuck (health check failures, zombie jobs). Often 20% of devices are in MAINTENANCE state that nobody cleared. Medium-term: add priority lanes so release-blocking tests can preempt nightly runs. Long-term: track utilization metrics to justify budget for more devices."

"Your AI bug triage tool has 60% accuracy. Ship it or wait for 80%?"

Answer: "Ship it with a confidence threshold — only auto-assign when confidence > 90%. Below that, suggest but require human confirmation. 60% overall means the top-confidence predictions are probably 85%+. Let humans handle the uncertain ones. Measure: did we reduce average triage time? If yes, it's already providing value."

"How do you measure engineering productivity? What metrics?"

Answer: "I focus on cycle time (code push → test result), not lines of code. Key metrics: (1) P50/P90 time from push to green CI, (2) flaky test rate, (3) device utilization, (4) MTTR for regressions. I'd avoid vanity metrics like 'number of tests' — a team can write 1000 trivial tests that catch nothing."

People Management

"If you don't have domain expertise in this area, how do you manage the team?"

This exact question was asked in a past coffee chat. Key question for EM roles where you're new to the domain.

Answer: "I don't need to be the deepest domain expert on my team — that's what my senior engineers are for. My value is: (1) defining process and quality bar, (2) unblocking cross-team dependencies, (3) growing people, and (4) making sure we ship on time. Domain knowledge I can ramp up on — I've done it multiple times. At my current company I took over an IoT test system with zero context and used AI tools to build a full architectural understanding in half a day. The engineering judgment and people management skills are what take years to build and are portable across domains."

"How do you grow engineers on your team?"

Answer: "I match stretch assignments to their growth areas. For a junior engineer who's technically strong but struggles with ambiguity, I give them a project with unclear requirements and coach them through the scoping process. I had a junior engineer I coached from IC to engineering manager in 2 years — by progressively expanding their scope from owning a feature, to owning a service, to leading a small sub-team."

"How do you handle an underperformer?"

Answer: "First, diagnose whether it's a skill issue, motivation issue, or clarity issue. Most 'underperformance' is actually unclear expectations. I start with a direct conversation: 'here's what I expect at your level, here's where I see a gap.' Set specific, measurable goals for 2-4 weeks. Meet weekly to check progress. If it's a skill gap, provide resources and pairing. If after 4-6 weeks there's no improvement despite clear support, then we have a harder conversation about fit."

"Tell me about a time you gave difficult feedback"

Answer: "At a previous company, I had an engineer who was technically brilliant but was dismissive in code reviews — leaving comments like 'this is wrong' without explanation. Other engineers started avoiding his reviews. I pulled him aside and showed him specific examples: 'when you write this, here's how the other person feels. Your technical insight is valuable but the delivery is blocking collaboration.' He was surprised — he thought he was being efficient. We agreed on a format: always explain why, always suggest an alternative. Within a month his review style changed and the team started actively requesting his reviews again."

"How do you manage across multiple teams/locations/time zones?"

Answer: "Async-first communication: decisions documented in writing, not just in meetings. Clear ownership boundaries: each team owns a service end-to-end. Overlap hours for sync: find the 2-3 hour window where all locations are online and use it for decisions that need real-time discussion. Everything else goes through async channels with clear SLAs for response time."

AI/LLM

"How would you use LLMs to generate unit tests?"

Answer: "Feed the LLM: (1) the function signature + docstring, (2) 2-3 example tests from the same codebase for style reference, (3) coverage report showing untested branches. Ask it to generate tests for uncovered paths. Human reviews generated tests for: correct assertions (not just 'it doesn't crash'), meaningful edge cases, no false positives. Measure: did coverage increase? Did we catch any real bugs?"

"What are the risks of AI-generated test code?"

Answer: "Three main risks: (1) Tests that always pass — assert nothing meaningful, just exercise code without checking correctness. (2) Tests that encode current behavior as 'correct' — if the code has a bug, the AI generates a test that expects the buggy behavior. (3) Maintenance burden — generated tests may be brittle, breaking on any refactor. Mitigation: require human review, mutation testing to verify tests actually catch bugs, and prefer generated tests that test contracts/interfaces not implementation details."

"How do you handle LLM hallucination in an automated system?"

Answer: "Never trust LLM output for consequential actions without verification. Three layers: (1) Constrain output format — use function calling / structured output so the LLM can only emit valid actions. (2) Confidence gating — only auto-execute above threshold, human review below. (3) Reversibility — if the action is reversible (e.g., assign a bug), auto-execute. If irreversible (e.g., close a bug, deploy code), always require human approval."

Debugging / RCA

"Tests pass locally but fail in CI. How do you debug?"

Answer: "Classic categories: (1) Environment difference — different OS, different dependencies, different timezone/locale. (2) Ordering dependency — locally tests run in one order, CI shuffles. (3) Resource constraint — CI has less memory/CPU, causes timeouts. (4) Shared state — test pollutes DB/filesystem, locally it's fresh each run. My approach: reproduce in CI environment first (run in Docker matching CI image), then bisect which category it is."

"A pipeline has been silently broken for weeks. How do you detect and prevent this?"

Answer: "I've seen this firsthand — a container image pipeline was blocked for 2 months and nobody noticed because the running service used a cached image. Prevention: (1) Alert on pipeline stage failures, not just final build status. (2) Monitor artifact freshness — alert if deployed image is older than X days. (3) Compare artifact digests across environments — catch staleness early. (4) Weekly pipeline health dashboard reviewed in team standup. The key insight: monitor deployments, not just builds. A passing build that never deploys is invisible."

"How do you debug a device test that fails intermittently?"

Answer: "Three suspects: (1) Device hardware — run same test on multiple devices, if it only fails on one device → device issue. (2) Timing/race condition — add retries, check if failure correlates with concurrent tests on same device or network traffic. (3) Environmental — correlate with time of day, other running jobs, or infrastructure state. Collect: full device logs, host logs, network state at time of failure. The hardest bugs are 'once a day' failures — for those I instrument heavily and wait."

How to Measure and Record Impact (Without Being Self-Promotional)

Problem: "I can't quantify my impact because I didn't track metrics before/after."

Reality: You don't need before/after dashboards. You need to document the PROBLEM (what was broken), the ACTION (what you did), and the OUTCOME (what changed). Specific anecdotes with timestamps are more credible than made-up percentages.

What to Track Weekly (takes 5 min)

  • Unblocks: "Team X was stuck on Y, I connected them with Z" — date, who, what
  • Decisions: "Decided to prioritize A over B because C" — captures judgment
  • People moments: "Gave feedback to X about Y, their response was Z"
  • Shipped: "Feature Y launched, used by N teams"
  • Incidents: "Pipeline broke, RCA found X, prevention: Y"

How to Quantify Without Pre-existing Metrics

Instead ofSay
"Improved build speed""Engineers were complaining builds took 30+ min. After the change, the common case is under 10 min. I know because I asked 5 engineers and they confirmed."
"Reduced flaky tests""Before: team Slack had 3-4 flaky test complaints per day. After: about 1 per week. I tracked this for 2 weeks before and after."
"Improved team morale""Two engineers told me separately they feel more productive since we changed the code review process. One said they ship 2x faster."
"Unblocked the team""Firmware team was stuck for a month. I resolved the root cause in 2 days. They shipped their feature the same week."

Rule of thumb: If you can say "who was affected, how long was the problem, what's different now" — that's quantified enough. You don't need a Grafana dashboard for everything. Anecdotes with specifics (dates, names, durations) are credible because they can be verified.

Joining a New Team as EM (Without Domain Knowledge or Social Skills)

First 2 Weeks: Listen Only

  • 1:1 with every team member (use template questions below — no need to improvise)
  • Read their code, infra, dashboards (you can understand systems without people explaining)
  • Write down a pain points list (don't share yet, just absorb)
  • DO NOT change anything yet. Don't propose improvements. Don't suggest tools. Just listen and learn.

Week 3-4: Fix One Thing

  • Pick the #1 pain point everyone mentioned → fix it yourself or assign it as top priority
  • This single action builds more trust than 4 weeks of socializing
  • Engineers respect "this person fixed the thing we've been complaining about for months"

1:1 Template (for people who don't know how to make conversation)

You don't need to be chatty. Use these 4 questions and let them talk:

  1. "What are you working on this week? Anything blocking you?"
  2. "Is there anything I can do to help you move faster?"
  3. "What's the most frustrating part of your workflow right now?"
  4. "Where do you want to be in a year? What skills are you trying to build?"

That's it. 30 minutes. They talk 80%, you talk 20%. Take notes. Follow up on what they said next week. Consistency > charisma.

The introvert EM advantage: You won't micro-manage (engineers hate that). You won't waste meeting time with chatter. You'll give people space. You'll communicate in writing (which is clearer and more inclusive for remote/async teams). Engineers prefer "quiet but useful" over "energetic but unhelpful."

How to "Ramp Up Domain" Without Asking People to Teach You

  • Read the team's design docs and wikis (even if outdated — you'll know what questions to ask)
  • Use AI to analyze their codebase and generate architecture summaries
  • Look at recent incident reports / postmortems — fastest way to understand what matters
  • Read their dashboards — what do they monitor = what they care about
  • Attend their standups silently for 1 week before saying anything

Goal: Within 2 weeks, you should be able to draw the system architecture on a whiteboard without anyone teaching you. This earns instant respect.

How to Build a Roadmap When You're New (and Not from the Domain)

Problem: "I don't know the domain well enough to define what we should build next."

Answer: EngProd roadmaps are demand-driven, not vision-driven. You don't invent what to build — you listen to what hurts and fix it in priority order. This is actually easier than product roadmaps (where you have to predict what users want).

Step 1: Collect Pain Points (Week 1-2)

  • Ask every partner team: "What slows you down the most day-to-day?"
  • Ask your own team: "What support requests do you get most often?"
  • Look at Slack: search for "broken", "stuck", "help", "flaky" in engineering channels
  • Look at incident history: what keeps breaking?
  • Run a 5-question survey if you have 50+ engineers (keep it short)

Step 2: Categorize and Prioritize (Week 3)

PriorityCriteriaExample
P0: BrokenSomething is actively broken, teams are working around it dailyCI flakes 15% of runs, engineers re-trigger manually
P1: SlowWorks but painfully slow, every engineer feels itBuild takes 40 min, could be 10 with caching

Step 3: Frame as Roadmap (Week 4)

Q3 2026 Roadmap: Engineering Productivity

P0 (must-do):
  - Reduce CI flake rate from 15% to <3% [blocks every team]
  - Fix device lab scheduling deadlock [QA blocked 2x/week]

  - Build cache layer to cut build time 40min → 15min [200 engineers affected]
  - Unified test report dashboard [currently 3 separate tools]

The Trick: You Don't Need Domain Expertise for This

Notice: Nothing in the roadmap above requires knowing how the firmware works, what the device architecture is, or what the product features are. You just need to know: (1) what's broken, (2) how many people it affects, (3) how hard it is to fix. That's universal EngProd reasoning.

How to Present Roadmap to Leadership

  • Lead with the problem, not the solution: "200 engineers lose 30 min/day to flaky CI" not "we're implementing a retry mechanism"
  • Show impact in engineer-hours: "This saves 100 engineer-hours per week" → that's 2.5 FTEs worth of recovered time
  • Show what you're NOT doing and why: This proves you made priority calls, not just listed everything
  • Get buy-in from partner team leads: If they agree with your priorities, leadership will too

Hard Problem: "I can't tell what's important vs not"

You don't need to know the domain to prioritize. You need to know who's in pain and how much.

Prioritization formula without domain knowledge:

  • How many people affected? 200 engineers vs 3 engineers → obvious
  • How often? Daily pain vs once-a-month annoyance → obvious
  • What's the workaround cost? 30 min manual work every time vs 2 min retry → quantifiable
  • Is it getting worse? Growing test suite = flake rate grows too → urgent

If 3 different partner teams independently mention the same problem, it's P0. You don't need to understand WHY it's broken to know it's worth fixing.

Hard Problem: "I can't push back on low-priority requests"

Say no with data, not with opinion:

"I understand you need X. Here's our current priority:
 - P0: CI flaky rate blocks 200 engineers daily (100 hrs/day lost)
 - P1: Build time affects every push (saving 20 min × 50 pushes/day)
 - Your request: affects 3 engineers occasionally

I'm happy to do X in Q4, or if you'd like it sooner,
help me understand which P0/P1 item you'd swap it with."

Escalation path: If they insist and won't accept your prioritization, that's fine — escalate to both your managers. "We disagree on priority, can you two align?" This isn't a failure, it's the system working. Don't cave just to avoid conflict.

Hard Problem: "Team has no direction — just keeping things alive"

"Maintenance only" means nobody has looked for opportunities. This is actually the best situation for a new EM — you have a blank canvas.

Ask these questions to find direction:

  • "What do engineers do manually that could be automated?" → that's your roadmap
  • "What takes the longest in the development cycle?" → optimize it
  • "What tools does another team have that we don't?" → build or adopt
  • "What broke in the last 3 months?" → prevent it from happening again
  • "If you could wave a magic wand and fix one thing, what would it be?" → that's P0
  • "What would AI be good at in our workflow?" → future roadmap

If after all this you truly find nothing to improve — your team is overstaffed. But this never happens in EngProd. There's always more to automate, more to speed up, more to make self-service.

The real risk of "maintenance only" teams: They become invisible. Leadership forgets they exist. Budget gets cut. People leave because there's no growth. As EM, your job is to turn "keeping the lights on" into "enabling the org to ship faster" — same work, different framing, completely different perception.

System Design: Test Selection Engine

Problem: You have 4 million tests. A developer pushes a 10-line code change. Running all 4M tests takes 8 hours. How do you select the relevant subset to run in <10 minutes?

Why This Is the Hardest Problem in Test Automation

Too few tests → miss a bug → production incident.
Too many tests → 8 hour feedback loop → developers hate CI.
The sweet spot is: run exactly the tests that could possibly be affected by this change.

Approach 1: Dependency Graph (Static Analysis)

Idea: Build a graph: "file A is imported by file B, which is tested by test C." If file A changes, run test C.

  • How: Parse import statements / build dependencies (Bazel does this natively)
  • Pros: Deterministic, no training data needed, zero false negatives
  • Cons: Over-selects (interface change → everything depends on it → run thousands). Can't handle cross-service effects (service A's API change affects service B's integration test)

Approach 2: Historical Correlation (ML-based)

Idea: Historically, when file X changed, tests T1, T2, T3 failed. Learn this pattern. When X changes again, predict T1-T3 might fail.

  • How: Train model on (changed_files, failed_tests) pairs from last 6 months
  • Pros: Catches non-obvious dependencies, cross-service effects, config changes
  • Cons: Can miss new tests (no history). Can have false negatives (never seen this file fail this test before). Needs retraining.

Approach 3: Hybrid (Recommended for Interview)

For every code change:
1. ALWAYS run: dependency-graph-selected tests (guaranteed safety)
2. ALSO run: ML-predicted high-risk tests (catches cross-boundary effects)
3. SKIP: tests with zero dependency + zero historical correlation
4. NIGHTLY: run ALL tests as safety net (catch what we missed)
5. LEARN: if nightly catches something we skipped → retrain model

Architecture

Key Trade-offs (Discussion Points)

Trade-offOption AOption BRecommendation
Safety vs SpeedRun all (safe, slow)Run subset (fast, risk)Subset for presubmit + nightly full run as safety net
Graph granularityFile-level (coarse, over-selects)Function-level (precise, expensive to compute)File-level for most, function-level for hot paths
ML model freshnessRetrain daily (accurate)Retrain weekly (cheaper)Daily retrain, model is small (not expensive)
New tests (no history)Always run (safe)Skip until history buildsAlways run new tests for first 2 weeks

Real-World References

  • Google TAP: Uses build dependency graph (Bazel targets) + historical flake data to skip known-flaky
  • Meta TestPilot: ML-based predictive test selection, claims 50%+ reduction in test execution
  • Uber SubmitQueue: Optimistic merging — merge first, test after. If test fails, auto-revert. Radically different philosophy.

System Design: Build Speed Optimization

Problem: Firmware builds take 40 minutes. Developers get coffee, context-switch, forget what they were doing. How do you get it under 10 minutes?

Why Build Speed Matters (Numbers)

200 engineers × 5 builds/day × 30 min wasted per build = 500 engineer-hours/day wasted.
Cutting build time from 40 min to 10 min = saving 375 engineer-hours/day = ~9 FTEs worth of recovered time.

Optimization Strategies (in order of impact)

1. Remote Caching (biggest win)

  • Idea: If someone else already built the same target with the same inputs, reuse their output
  • How: Hash all inputs (source files + dependencies + compiler flags) → check remote cache → hit = skip build
  • Impact: 60-80% cache hit rate in practice → build goes from 40 min to 8-10 min
  • Tools: Bazel remote cache, ccache (C++), Gradle build cache, Turborepo (JS)

2. Incremental Builds

  • Idea: Only rebuild what actually changed (and its dependents)
  • How: Dependency graph + file modification timestamps → minimal rebuild set
  • Pitfall: C++ header files can trigger massive rebuilds (one .h change → recompile everything that includes it)
  • Fix: Forward declarations, pimpl pattern, module boundaries

3. Distributed Build (parallel across machines)

  • Idea: Split compilation across a cluster of build workers
  • How: Build graph has natural parallelism → ship independent targets to different machines
  • Tools: Bazel remote execution, distcc (C++), BuildBuddy, EngFlow
  • Limit: Critical path can't be parallelized (longest dependency chain)

4. Build Avoidance

  • Idea: Don't build what you don't need for this test
  • How: If test only needs library A, don't build library B even if it changed
  • Requires: Fine-grained dependency declaration (Bazel targets, not Makefile blobs)

5. Pre-built Dependencies (artifact registry)

  • Idea: Third-party deps and stable internal deps → pre-built binaries from registry
  • How: Only rebuild from source when dependency version changes
  • Tools: Maven/Nexus (Java), npm registry, Conan (C++), internal artifact stores

Metrics to Track

  • Build time P50/P90: Target <5 min / <15 min
  • Cache hit rate: Target >70% (below = cache invalidation too aggressive)
  • Build success rate: >95% (failures from infra, not code)
  • Developer wait time: Time from "save" to "test result" (includes build + test)

EngProd Domain Knowledge (Crash Course)

Flaky Tests

Definition: A test that sometimes passes and sometimes fails with the same code. Not a bug in the product — a bug in the test (or test environment).

Common Causes

CauseExampleFix
Timing / race conditionTest checks UI element before it renders; test depends on thread scheduling orderAdd explicit waits, use synchronization primitives, don't depend on timing
Shared stateTest A writes to DB, test B reads it — passes when A runs first, fails otherwiseIsolate test data: each test creates its own data, teardown after
Order dependencyTest assumes previous test set up some stateEach test must be independently runnable (setup from scratch)
External dependencyTest calls real API that's occasionally slow or downMock external services, use contract testing
Resource exhaustionTest passes alone but fails when 50 tests run in parallel (port conflict, memory)Unique ports per test, proper resource cleanup
Non-deterministic dataTest uses current timestamp, random values, or floating-point comparisonInject time/random seed, use epsilon comparison
Environment differencePasses on Mac, fails on Linux CI (path separator, locale, timezone)Standardize CI environment, test in container matching CI

How to Detect Flaky Tests

  • Re-run failed tests: If it passes on retry without code change → flaky
  • Track pass rate per test: If a test has 80% pass rate over last 20 runs → flaky
  • Correlation analysis: Fails only at certain time / only on certain machine / only with certain parallel tests

How to Handle Flaky Tests (as EM)

  1. Detect: Automated flaky detection (track pass/fail per test per commit)
  2. Quarantine: Move flaky test out of critical path (don't block merges/releases)
  3. Notify owner: Auto-assign bug to test owner with flake data
  4. SLA: Flaky tests must be fixed within 1 week or deleted
  5. Metrics: Track flaky rate as team KPI (target: <2%)

Why flaky tests are dangerous: Engineers start ignoring test failures ("probably just flaky"). Then a real bug slips through because nobody looked. Flaky tests erode trust in CI — once trust is gone, CI becomes useless.

CI/CD Concepts

Continuous Integration (CI)

  • Every code push triggers: build → lint → unit test → integration test
  • Goal: catch problems within minutes of commit, not days later
  • Key metric: time from push to green/red signal (target: <10 min for unit, <30 min for full)

Continuous Deployment (CD)

  • After CI passes: auto-deploy to staging → canary → production
  • Gate: only deploy if all tests pass + no open P0 bugs
  • Rollback: auto-rollback if error rate spikes after deploy

Test Automation Framework Concepts

What a Good Test Framework Provides

  • Test runner: Discover, execute, and report tests
  • Assertions: Verify expected vs actual results
  • Fixtures / Setup-Teardown: Prepare environment before, cleanup after
  • Mocking: Replace real dependencies with controllable fakes
  • Parameterization: Run same test with different inputs
  • Reporting: Pass/fail summary, failure details, screenshots/logs
  • Retry: Automatic retry for flaky-prone tests
  • Parallelization: Run tests concurrently to reduce total time

Code Coverage

Types

  • Line coverage: % of lines executed by tests (most common, easy to game)
  • Branch coverage: % of if/else branches taken (better signal)
  • Mutation testing: Inject bugs → check if tests catch them (best signal, expensive)

EM Perspective on Coverage

Coverage is a lagging indicator, not a goal. 80% coverage with meaningful assertions > 95% coverage with trivial tests. Don't mandate a number — mandate that every PR includes tests for the new logic. Review test quality in code review, not just coverage %.
