The one-line idea

BeyondProd is "zero-trust, but for production services instead of users." It replaces the old assumption — "if you're inside the network perimeter, you're trusted" — with: trust a service based on its code provenance, the hardware it runs on, and its cryptographic identity — never its IP address or hostname.

The explicit analogy Google draws: BeyondProd is to production services what BeyondCorp / Chrome Enterprise Premium is to users. For users, trust depends on device context, not on being on the corporate network. For services, trust depends on code provenance + hardware + identity, not on network location.

Why the perimeter model breaks for microservices

Google's own framing: "Microservices move and are deployed in different environments, across heterogeneous hosts, and operate at various levels of trust and sensitivity." Borg deploys several billion containers a week. A firewall around the datacenter can't express "service A may call service B but not C" for ephemeral workloads that reschedule constantly. So the trust decision has to move into every service-to-service call.

The one idea, stated carefully (easy to get backwards): the old model says "trust = you're inside the wall." BeyondProd doesn't say "trust everything regardless of where it's from" — it says the opposite: network location stops being a trust signal at all. A call isn't trusted because it came from inside; it's re-verified every time on its own merits — the caller's identity (cert), the provenance of the code it's running (built by the trusted pipeline?), and the hardware it sits on (verified boot). Location tells you nothing; identity + provenance + hardware do. "It came from inside the network" is worth exactly zero.

Jargon buster — read this first

BeyondProd is heavy on acronyms. Plain-English versions so nothing trips you up:

TermPlain meaning
authN (authentication)"Who are you?" — proving identity. Like badging in with your card.
authZ (authorization)"Are you allowed to do this?" — checking permission. Like whether your badge opens this room. (Written authN/authZ because both start with "Auth…".)
code provenanceThe origin / pedigree of the running code — can you prove this binary came from the trusted pipeline and was reviewed, not slipped in by someone? Like a food supply-chain trail.
BAB (Binary Authorization for Borg)The deploy-time gate that enforces code provenance: only binaries with a signed "build manifest" (2nd-engineer review + built on trusted infra) may launch. Borg = Google's container scheduler (Kubernetes' ancestor).
EUC ticket (End-User Context)A pass that says "this request is on behalf of user alice", forwarded between services. Separate from service identity ("which service is calling"). Like a visitor sticker vs. an employee badge.
confused deputyThe bug EUC tickets prevent: a service that has permission is tricked into doing something for a user who doesn't. Keeping "who is calling" and "on whose behalf" separate stops it.

GFE (Google Front End)The edge proxy every external request hits first — terminates TLS, routes, load-balances, absorbs DoS before traffic reaches backends.

How they chain in one request: authN (ALTS proves which service is calling) → provenance/BAB (the code running is known-good) → EUC ticket (which real user this is for) → authZ (is that user actually allowed — possibly answered by Zanzibar). All must pass, or the request is denied.

How service identity & trust are established

The chain of trust runs from silicon up to an authorized RPC:

  • Each machine's ALTS credential is provisioned via the host integrity system and is only decryptable if secure boot was verified — so only verified-boot hosts can run workloads.
  • Borg Prime (the centralized controller) grants microservice ALTS credentials based on microservice identity, using the machine-level credential as the secure channel.

First, clear up: identity vs. code vs. provenance

These three get muddled easily — and BeyondProd's whole "trust by identity + provenance" idea only makes sense once they're separate. Note "code" here means the actual software/binary running on the machine — not an ID or a password.

ThingAnswersEveryday analogy
identity"Who is calling?" — which service (or which user)the name on your badge
code / binaryThe actual program currently running on that machinethe tools/cargo you're carrying
provenance"Where did that program come from — is it from the trusted pipeline, reviewed, not slipped in?"the supply-chain / inspection sticker on the cargo

Why identity alone isn't enough: a service can have a perfectly real identity (ALTS proves "I'm the Photos backend" ✅) but be running a tampered binary that nobody reviewed. Identity-only trust would let it through — the name is genuine. So BeyondProd requires both: a trusted identity and known-provenance code, before it ever gets to "is this allowed?" (authZ).

So "how identity + provenance become an authorization model" just means: combine who is calling (identity) with what code is running (provenance) as two trust signals, then decide whether to allow the action (authZ). Badge name + inspected toolbox, both checked, before you're let into the server room.

Binary Authorization for Borg (BAB) — provenance you can't fake

BAB is a deploy-time enforcement check ensuring code meets security requirements before it can run. It enforces:

  • Second-engineer code review before submission,
  • Binaries verifiably built on dedicated infrastructure,
  • Blocking deployment of unauthorized microservices.

Flow: approved change → trusted build system produces a package with a signed, verifiable build-manifest certificate → BAB validates that certificate at deploy time. So "what's running" is provably "what was reviewed and built by the trusted pipeline."

Key insight: BAB turns "we trust this binary" into a cryptographic fact anchored in the build system, not a hope. Combined with immutable containers (they don't change after deploy), you know exactly what code is running and where it came from.

Ingress, containers, and isolation

  • Google Front End (GFE) is the first point of entry: terminates TLS, enforces TLS best practices, routes to the right microservice, load-balances, reroutes across regions, and defends against DoS. Cloud Front End is a GFE variant on the Compute Engine network stack.
  • Containers on Borg are immutable — they don't change after deployment and are rebuilt/redeployed frequently. Borg tooling does blue-green deployments, migrating traffic gradually with no downtime.
  • gVisor is a user-space kernel that isolates workloads sharing an OS by intercepting syscalls — shrinking the host attack surface, especially for untrusted external code.

EUC (End-User Context) tickets are integrity-protected, centrally-issued, forwardable credentials that attest to the end user's identity — kept separate from service identity so the two concerns don't get conflated.

Why this design

Why abandon the perimeter?

A perimeter gives you one wall and total trust inside it — so a single breach is catastrophic, and it can't express fine-grained "who may talk to whom" for workloads that move constantly. BeyondProd assumes no inherent mutual trust between services, so every call is authenticated and authorized on its own merits. The blast radius of any single compromise is bounded.

Why separate service identity from end-user identity?

A backend needs to know two different things: "which service is calling me?" (ALTS identity) and "on whose behalf?" (EUC ticket). Conflating them is a classic confused-deputy setup. Keeping them separate lets a service prove its own identity and forward the user's context without impersonation.

Why enforce provenance at deploy time (BAB)?

Authentication tells you who is running; provenance tells you what is running and whether it came from the trusted pipeline. Without BAB, a valid service identity could still run unreviewed or maliciously-built code. BAB closes that gap by making "known-provenance code" a precondition for deployment.

Why root everything in hardware?

The whole chain is only as trustworthy as its anchor. If the machine credential could be extracted from a compromised host, service identity would be forgeable. Tying the machine credential's decryptability to verified secure boot (via Titan) means a tampered host simply can't obtain a working identity.

Scale & throughput — per-call authZ that's near-free

BeyondProd authorizes every service-to-service call — cert validity + revocation + service-access policy + EUC ticket — for a fleet that deploys billions of containers a week. For that to be viable, each check must be almost free:

  • Built on ALTS's cheap identity. Mutual auth is already amortized by ALTS (resumption, long-lived connections), so BeyondProd's authorization rides on top without re-paying for handshakes.
  • Policy & CRL evaluated locally. Service-access policy and revocation lists are distributed to machines and checked at verification time — no central authorization call on every RPC.
  • Identity, not location, enables elasticity. Because trust binds to service identity (not host/IP), workloads replicate, load-balance, and reschedule freely — the security model doesn't fight autoscaling.
  • Edge absorbs the crowd. GFE terminates TLS, load-balances, reroutes across regions, and absorbs DoS at the edge, so backends only see already-filtered, already-routed traffic.
  • Immutable containers + blue-green. Immutable images and gradual traffic migration let the fleet roll out constantly without downtime — provenance (BAB) is checked once at deploy, not per request.

The scaling lesson: push authorization checks to local, cached data evaluated at the point of use, bind trust to identity so security doesn't block elasticity, and let the edge shed load before it reaches backends.

Hard problems

Migrating an existing fleet off the perimeter

Google didn't start zero-trust; it retrofitted a massive existing production estate. Doing that without breaking billions of RPCs means making the new controls default and invisible (ALTS zero-config) and rolling them out gradually — the same lesson as BeyondCorp for users.

Performance of per-call authorization

Checking cert validity + revocation + service policy + EUC ticket on every RPC, at Google scale, must be near-free. This is why ALTS resumption, local CRL copies, and cached policy exist — the security checks have to disappear into the noise.

Trust bootstrapping (the chicken-and-egg)

A workload needs an identity to be trusted, but issuing that identity requires a trusted channel. Resolved by layering: hardware (Titan) → machine credential (needs verified boot) → workload credential (issued by Borg Prime over the machine-secured channel).

Isolating untrusted / multi-tenant code

Running customer or untrusted code next to Google services on shared hosts is dangerous at the kernel-syscall level. gVisor addresses this with a user-space kernel, but sandboxing always trades some performance/compatibility for a smaller attack surface — a deliberate, ongoing tension.

Interview angle

Why this matters: BeyondProd is the umbrella story that ties ALTS, Titan, and KMS together. For a Security Backend / Silicon role, being able to narrate "how a request stays secure from silicon to authorized RPC" is exactly the systems-thinking they want.

Likely questions

Q: "Design security for a microservice platform where you can't trust the network."

Service identity (ALTS) not host; mutual auth on every call; hardware-rooted machine identity + verified boot; deploy-time provenance (BAB); separate service identity from end-user context (EUC); edge termination (GFE); least-privilege service access management. Then discuss performance and gradual migration.

Q: "How is this different from a firewall / VPN?"

Perimeter = one wall, total interior trust, coarse and static. Zero-trust = per-call authN/authZ, no interior trust, fine-grained and dynamic — bounds blast radius and handles ephemeral workloads.

Q: "How do you know the code running is the code you reviewed?"

Trusted build system emits a signed build-manifest certificate; BAB validates it at deploy; immutable containers; second-engineer review as a gate. Provenance becomes a cryptographic fact.

Q: "Service A shouldn't be able to act as user X unless user X called it — how?"

Separate identities: ALTS proves service identity; EUC ticket (integrity-protected, forwardable) carries end-user context; backend verifies both independently. Prevents confused-deputy.

Signature senior move: narrate the full data-access flow (User → GFE → ALTS → EUC → backend verifies four things → deny on any failure), then generalize it to the operating principle — don't trust the reported/perimeter state; verify identity and provenance at the point of use.
