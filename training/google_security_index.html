How to use this guide

Five of Google's public security systems, each in its own deep-dive: what it is, where it's used, how it's designed, why that way, its hard problems, and how to reason about it in an interview. They're best read as one coherent story, not five isolated papers. Start with the one-diagram map below, then read in the suggested order.

Each doc closes with an Interview angle tab — reusable talking points and likely questions, framed as general principles you can reason from.

The system-of-systems map

These aren't five separate things — they stack. Hardware anchors machine identity; identity secures every RPC; a policy layer authorizes services and verifies their code; the data those services touch is encrypted with keys rooted in the same hardware/quorum discipline; and on top, a global service decides — at planet scale — whether a given user is actually allowed to touch a given object.

The through-line: "don't trust the network — trust hardware-rooted identity + provenance, and encrypt everything." Titan makes machine identity real; ALTS turns it into per-RPC service identity; BeyondProd authorizes services and verifies provenance; KMS keeps the data safe; Zanzibar decides whether the end user is allowed to touch the object — at planet scale. Being able to narrate this stack top-to-bottom is a staff-level signal.

The five deep-dives

SYSTEM 01 · start here

Titan — Hardware Root of Trust

Google's custom secure chip. Secure boot via SPI-bus interposition, per-chip key at fabrication, offline quorum CA, remote attestation, firmware-identity-hashed-into-keys. The foundation the other three build on — read this first.
SYSTEM 02

ALTS — Application Layer Transport Security

Mutual auth + encryption for ~1010 internal RPCs/sec. Identity over hostname, Protobuf certs, 1-RTT DH handshake, boundary-aware encryption modes, session resumption, CRL revocation. The connective tissue of Google's production security.
SYSTEM 03

BeyondProd — Zero-Trust for Production Services

"BeyondCorp for services." Trust by code provenance + hardware + identity, not network location. Borg, Binary Authorization for Borg (BAB), GFE ingress, EUC tickets, gVisor. Ties ALTS + Titan into a coherent zero-trust model.
SYSTEM 04

Key Management & Encryption at Rest

Cloud KMS + Keystore. Envelope encryption (DEK wrapped by KEK), the master-key wrapping chain to an offline root, software/HSM/external protection levels, rotation without rewriting petabytes. A canonical key-hierarchy design.
SYSTEM 05 · security + backend scale

Zanzibar — Global Authorization at Scale

Google's planet-scale permission system: relation tuples (ReBAC), userset rewrite rules, zookies for consistency (the "new enemy problem"), Spanner, Leopard indexing, hot-spot handling, request hedging. >10M QPS, <10ms p95. The one doc that is both a security system and a large-scale backend — highest interview ROI.
BRIDGE NOTE · optional

Content Protection & Broadcast Encryption (AACS ↔ Titan)

Beginner primer: symmetric vs asymmetric/ECC, how Blu-ray/AACS lets many device keys open one disc (Media Key Block + subset-difference tree + revocation), and why "shared software key" is fragile vs "per-device hardware key." A career bridge from content-protection key management to hardware roots of trust.

Suggested reading order

  1. Titan — start here; it's the hardware root of trust everything else anchors on.
  2. ALTS — how hardware identity becomes service identity on every RPC.
  3. BeyondProd — how identity + provenance become an authorization model.
  4. KMS — how the data those services touch is protected, via a layered key hierarchy.
  5. Zanzibar — how authorization ("is this user allowed?") is served at planet scale with strong consistency. The security-meets-backend-scale capstone.
