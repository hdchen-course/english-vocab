The core idea: envelope encryption

Google doesn't encrypt petabytes of data with one key. Instead it uses envelope encryption: a Data Encryption Key (DEK) encrypts the actual data chunks, and a Key Encryption Key (KEK) encrypts ("wraps") the DEK. The wrapped DEK can then be stored right next to the data it protects, because it's useless without the KEK — which lives in a separate, hardened key-management service.

Mental model: the data is in a locked box (DEK). The DEK is a small key you can leave taped to the box — but it's itself inside a smaller locked box (KEK). Only the key-management service (Keystore / Cloud KMS) holds the key to that box, and that service's own master key is guarded by a chain that bottoms out at an offline root.

Two related systems

  • Keystore — Google's internal key-management service, the root of trust for encryption at rest across Google's own storage systems (Spanner, Bigtable, Colossus, etc.).
  • Cloud KMS — the customer-facing product on Google Cloud. It uses the same root of trust as Keystore, and lets customers control their KEKs across software, HSM, and external backends. Key material never leaves the Cloud KMS system boundary.

Plain-English versions of the key-management terms used below:

DEK (data encryption key)The key that actually encrypts a chunk of your data.

The master-key wrapping chain (root of trust)

This is the part interviewers love, because it's turtles-all-the-way-down until you hit an offline root. Walking from the data up:

How the wrapping differs by protection level

LevelWrapping chain above the KEK
SoftwareKMS master key (per location) encrypts the KEK → re-encrypted periodically by the Keystore master key → protected by the root Keystore master key.
HardwareA location-specific HSM wrapping key encrypts the KEK, and that HSM wrapping key is then encrypted with the KMS master key.
ExternalAn EKM wrapping key encrypts the wrapped DEK, then is itself encrypted with the KMS master key.

KMS servers fetch the KMS master key at startup as a hard dependency and refresh it daily.

Encryption at rest, end to end

  1. Data is chunked; each chunk gets a DEK.
  2. The DEK is wrapped by a KEK and stored with the data.
  3. To read, the storage system asks the key service to unwrap the DEK (authorized by IAM), decrypts the chunk, and discards the plaintext DEK.
  4. Key material is always encrypted at rest and in transit, decrypted only during customer use or Google's internal rotation/integrity checks.
  5. Client↔server RPC is secured by ALTS; customer↔GFE by TLS.

Datastore protection nuance: key material is encrypted at the application layer before it hits storage — so even engineers with datastore access never see plaintext customer key material. Metadata is authenticated with HMAC and scanned hourly for integrity (verify HMAC + successful decryption).

Rotation & destruction

  • A new key version = new key material. Autokey rotates yearly; the Keystore master key has a ciphertext max age of 90 days; KMS refreshes master keys every 24h and re-encrypts all customer keys monthly.
  • Rotation behavior is product-dependent: Spanner re-encrypts the DEK; Cloud Storage encrypts only new resources; BigQuery doesn't auto-rotate table keys.
  • Destruction: a default 30-day Scheduled for destruction state precedes logical deletion. Keys/rings/versions themselves can't be deleted — only version material is destroyed.

Why this design

Why envelope encryption instead of one big key?

  • Blast radius. A DEK protects one chunk; compromising it doesn't expose everything. The high-value KEK is used rarely and never leaves the boundary.
  • Cheap rotation. Rotating a KEK means re-wrapping DEKs, not re-encrypting petabytes. You get "rotate the key" semantics without rewriting the data.
  • Locality. The wrapped DEK travels with the data, so the hot path only calls the key service to unwrap a small blob — not to stream data through it.

Why a multi-level master-key chain?

Each layer exists so that no single key and no single human is the whole game. The root Keystore master key can't be accessed by any individual; the KMS master key is refreshed daily; customer keys are re-wrapped monthly. Compromising one layer still leaves the layer above it protecting the secret — defense in depth applied to keys themselves.

Why three protection levels?

Different customers have different threat models and compliance needs. Software (FIPS L1) is the cheap default; HSM (FIPS L3) guarantees key material is never seen in plaintext by software; External/EKM lets a customer hold the ultimate key entirely off Google's cloud. Same envelope model, different root anchor.

Why "key material never leaves the boundary"?

It reduces the problem of "protect the data everywhere" to "protect the key service." As long as the KMS/Keystore boundary holds and no employee can read plaintext key material, the data at rest is safe even if storage media are physically stolen.

Scale & throughput — keeping the key service off the hot path

If every read of encrypted data had to call the key service, that service would be a global bottleneck and a hard availability dependency. The envelope model is specifically shaped to avoid that:

  • The wrapped DEK travels with the data. The hot path decrypts a chunk using a DEK stored next to it — the key service is only needed to unwrap a small blob, never to stream data through it.
  • Rotation re-wraps keys, not petabytes. Rotating a KEK/master key re-encrypts other keys, not the underlying data — so "rotate the key" is cheap regardless of data volume (KMS re-encrypts customer keys monthly, refreshes master keys every 24h).
  • Master key cached, bounded staleness. KMS fetches the master key at startup and refreshes daily; the Keystore master key has a 90-day ciphertext max age with a 1-day client cache — reads hit local cache, not a central call each time.
  • HSM only where required. FIPS-L3 HSM operations are more expensive, so they back only the keys that must never appear in software plaintext; the software tier (BoringSSL) handles the cheap common case.

The scaling lesson: keep the expensive, centralized resource (the key service) off the per-request hot path — store a cheap wrapped credential next to the data, unwrap rarely, cache the roots with bounded staleness, and make rotation a key-level not data-level operation.

Hard problems

Availability of the key service is a hard dependency

If KMS/Keystore is down, nothing can decrypt. KMS fetches the master key at startup as a hard dependency and refreshes daily — the design must make the key service extremely available and its master key highly cached-yet-fresh (90-day ciphertext max age, one-day client cache).

Rotation without rewriting the world

Re-encrypting petabytes on every rotation is impossible, so rotation happens at the key-wrapping layer (re-wrap DEKs, re-encrypt customer keys monthly). The subtlety: different products (Spanner vs GCS vs BigQuery) can only rotate what they can afford to rewrite, so rotation semantics deliberately vary.

Integrity, not just secrecy

An attacker who can tamper with a wrapped DEK or metadata could cause silent corruption. Hence HMAC-authenticated metadata and hourly integrity scans (verify HMAC + successful decryption) — catching tampering proactively, not on next read.

Keeping humans out

The hardest guarantee: no Google employee has access to unencrypted customer key material, and the master key can't be accessed by any individual. That requires the application-layer encryption of key material, quorum/root controls, Access Transparency, and Key Access Justifications — organizational and cryptographic controls working together.

Interview angle

Why this matters: "encryption at rest" and "key hierarchy" are among the most common security-backend system-design prompts, and the envelope + master-key-chain model is the canonical answer. The key move is being able to reason cleanly about why each layer exists and how the chain bottoms out at an offline root.

The strong framing: envelope encryption applies the same "hardware-rooted key hierarchy" idea to data at rest that hardware roots of trust apply to device identity — a data key wrapped by a key-encryption key, wrapped by a master key, anchored at an offline root. Same pattern, different asset.

Likely questions

Q: "Design encryption at rest for a large storage system."

DEK-per-chunk, wrap with KEK, store wrapped DEK with data, KEK in a hardened key service, master-key chain to an offline root. Then discuss rotation at the wrapping layer, availability of the key service, and integrity (HMAC + scans).

Q: "How do you rotate keys without re-encrypting all the data?"

Rotate/re-wrap at the KEK/master layer; only new writes use new DEKs; product-specific choices for whether to re-encrypt existing data.

Q: "How do you ensure even insiders can't read customer keys?"

Application-layer encryption of key material before storage, root key inaccessible to any individual, quorum controls, Access Transparency, Key Access Justifications.

Q: "Software vs HSM vs external keys — when would you pick each?"

Map to threat model / compliance: default software; HSM when key material must never be in software plaintext (FIPS L3); external/EKM when the customer must physically hold the root off-cloud.

System-of-systems close: "KMS protects the data; ALTS protects the RPCs that carry key-unwrap requests; the machines running KMS are rooted in Titan; BeyondProd authorizes which services may call KMS at all." Tie it together.
