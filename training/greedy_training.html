Greedy Algorithms — Training Guide

Proof Techniques • Interval Scheduling • Exchange Argument • Activity Selection

When Does Greedy Work?

The Greedy Paradigm

A greedy algorithm makes the locally optimal choice at each step, hoping that this leads to a globally optimal solution. Unlike DP, greedy never reconsiders previous choices.

Two Required Properties

  1. Greedy Choice Property — A globally optimal solution can be arrived at by making locally optimal (greedy) choices.
  2. Optimal Substructure — An optimal solution to the problem contains optimal solutions to sub-problems.

Signal Words That Suggest Greedy

High probability greedy when you see: "minimum number of ...", "maximum number of non-overlapping ...", "earliest/latest deadline", "lexicographically smallest", "assign resources optimally", "schedule tasks", "minimum cost to ..."

Decision Tree: Greedy vs DP vs Backtracking

Q1: Does each local choice lock in without needing future info? └— Yes → Likely Greedy └— No → Q2: Does the problem have overlapping subproblems? └— Yes → Dynamic Programming └— No → Q3: Do you need to enumerate all solutions? └— Yes → Backtracking └— No → Divide & Conquer / Other

Proof Techniques for Greedy Algorithms

Why Proofs Matter in Interviews

At Google L5, interviewers expect you to justify why your greedy approach is correct. You do not need a formal proof, but you must articulate the key insight. The three main techniques are:

1. Exchange Argument (Most Common)

Assume an optimal solution O differs from your greedy solution G. Show you can swap an element in O with the greedy choice and the result is no worse. This implies G is also optimal.

Template: "Suppose OPT makes a different choice at step k. If we swap OPT's choice with our greedy choice, the solution does not get worse because [reason]. Therefore, greedy is also optimal."

Concrete Example: Activity Selection (Sort by End Time)

Claim: Choosing the activity with the earliest finish time that is compatible with previously selected activities yields a maximum-size set.

Proof sketch:

  1. Let G = {g1, g2, ..., gk} be greedy's solution (sorted by finish time).
  2. Let O = {o1, o2, ..., om} be an optimal solution with m ≥ k (sorted by finish time).
  3. Exchange step: Consider the first index i where gi ≠ oi. Since greedy picks the earliest finish time compatible activity, finish(gi) ≤ finish(oi).
  4. Replace oi with gi in O. Since finish(gi) ≤ finish(oi), all subsequent activities in O remain compatible.
  5. The new O is still valid and has the same size. We can repeat until O = G.
  6. Therefore m = k: greedy is optimal.

2. Greedy Stays Ahead (Induction)

Show by induction that after each step, the greedy solution is at least as good as any other solution at that point.

Structure: Define a measure M(i) after step i. Prove M_greedy(i) ≥ M_other(i) for all i. Conclude greedy's final answer is at least as good as optimal.

3. Structural / Cut-and-Paste

Show that any optimal solution can be transformed into the greedy solution by a series of local modifications, each preserving optimality.

Often used for problems like Huffman coding where the structure (tree shape) matters.

Interview Communication Pattern

// In interview, say something like:
// "The key insight is that choosing [greedy criterion] is safe because
//  if there were a better choice X, we could swap X with our choice
//  and the solution would not improve, because [specific reason]."
//
// This shows you understand correctness without writing a formal proof.

Interval Scheduling

Pattern 1: Maximum Non-Overlapping Intervals

Strategy: Sort by end time. Greedily pick the interval that ends earliest and is compatible with the last picked interval.

// Template: Maximum non-overlapping intervals
public int maxNonOverlapping(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[1] - b[1]); // sort by end time
    int count = 0, prevEnd = Integer.MIN_VALUE;
    for (int[] interval : intervals) {
        if (interval[0] >= prevEnd) { // compatible
            count++;
            prevEnd = interval[1];
        }
    }
    return count;
}
// LC 435: eraseOverlapIntervals = n - maxNonOverlapping

Pattern 2: Minimum Intervals to Cover All Points

Strategy: Sort by start time. For each uncovered point, greedily extend coverage as far as possible using available intervals.

// Template: Minimum intervals to cover a range [0, T]
public int minIntervalsToCover(int[][] intervals, int T) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    int count = 0, i = 0, curEnd = 0;
    while (curEnd < T) {
        int farthest = curEnd;
        while (i < intervals.length && intervals[i][0] <= curEnd) {
            farthest = Math.max(farthest, intervals[i][1]);
            i++;
        }
        if (farthest == curEnd) return -1; // gap, impossible
        count++;
        curEnd = farthest;
    }
    return count;
}

Pattern 3: Minimum Number of Arrows (Interval Intersection)

// LC 452: Minimum Number of Arrows to Burst Balloons
public int findMinArrowShots(int[][] points) {
    Arrays.sort(points, (a, b) -> Integer.compare(a[1], b[1]));
    int arrows = 1;
    int prevEnd = points[0][1];
    for (int i = 1; i < points.length; i++) {
        if (points[i][0] > prevEnd) { // no overlap
            arrows++;
            prevEnd = points[i][1];
        }
    }
    return arrows;
}

Meeting Rooms Pattern (Sweep Line / Heap)

// LC 253: Meeting Rooms II - minimum rooms needed
public int minMeetingRooms(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    PriorityQueue<Integer> pq = new PriorityQueue<>(); // min-heap of end times
    for (int[] interval : intervals) {
        if (!pq.isEmpty() && pq.peek() <= interval[0]) {
            pq.poll(); // reuse room
        }
        pq.offer(interval[1]);
    }
    return pq.size();
}

Key insight: "Sort by end time" maximizes non-overlapping count. "Sort by start time + heap" counts concurrent overlaps. Know both patterns cold.

Jump & Coverage Greedy

Core Pattern: Track Farthest Reachable

At each position, maintain the farthest reachable index. When you reach the boundary of the current jump, you must take another jump (increment count) and extend boundary to farthest.

// Template: Minimum jumps to reach end (BFS-style greedy)
public int minJumps(int[] nums) {
    int jumps = 0, curEnd = 0, farthest = 0;
    for (int i = 0; i < nums.length - 1; i++) {
        farthest = Math.max(farthest, i + nums[i]);
        if (i == curEnd) { // must jump
            jumps++;
            curEnd = farthest;
            if (curEnd >= nums.length - 1) break;
        }
    }
    return jumps;
}

Can Reach (Jump Game I)

// LC 55: Jump Game - can you reach the end?
public boolean canJump(int[] nums) {
    int farthest = 0;
    for (int i = 0; i < nums.length; i++) {
        if (i > farthest) return false;
        farthest = Math.max(farthest, i + nums[i]);
    }
    return true;
}

Video Stitching / Minimum Taps

// LC 1024: Video Stitching / LC 1326: Minimum Taps
// Convert to intervals, then use "min intervals to cover [0, T]" template
public int videoStitching(int[][] clips, int time) {
    Arrays.sort(clips, (a, b) -> a[0] - b[0]);
    int count = 0, i = 0, curEnd = 0;
    while (curEnd < time) {
        int farthest = curEnd;
        while (i < clips.length && clips[i][0] <= curEnd) {
            farthest = Math.max(farthest, clips[i][1]);
            i++;
        }
        if (farthest == curEnd) return -1;
        count++;
        curEnd = farthest;
    }
    return count;
}

Why it works: At each "level" (BFS analogy), we explore all reachable positions and extend to the farthest. This is essentially BFS on implicit graph where edges are jumps, but implemented in O(n) with the greedy trick.

Two-Pass Greedy

Core Pattern

When the answer at position i depends on both neighbors (left and right), make two passes:

  1. Left-to-right: Satisfy constraints from the left neighbor.
  2. Right-to-left: Satisfy constraints from the right neighbor.
  3. Combine: Take the max (or merge) of both passes.

Candy (LC 135)

// LC 135: Candy - minimum candies satisfying both neighbor constraints
public int candy(int[] ratings) {
    int n = ratings.length;
    int[] candies = new int[n];
    Arrays.fill(candies, 1);

    // Left-to-right: if rating[i] > rating[i-1], give more than left
    for (int i = 1; i < n; i++) {
        if (ratings[i] > ratings[i - 1]) {
            candies[i] = candies[i - 1] + 1;
        }
    }

    // Right-to-left: if rating[i] > rating[i+1], must be more than right
    for (int i = n - 2; i >= 0; i--) {
        if (ratings[i] > ratings[i + 1]) {
            candies[i] = Math.max(candies[i], candies[i + 1] + 1);
        }
    }

    int sum = 0;
    for (int c : candies) sum += c;
    return sum;
}

Trapping Rain Water (LC 42)

// LC 42: Trapping Rain Water - two-pointer approach (optimized two-pass)
public int trap(int[] height) {
    int left = 0, right = height.length - 1;
    int leftMax = 0, rightMax = 0, water = 0;
    while (left < right) {
        if (height[left] < height[right]) {
            leftMax = Math.max(leftMax, height[left]);
            water += leftMax - height[left];
            left++;
        } else {
            rightMax = Math.max(rightMax, height[right]);
            water += rightMax - height[right];
            right--;
        }
    }
    return water;
}

Gas Station (LC 134)

// LC 134: Gas Station - single pass greedy
public int canCompleteCircuit(int[] gas, int[] cost) {
    int totalSurplus = 0, currentSurplus = 0, start = 0;
    for (int i = 0; i < gas.length; i++) {
        int diff = gas[i] - cost[i];
        totalSurplus += diff;
        currentSurplus += diff;
        if (currentSurplus < 0) { // cannot start from any station before i+1
            start = i + 1;
            currentSurplus = 0;
        }
    }
    return totalSurplus >= 0 ? start : -1;
}

Gas Station insight: If total gas ≥ total cost, a solution exists. The start point is the station after the last point where cumulative surplus went negative. This is a "reset" greedy pattern.

Sort + Greedy

Core Idea

Many greedy problems become straightforward once you sort by the right criterion. The sort exposes the greedy order, and a single scan produces the answer.

Sorting criterion is everything. If you sort by the wrong key, greedy fails. Always ask: "What should I sort by, and why does this ordering let me make irrevocable decisions?"

Task Scheduler (LC 621)

// LC 621: Task Scheduler - greedy by most frequent task
public int leastInterval(char[] tasks, int n) {
    int[] freq = new int[26];
    for (char t : tasks) freq[t - 'A']++;
    Arrays.sort(freq);
    int maxFreq = freq[25];
    int idleSlots = (maxFreq - 1) * n;

    // Fill idle slots with other tasks
    for (int i = 24; i >= 0 && freq[i] > 0; i--) {
        idleSlots -= Math.min(freq[i], maxFreq - 1);
    }
    idleSlots = Math.max(0, idleSlots);
    return tasks.length + idleSlots;
}

Boats to Save People (LC 881)

// LC 881: Boats to Save People - sort + two pointers
public int numRescueBoats(int[] people, int limit) {
    Arrays.sort(people);
    int left = 0, right = people.length - 1, boats = 0;
    while (left <= right) {
        if (people[left] + people[right] <= limit) {
            left++; // pair lightest with heaviest
        }
        right--; // heaviest always takes a boat
        boats++;
    }
    return boats;
}

Queue Reconstruction by Height (LC 406)

// LC 406: Sort by height desc, then by k asc. Insert at index k.
public int[][] reconstructQueue(int[][] people) {
    // Taller people first; among same height, fewer people-in-front first
    Arrays.sort(people, (a, b) ->
        a[0] != b[0] ? b[0] - a[0] : a[1] - b[1]);
    List<int[]> result = new ArrayList<>();
    for (int[] p : people) {
        result.add(p[1], p); // insert at index k
    }
    return result.toArray(new int[0][]);
}

Why height-desc works: When we insert a shorter person, they do not affect the k-value of already-placed taller people. This is a "process in order of decreasing influence" pattern.

Smallest Non-Constructible Sum (CSES: MissingCoinSum)

Problem: Given coins, find the smallest sum you CANNOT make.
Key insight: Sort coins. If you can make all sums in [0, reach], and next coin ≤ reach+1, then you can now make [0, reach+coin]. If next coin > reach+1, answer is reach+1.

// Sort + greedy: O(n log n)
Arrays.sort(coins);
long reach = 0;
for (int coin : coins) {
    if (coin > reach + 1) break;
    reach += coin;
}
return reach + 1;  // smallest non-constructible

Exchange Argument Problems

The Pattern

Design a custom comparator that defines the optimal ordering. Prove via exchange argument: if two adjacent elements are in the wrong order, swapping them improves (or does not worsen) the solution.

Template: For elements A and B, compare f(A, B) vs f(B, A). If placing A before B is always at least as good, sort by this comparator. The proof shows any inversion can be fixed by swapping.

Largest Number (LC 179)

// LC 179: Largest Number
// Exchange argument: compare "AB" vs "BA" as strings
public String largestNumber(int[] nums) {
    String[] strs = new String[nums.length];
    for (int i = 0; i < nums.length; i++) strs[i] = String.valueOf(nums[i]);

    // If "ab" > "ba", then a should come before b
    Arrays.sort(strs, (a, b) -> (b + a).compareTo(a + b));

    if (strs[0].equals("0")) return "0";
    StringBuilder sb = new StringBuilder();
    for (String s : strs) sb.append(s);
    return sb.toString();
}
// Proof: If we have ...X Y... and "YX" > "XY", swapping gives a larger number.
// So the optimal arrangement has no inversions under this comparator = sorted order.

Minimum Initial Energy (LC 1665)

// LC 1665: Minimum Initial Energy to Finish Tasks
// Each task [actual_i, minimum_i]: need energy >= minimum_i, spend actual_i
// Exchange argument: sort by (minimum_i - actual_i) descending
public int minimumEffort(int[][] tasks) {
    // Sort by (minimum - actual) descending = sort by "surplus needed" desc
    Arrays.sort(tasks, (a, b) -> (b[1] - b[0]) - (a[1] - a[0]));

    int energy = 0, curEnergy = 0;
    for (int[] task : tasks) {
        int actual = task[0], minimum = task[1];
        if (curEnergy < minimum) {
            energy += minimum - curEnergy;
            curEnergy = minimum;
        }
        curEnergy -= actual;
    }
    return energy;
}
// Proof: For adjacent tasks A, B: placing A first needs max(minA, minB + actA)
// Placing B first needs max(minB, minA + actB). Compare to find optimal order.

Earliest Full Bloom (LC 2136)

// LC 2136: Earliest Possible Day of Full Bloom
// Plant seeds sequentially. Plant i takes plantTime[i] days, then growTime[i] days.
// Exchange argument: sort by growTime descending (plant slow-growers first)
public int earliestFullBloom(int[] plantTime, int[] growTime) {
    int n = plantTime.length;
    Integer[] idx = new Integer[n];
    for (int i = 0; i < n; i++) idx[i] = i;
    // Sort by growTime descending
    Arrays.sort(idx, (a, b) -> growTime[b] - growTime[a]);

    int curDay = 0, ans = 0;
    for (int i : idx) {
        curDay += plantTime[i];
        ans = Math.max(ans, curDay + growTime[i]);
    }
    return ans;
}
// Proof: For tasks A, B consider "A then B" vs "B then A".
// "A then B" finishes at max(pA + gA, pA + pB + gB)
// "B then A" finishes at max(pB + gB, pB + pA + gA)
// A first is better when gA >= gB. So sort by growTime desc.

Common mistake: Sorting by total time (plant + grow) does NOT work. The exchange argument specifically proves growTime-descending is optimal. Always derive the comparator from the exchange, not from intuition alone.

Greedy with Heap

Core Pattern: Sort + Heap for Regret/Replacement

When a simple sort-and-scan greedy is not optimal because an early greedy choice might be suboptimal later, use a heap to enable regret: undo a previous choice when a better one appears.

Key idea: Sort items by one criterion. Process them in order, maintaining a heap that tracks your current selections. When a new item is better than your worst current selection, swap them (regret the old choice).

Course Schedule III (LC 630)

// LC 630: Schedule Maximum Courses
// Sort by deadline. Greedily take courses; if time exceeds deadline,
// drop the longest course taken so far (regret via max-heap).
public int scheduleCourse(int[][] courses) {
    Arrays.sort(courses, (a, b) -> a[1] - b[1]); // sort by deadline
    PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
    int time = 0;
    for (int[] course : courses) {
        int duration = course[0], deadline = course[1];
        time += duration;
        maxHeap.offer(duration);
        if (time > deadline) { // exceeded deadline, drop longest
            time -= maxHeap.poll();
        }
    }
    return maxHeap.size();
}

IPO (LC 502)

// LC 502: IPO - maximize capital after k investments
// Sort projects by capital needed. Use max-heap for profit of affordable projects.
public int findMaximizedCapital(int k, int w, int[] profits, int[] capital) {
    int n = profits.length;
    int[][] projects = new int[n][2];
    for (int i = 0; i < n; i++) projects[i] = new int[]{capital[i], profits[i]};
    Arrays.sort(projects, (a, b) -> a[0] - b[0]); // sort by capital

    PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
    int i = 0;
    while (k-- > 0) {
        while (i < n && projects[i][0] <= w) {
            maxHeap.offer(projects[i][1]);
            i++;
        }
        if (maxHeap.isEmpty()) break;
        w += maxHeap.poll(); // take most profitable affordable project
    }
    return w;
}

Hire K Workers (LC 857)

// LC 857: Minimum Cost to Hire K Workers
// Sort by wage/quality ratio. For each worker as the "max ratio" worker,
// pick K-1 workers with smallest quality (using max-heap to maintain).
public double mincostToHireWorkers(int[] quality, int[] wage, int k) {
    int n = quality.length;
    double[][] workers = new double[n][2];
    for (int i = 0; i < n; i++)
        workers[i] = new double[]{(double) wage[i] / quality[i], quality[i]};
    Arrays.sort(workers, (a, b) -> Double.compare(a[0], b[0])); // sort by ratio

    PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
    int sumQ = 0;
    double ans = Double.MAX_VALUE;
    for (double[] w : workers) {
        int q = (int) w[1];
        maxHeap.offer(q);
        sumQ += q;
        if (maxHeap.size() > k) sumQ -= maxHeap.poll();
        if (maxHeap.size() == k) ans = Math.min(ans, sumQ * w[0]);
    }
    return ans;
}

When to use heap with greedy: (1) You need to maintain top-K or bottom-K elements. (2) You want to "regret" a previous choice when a better option appears. (3) You process items in sorted order but need to track a dynamic set of selections.

Greedy vs Dynamic Programming

When Greedy Fails

Greedy fails when the locally optimal choice does not lead to globally optimal. Classic example:

Coin Change with Arbitrary Denominations

// Greedy approach: always pick largest coin. FAILS for some denominations!
// Example: coins = [1, 3, 4], amount = 6
// Greedy: 4 + 1 + 1 = 3 coins
// Optimal: 3 + 3 = 2 coins
// Must use DP when coin denominations are arbitrary.

Comparison Table

AspectGreedyDynamic Programming
Choice timingMakes irrevocable choice before solving subproblemsMakes choice after seeing subproblem solutions
SubproblemsOnly one subproblem remains after choiceMultiple overlapping subproblems
Correctness proofExchange argument / greedy stays aheadOptimal substructure + recurrence
EfficiencyUsually O(n log n) or O(n)O(states x transitions), often O(n^2) or more
ImplementationSort + single passTable filling or memoization
When to useLocal choice is provably safeNeed to explore multiple choices
RevisitsNever reconsidersConsiders all combinations via memo

Decision Guide: Is This Greedy or DP?

Q1: Can I sort and make one irrevocable pass? └— Yes, and I can prove each choice is safe → Greedy └— Not sure → Q2: Does current choice affect which subproblems I can solve? └— No (only one subproblem remains) → Probably Greedy └— Yes (choice branches into multiple subproblems) → DP Q3: Can I find a counterexample where greedy fails? └— Yes → Definitely DP └— No → Try Greedy (but prove it!)

Interview Tips for Greedy Problems

Step-by-Step Communication

  1. Identify the greedy signal: "This problem asks for the minimum/maximum of something with a sorting-friendly constraint."
  2. State your greedy criterion: "I will sort by [X] and greedily choose [Y] at each step."
  3. Justify correctness (30 seconds): "The exchange argument shows that if we had a different choice, swapping would not improve the solution because [reason]."
  4. Analyze complexity: "Sort is O(n log n), single pass is O(n), total O(n log n)."
  5. Code it up.

Common Follow-Up Questions

  • "What if the constraints change?" — Often breaks greedy. Discuss when you would switch to DP.
  • "Can you prove this is correct?" — Use exchange argument: "Assume OPT differs at position k..."
  • "What's the time complexity?" — Usually O(n log n) for sort-based, O(n) for scan-based.
  • "Can you do better?" — If O(n log n), check if sorting is avoidable (e.g., bucket sort, counting).
  • "What if items have weights/values?" — Usually signals knapsack (DP), not greedy.

Red Flags That Greedy Won't Work

Greedy likely fails when:

  • The problem involves choosing subsets with constraints (knapsack variant)
  • Current choice affects future available choices in complex ways
  • Problem says "number of ways" (usually DP/combinatorics)
  • Small counterexample can be constructed easily
  • Items have two independent dimensions that both matter (weight AND value)

Green Flags That Greedy Will Work

Greedy likely works when:

  • Problem involves intervals/events with deadlines
  • There is an obvious sorting criterion that "unlocks" the problem
  • The problem has a "coverage" or "farthest reachable" flavor
  • You need to minimize/maximize something with a single constraint
  • The problem structure allows exchange argument (pairwise comparison works)
  • Classic pattern: "assign resources to tasks one by one"
