Knowledge Retrieval (RAG) Overview
What RAG is, why it matters for contact centers, and what interviewers evaluate.

What Is RAG?

RAG = Retrieval-Augmented Generation. Instead of asking a language model to answer from its own memory (which may be outdated or hallucinated), you first retrieve relevant documents from a trusted knowledge base, then feed those documents to the LLM to generate a grounded answer. The answer is only as good as what you retrieve — which is why the retrieval pipeline (your job) matters more than the LLM (someone else's black box).

The Contact Center Angle

Agents waste up to 30% of their handle time searching internal wikis, PDFs, and FAQ databases for the right answer. A RAG system listens to the live conversation and proactively surfaces the answer before the agent even finishes searching — turning a 45-second lookup into an instant suggestion.

Core Capabilities

What the System Does

  • Ingest enterprise documents (Confluence, SharePoint, PDFs, FAQ DBs) and keep them fresh
  • Index content for both keyword search (exact terms) and semantic search (meaning)
  • Retrieve the most relevant passages given the current conversation context
  • Generate a concise answer with citations back to source documents
  • Proactively trigger — no explicit query needed; it reads the conversation and searches on its own

What's Black Box vs Your Code

BLACK BOX (someone else builds): the embedding model (text → vector), the cross-encoder reranker, and the LLM that writes the final answer. YOUR CODE: the ingestion pipeline, the chunking logic, the dual indexing, the hybrid search + fusion, the query assembly, caching, and the API. In an interview, be explicit about this boundary — it shows maturity.

"RAG is a retrieval problem wearing an AI costume. The LLM at the end is a commodity — everyone has access to the same models. The differentiation, and where 90% of the engineering effort goes, is in retrieving the right context. So I'll focus my design on the retrieval pipeline and treat the embedding model and LLM as well-defined black-box interfaces."

Back-of-the-Envelope Math

Corpus Size

  • 100K documents × ~5 pages each = 500K pages
  • Each page ~500 words; chunk at 512 tokens with 100-token overlap → ~2 chunks per page
  • Total: ~2M chunks

Vector Storage

  • Embedding dimension: 768 floats × 4 bytes = 3 KB per chunk
  • 2M chunks × 3 KB = ~6 GB of raw vectors (fits in RAM on a single beefy node)
  • With HNSW index overhead (~1.5×): ~9 GB. Still single-node feasible.

Query & Update Load

  • 10K concurrent conversations × 1 retrieval per ~10 sec = ~1K queries/sec
  • Document updates: ~100 docs/day = trivial for incremental indexing
  • Each query fans out to Elasticsearch + Vector DB in parallel → ~2K internal search ops/sec
Interview Signal

The key realization from this math: 6 GB of vectors fits on one machine. You don't need a distributed vector database for 2M chunks — say so. Reaching for Pinecone-at-massive-scale when pgvector on one node would work is over-engineering, and interviewers notice.

High-Level Architecture
Two pipelines: offline ingestion and real-time retrieval.

The Two-Pipeline Model

Every RAG system splits into an offline ingestion pipeline (runs when documents change) and a real-time retrieval pipeline (runs on every query). Keeping them separate is the single most important architectural decision — ingestion is throughput-bound and can be slow; retrieval is latency-bound and must be fast.

Latency Budget (500ms total)

Query processing 20ms + parallel search (ES ∥ Vector) 80ms + RRF merge 10ms + reranking 120ms + LLM generation 220ms + network/buffer 50ms = 500ms. Note the LLM dominates — which is why some designs skip generation and just show the top retrieved passage directly for lowest latency.

Document Ingestion Pipeline
How raw enterprise documents become searchable, embedded, indexed chunks.

Source Connectors

Enterprise knowledge lives everywhere: Confluence wikis, SharePoint, Google Docs, uploaded PDFs, structured FAQ databases, even Slack threads. Build pluggable connectors with a common interface so adding a new source doesn't touch the core pipeline.

Crawling Strategy: Webhook + Periodic Sync

  • Webhook-triggered (instant): most sources fire an event when a doc changes — re-index just that doc within seconds
  • Periodic full-sync (safety net): nightly full crawl catches anything the webhooks missed (deleted docs, failed webhooks)
Don't Rely on Webhooks Alone

Webhooks get dropped, sources go down, edits slip through. The nightly reconciliation sync is what makes freshness reliable. In the interview, mention both — relying only on webhooks is a common junior mistake.

Chunking Strategies

StrategyHowTrade-off
Fixed-size512 tokens + 100-token overlapSimple, predictable; may split mid-thought
SemanticSplit on headers/paragraphsBetter relevance; complex, variable size
RecursiveTry paragraph, fall back to sentence, then tokenGood balance; more code
Pragmatic Answer

"Start with fixed-size 512-token chunks with 100-token overlap. Overlap prevents an answer from being split across a chunk boundary and lost. If retrieval quality is poor, upgrade to semantic chunking. Don't over-engineer chunking before you have a relevance problem."

Data Models

type Document struct {
    ID           string
    TenantID     string
    Source       string    // "confluence" | "sharepoint" | "pdf" | "faq"
    Title        string
    URL          string
    ContentHash  string    // detect changes without re-reading full content
    LastModified time.Time
    IndexedAt    time.Time
    Status       string    // "active" | "deleted"
}

type Chunk struct {
    ID           string
    DocumentID   string
    TenantID     string
    Content      string
    ChunkIndex   int       // position within document
    SectionPath  string    // "Billing > Refunds > Eligibility"
    Embedding    []float32 // 768 dims (also stored in vector DB)
    Metadata     map[string]string
    EmbeddingVersion string // which embedding model produced this
}

Dual Indexing

Each chunk goes to both stores: full text to Elasticsearch (for BM25 keyword search), embedding vector to the Vector DB (for semantic search). They share the same chunk_id so results can be cross-referenced and merged.

Handling Updates & Deletes

  • Update: compare ContentHash; if changed, re-chunk only the modified sections, re-embed, upsert into both indexes
  • Delete: soft-delete (mark status = "deleted"), filter out at query time, hard-delete in nightly cleanup
  • Idempotency: use deterministic chunk IDs (hash(doc_id + chunk_index)) so re-indexing overwrites cleanly instead of duplicating
Hybrid Search Deep Dive
Why you need both keyword and semantic search — and how to merge them.

Why Hybrid?

Neither search type wins alone:

QueryBM25 (keyword)Vector (semantic)
"order #12345 status"✅ Nails the exact ID❌ IDs mean nothing semantically
"how do I stop my subscription"❌ Doc says "cancel", not "stop"✅ Understands "stop" ≈ "cancel"
"SKU-889 return policy"✅ Exact product code match⚠️ May drift to general returns
"customer is really upset about fees"❌ Too fuzzy for keywords✅ Matches "fee dispute" docs

BM25: Keyword Relevance

BM25 (Elasticsearch's default) scores by term frequency × inverse document frequency — a term that appears often in this document but rarely across the corpus scores high. It's unbeatable for exact matches: IDs, codes, product names, specific error messages.

Vector Similarity: Semantic Relevance

The embedding model maps text to a point in high-dimensional space where meaning is geometry. "Cancel my plan" and "stop my subscription" land close together even with zero shared words. Similarity = cosine of the angle between query vector and chunk vector.

Reciprocal Rank Fusion (RRF)

The problem: BM25 scores and cosine scores are on totally different scales — you can't just add them. RRF solves this by using rank position, not raw score:

RRF_score(doc) = Σ  1 / (k + rank_i(doc))
                 i

  where:
    k    = 60  (standard constant, dampens the impact of top ranks)
    rank = position of doc in result list i (1-indexed)

Example — a chunk ranked #1 in BM25 and #3 in vector search:
    RRF = 1/(60+1) + 1/(60+3)
        = 1/61 + 1/63
        = 0.01639 + 0.01587
        = 0.03226   ← high combined score

A chunk that ranks #1 in ONLY one list scores lower than
one that ranks well in BOTH — which is exactly what we want.
Name-Drop RRF

Saying "I'd merge with Reciprocal Rank Fusion, k=60" instantly signals you know modern retrieval. It's the standard technique used in production hybrid search (Elasticsearch even has native RRF support now). Cheap, effective, no tuning of score weights needed.

When to Weight One Side Higher

  • Boost BM25 when the query contains IDs, error codes, product SKUs, exact phrases
  • Boost vector for vague, conversational, paraphrased questions ("how do I...", "what happens if...")
  • Detection: a lightweight regex/classifier on the query decides the weighting — or just use balanced RRF and let it self-correct

Domain Synonyms Matter

Enterprise docs use formal language ("terminate service") while customers use casual language ("stop my thing"). Synonym expansion at index time bridges this gap on the keyword side — complementing what the embedding model does on the semantic side.

Multi-Tenancy

Always filter by tenant_id as a mandatory query clause. Options: shared index with tenant filter (simple, good for many small tenants) vs index-per-tenant (stronger isolation, good for few large tenants). Start shared; split out whales.

Index Lifecycle Management (ILM)

  • Hot: current, frequently-queried docs on fast SSD nodes
  • Warm: older docs, read-only, on cheaper storage
  • Delete: expired/superseded docs removed automatically

Cluster Sizing

For 1K queries/sec over 2M chunks: a 3-node cluster (1 primary + 2 replicas per shard) with ~32 GB RAM each. Replicas serve reads in parallel and provide failover. Shard count = keep shards 10–50 GB each; 2M small chunks fit in a few shards.

For This Scale (2M chunks / 6GB)

"pgvector on a single Postgres node handles 2M vectors comfortably, and I already store chunk metadata in Postgres — so I keep everything in one system and avoid a separate vector database entirely. I'd only reach for Milvus or Pinecone if the corpus grew past ~10M vectors or query latency degraded." Choosing the simplest thing that works is a strong signal.

HNSW: Approximate Nearest Neighbor

Exact nearest-neighbor search over 2M vectors is too slow. HNSW (Hierarchical Navigable Small Worlds) builds a layered graph that finds approximate nearest neighbors in logarithmic time — trading a tiny bit of accuracy for a huge speedup.

HNSW parameters:
  M              = 16    // edges per node (higher = better recall, more memory)
  efConstruction = 200   // build-time search width (higher = better graph, slower build)
  efSearch       = 100   // query-time search width (higher = better recall, slower query)

Trade-off: efSearch is the knob you tune at query time.
  efSearch=50  → ~95% recall, fastest
  efSearch=100 → ~98% recall, balanced   ← good default
  efSearch=200 → ~99.5% recall, slower

Distance Metric

  • Cosine similarity: measures angle, ignores magnitude — the standard choice for text embeddings
  • Dot product: faster, equivalent to cosine if vectors are normalized to unit length (normalize once at index time and you get cosine speed for free)

Sharding

Shard by tenant_id for multi-tenant isolation and query locality — a query only touches one tenant's shard, and one tenant's load doesn't affect others.

The Embedding-Upgrade Problem

Model Upgrades Require Full Re-Indexing

Embeddings from model v1 and model v2 live in different vector spaces — you cannot compare them. When you upgrade the embedding model, every chunk must be re-embedded and re-indexed. For 2M chunks, budget 2–4 hours. Strategy: run old and new indexes in parallel, dual-write during migration, cut over atomically, then drop the old index. The embedding_version field on each chunk tracks which model produced it.

Query Processing & Context Assembly
Turning a live, messy conversation into an effective search query.

The Proactive Trigger Problem

Unlike a search box, there's no explicit query here. The system reads the ongoing conversation and decides when to search and what to search for. This is the hardest part of the retrieval side.

From Conversation to Query

  • Context window: take the last N utterances (not the whole call) — recent context is what matters
  • Intent extraction: the NLP layer already classified intent ("billing_dispute") — use it to scope the search
  • Query reformulation: "um, so my thing got charged twice" → "duplicate charge refund policy"
  • Entity injection: pull entities (product name, plan type) into the query for precision
type QueryContext struct {
    ConversationID string
    TenantID       string
    RecentTurns    []Utterance   // last 3-5 turns
    CurrentIntent  string        // from NLP pipeline
    Entities       map[string]string
    LastQueryAt    time.Time     // debounce: don't re-search every word
}

// Debounce: only trigger a new search when intent changes
// or 5+ seconds passed AND a new entity appeared.
func (q *QueryContext) ShouldSearch(prev *QueryContext) bool {
    if q.CurrentIntent != prev.CurrentIntent { return true }
    if time.Since(prev.LastQueryAt) > 5*time.Second &&
       hasNewEntity(q, prev) { return true }
    return false
}
Don't Search on Every Word

Re-running retrieval on every transcript token would hammer your search cluster and flood the agent with flickering suggestions. Debounce: trigger only on intent change or meaningful new context. This is a top follow-up question.

Query Expansion

Send the reformulated query to both paths: the raw keywords to Elasticsearch (BM25 handles exact terms), and the full natural-language phrasing to the embedding model (semantics handle paraphrase). Don't pre-strip the query — each side wants different input.

Reranking & Answer Generation
From a merged candidate list to a single trustworthy answer.

Why Rerank After RRF?

RRF gives you a decent merged list fast, but it only knows rank positions, not deep relevance. A cross-encoder reranker (black box) reads the query and each candidate together and scores true relevance — far more accurate, but too slow to run on all 2M chunks. So: retrieve ~50 with hybrid search, rerank the top 50 down to the best 3-5.

Two-Stage Retrieval Is Standard

Stage 1 (cheap, recall-focused): hybrid search returns 50 candidates. Stage 2 (expensive, precision-focused): cross-encoder reranks to top 5. This "retrieve then rerank" pattern is the backbone of every production RAG system. Name it.

Answer Generation with Citations

// Assemble the LLM prompt from reranked chunks
type GenerationRequest struct {
    Query      string
    Context    []RankedChunk   // top 3-5 after reranking
    TenantID   string
    MaxTokens  int
}

// The prompt instructs the LLM to ONLY use provided context
// and to cite the source chunk for each claim.
prompt := `Answer the agent's question using ONLY the context below.
Cite the source document for each fact as [doc_id].
If the context does not contain the answer, say "No KB match found."

Context:
{{range .Context}}[{{.DocumentID}}] {{.Content}}
{{end}}
Question: {{.Query}}`

Hallucination Prevention

  • Grounding constraint: prompt forces "use ONLY provided context"
  • Citation requirement: every claim must reference a chunk — uncited claims are flagged
  • Confidence floor: if top reranked score is below threshold, return "No confident answer" instead of guessing
  • Fallback: if the LLM times out, show the top reranked passage directly (no generation) — still useful to the agent
Latency Escape Hatch

LLM generation eats ~220ms of your 500ms budget. For the lowest-latency tier, skip generation entirely and surface the top reranked passage verbatim. The agent reads it themselves. Generation is a quality upgrade, not a hard requirement.

Scaling & Reliability
Caching, cluster sizing, and the freshness-vs-performance trade-off.

Caching Layers

CacheKeyTTLBenefit
Query result cachehash(query + tenant + intent)5 minPopular questions served instantly
Embedding cachehash(query text)1 hourSkip embedding model call for repeat queries
Document metadatadocument_id10 minFast citation enrichment
The Embedding Cache Wins Big

Contact centers have massive query repetition — the same top-50 questions recur constantly. Caching query embeddings skips a black-box model call entirely, cutting latency and cost. Cache hit rate here can exceed 40%.

The Freshness vs Performance Trade-off

Aggressive caching improves latency but risks serving stale answers after a document updates. Resolution: invalidate query caches on document change. When a doc is re-indexed, purge cache entries whose citations reference it. Freshness wins for compliance-sensitive content.

Failure Modes

Embedding model down

  • Fall back to BM25-only (keyword) search
  • Degraded relevance, but still functional

LLM generator down

  • Skip generation, return top reranked passage
  • Agent reads raw snippet — still helpful

Vector DB down

  • Fall back to BM25-only
  • Semantic queries suffer; exact-match queries fine

Elasticsearch down

  • Fall back to vector-only
  • Lose exact ID/code matching
Graceful Degradation Pattern

Because you run two independent search backends, losing one degrades quality but never causes total failure. This redundancy is a feature — mention that hybrid search is also a reliability strategy, not just a relevance one.

Pitfalls & Follow-ups
Mistakes to avoid and how to answer the hard questions.

Common Pitfalls

Pitfall 1 — Chunks too large

Huge chunks dilute relevance (the answer is buried in noise) and blow the LLM context budget. 512 tokens with overlap is a sane default. Mention the trade-off proactively.

Pitfall 2 — Ignoring freshness

Designing ingestion as a one-time batch. Enterprise docs change constantly; stale answers can be compliance violations. Always design incremental updates + cache invalidation.

Pitfall 3 — Vector-only search

Assuming embeddings solve everything. They fail on exact IDs, codes, and product names. Hybrid (BM25 + vector) is non-negotiable for enterprise content.

Pitfall 4 — Over-engineering the vector DB

Reaching for a distributed Pinecone cluster for 2M vectors that fit in 6GB on one node. Right-size to the data.

Likely Follow-up Questions

Q: How do you keep the index fresh when a doc updates?▶

Webhook fires on change → re-chunk only modified sections → re-embed → upsert into both ES and vector DB with deterministic chunk IDs → invalidate query caches referencing that doc. Nightly full-sync as a safety net for missed webhooks.

Q: When would BM25 beat vector search?▶

Exact identifiers: order numbers, SKUs, error codes, specific product names. Embeddings treat "SKU-889" as semantically meaningless; BM25 nails the exact token. That's precisely why we run both and fuse with RRF.

Q: What happens when you upgrade the embedding model?▶

Full re-index — v1 and v2 embeddings live in different vector spaces and can't be compared. Run old + new indexes in parallel, dual-write during migration, cut over atomically, drop the old. The embedding_version field tracks each chunk. Budget 2-4 hours for 2M chunks.

Q: How do you prevent the LLM from hallucinating?▶

Grounding: prompt forces "use ONLY provided context." Citations: every claim references a chunk. Confidence floor: if top reranked score is too low, return "no confident answer" rather than guess. Fallback: on LLM failure, show the raw top passage.

Q: How do you decide when to trigger a search?▶

Debounce on the conversation stream: trigger on intent change, or after 5+ seconds when a meaningful new entity appears. Never search on every transcript token — it hammers the cluster and flickers the UI.
