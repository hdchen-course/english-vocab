Pattern Finding & Mathematical Induction

Compute Small Cases • Spot the Pattern • Prove by Induction • Bit Tricks • Recursive Structure

The Pattern Finding Process

These problems look unsolvable at first. The key: compute small cases by hand, observe the pattern, then prove/implement it.

The 4-Step Method

Step 1: Brute Force Small Cases — Compute f(0), f(1), f(2), f(3), f(4) by hand or with code.
Step 2: Look for Structure — Is it Fibonacci? Powers of 2? Catalan? Bit-reversal? Cycle?
Step 3: Form a Hypothesis — f(n) = 2*f(n-1)? f(n) = f(n/2) XOR something?
Step 4: Verify & Implement — Check f(5), f(6). If holds, implement. If not, revise.

When to Suspect "Pattern Finding"

Signals:
• Problem seems impossible with standard algorithms
• Constraint is very large (n up to 10^9 or 10^18) but answer is small
• Problem involves binary/tree structure with recursive self-similarity
• "Find the k-th element in a sequence defined by rule X"
• Standard DP would be O(n) but n is too large — needs O(log n) via pattern

Bit-Based Pattern Discovery

Many problems have elegant solutions once you write the answers in binary and observe structure.

Example: K-th Symbol in Grammar (LC 779)

n=1: 0 n=2: 01 n=3: 0110 n=4: 01101001 Pattern: each row = prev + complement(prev) Key insight: k-th symbol = popcount(k-1) % 2
// LC 779: the k-th bit in row n
// Observe: answer only depends on number of 1-bits in (k-1)
int kthGrammar(int n, int k) {
    return Integer.bitCount(k - 1) % 2;
}
// Or recursive: kthGrammar(n, k) = kthGrammar(n-1, (k+1)/2) ^ ((k-1)&1)

Example: Minimum One Bit Operations (LC 1611)

Two allowed operations: (1) flip rightmost bit, (2) flip bit left of the rightmost 1. Goal: reach 0 from n.

Compute small cases: n = 1 (001) → 0 : 1 step n = 2 (010) → 011 → 001 → 0 : 3 steps n = 3 (011) → 010 → 0* : 2 steps (*shortcut via op2) n = 4 (100) → ... → 0 : 7 steps n = 5 (101) → ... → 0 : 6 steps n = 6 (110) → ... → 0 : 5 steps n = 7 (111) → ... → 0 : 4 steps Sequence of answers: 0, 1, 3, 2, 7, 6, 5, 4, 15, 14, ... This IS the Gray code sequence! Insight: the answer for n = the position of n in Gray code order. Equivalently: convert n from Gray code to binary integer.

Why Gray code? The two allowed operations are exactly how you step through a Gray code: each step flips exactly one bit, and the allowed flip positions follow the Gray code generation rule. So "number of steps to reach 0" = "position of n in the Gray code sequence" = Gray-to-binary conversion of n.

// Gray code → binary conversion = the answer
// Binary of Gray code g: b[i] = g[i] ^ g[i+1] ^ g[i+2] ^ ...
// Efficiently: b = g ^ (g>>1) ^ (g>>2) ^ ... = running XOR
int minimumOneBitOperations(int n) {
    int result = 0;
    while (n > 0) {
        result ^= n;
        n >>= 1;
    }
    return result;
}
// Why this works:
// Gray-to-binary: binary[i] = XOR of all Gray bits from MSB to position i
// The loop computes exactly this: result accumulates n, n>>1, n>>2, ...
// Each XOR "propagates" the Gray code bits into binary position values

Example: Integer Replacement (LC 397)

Given n, reach 1 using: if even → n/2, if odd → n+1 or n-1. Minimize steps.

Compute small cases (binary view): n=2 (10) → 1 : 1 step (÷2) n=3 (11) → 2 → 1 : 2 steps (-1, ÷2) n=4 (100) → 2 → 1 : 2 steps n=7 (111) → 8 → 4 → 2 → 1 : 4 steps (+1 path) → 6 → 3 → 2 → 1 : 4 steps (-1 path) ← same! n=15 (1111) → 16 → 8 → 4 → 2 → 1 : 5 steps (+1 clears all bits at once!) → 14 → 7 → ... → 1 : 6 steps (-1 is worse) Pattern in binary: - Last 2 bits = 01 (e.g. 5=101): prefer -1 (removes the 1, next step ÷2) - Last 2 bits = 11 (e.g. 7=111): prefer +1 (carries propagate, clears trailing 1s) - Exception: n=3 (11) → prefer -1 (3→2→1 is 2 steps, 3→4→2→1 is 3 steps)

Why +1 when bits end in ...11? Adding 1 to ...0111 gives ...1000 — it clears all trailing 1s in one carry, creating a power of 2 which halves away quickly. Subtracting gives ...0110 which still has multiple 1-bits to deal with.

int integerReplacement(int n) {
    int steps = 0;
    long x = n;
    while (x != 1) {
        if ((x & 1) == 0) x >>= 1;        // even: always halve
        else if (x == 3 || (x & 3) == 1) x--; // ...01: subtract
        else x++;                            // ...11: add (carry clears trailing 1s)
        steps++;
    }
    return steps;
}

Recursive Structure (Divide & Conquer Patterns)

The problem has self-similar structure: f(n) can be expressed in terms of f(n/2) or f(n-1) with a simple transformation.

Example: Beautiful Array (LC 932)

Construct a permutation of [1..n] where no three indices i < k < j satisfy A[k]*2 = A[i] + A[j] (no arithmetic triple).

Compute small cases: n=1: [1] n=2: [1,2] (no triple possible with 2 elements) n=3: [1,3,2] ✓ (check: 3*2≠1+2, 3*2≠1+2... no violation) n=4: [1,3,2,4] n=5: [1,5,3,2,4] Key observations: 1. If A is beautiful, then (2*A - 1) is also beautiful (affine transform preserves the property) 2. If A is beautiful, then (2*A) is also beautiful 3. odd + even = odd → an odd number can NEVER be the average of an odd and an even 4. So: put all odds on the left, all evens on the right → cross-pairs never violate! 5. Recurse: make the odd-half and even-half each beautiful independently

Why it works: A[i] + A[j] = 2*A[k] requires all three to have the same parity. By separating odds and evens, we eliminate all cross-parity triples. Within each group, we recursively apply the same trick (odd positions / even positions of a smaller beautiful array).

// LC 932: iteratively double the array
int[] beautifulArray(int n) {
    List<Integer> res = new ArrayList<>();
    res.add(1);
    while (res.size() < n) {
        List<Integer> next = new ArrayList<>();
        for (int x : res) if (2*x - 1 <= n) next.add(2*x - 1); // odds
        for (int x : res) if (2*x <= n) next.add(2*x);         // evens
        res = next;
    }
    return res.stream().mapToInt(Integer::intValue).toArray();
}

Example: Elimination Game (LC 390)

Start with 1,2,...,n. Alternately eliminate from left then from right. What's the last number?

Trace through: n=1: [1] → 1 n=2: [1,2] →remove 1 from left → 2 n=3: [1,2,3] →remove 1,3 from left → [2] → 2 n=4: [1,2,3,4] →remove 1,3 → [2,4] →remove 4 from right → 2 n=5: [1,2,3,4,5] →remove 1,3,5 → [2,4] →remove 2 from right → 4 n=6: [1..6] →remove 1,3,5 → [2,4,6] →remove 6,2 → 4 Pattern: after removing from left, survivors are 2,4,6,... = 2 * [1,2,3,...,n/2] but now eliminate from RIGHT = 2 * "mirror problem on n/2 elements" f(n) = 2 * (n/2 + 1 - f(n/2)) Why the formula: remaining elements after left-pass are the evens. Divide by 2 → reduced to same game on n/2 elements, but direction flipped. "Flipped" position k in [1..m] = m+1-k, hence the (n/2 + 1 - f(n/2)).
// LC 390: recursive one-liner
int lastRemaining(int n) {
    return n == 1 ? 1 : 2 * (n / 2 + 1 - lastRemaining(n / 2));
}
// Derivation:
// After left-pass: survivors = {2, 4, ..., 2*floor(n/2)} (n/2 elements)
// Divide by 2: same game on [1..n/2] but starting from RIGHT
// Position of survivor in right-first game on m elems = m+1 - f(m) where f = left-first
// So f(n) = 2 * (n/2 + 1 - f(n/2))

State Cycle Detection

When n is huge but the state space is bounded, the sequence must cycle. Find the period, then answer = f(n mod period).

Example: Prison Cells After N Days (LC 957)

8 cells, only 6 inner cells matter → at most 2^6 = 64 states After at most 64 steps, must repeat → find cycle length Typically cycle = 14 for this problem
// LC 957: detect cycle, then mod
int[] prisonAfterNDays(int[] cells, int n) {
    Map<String, Integer> seen = new HashMap<>();
    while (n > 0) {
        String key = Arrays.toString(cells);
        if (seen.containsKey(key)) {
            int cycle = seen.get(key) - n;
            n %= cycle;  // skip full cycles
        }
        seen.put(key, n);
        if (n >= 1) { cells = nextDay(cells); n--; }
    }
    return cells;
}

General Cycle Detection Template

// When state space is bounded: find first repeat
Map<State, Integer> seen = new HashMap<>();
State state = initial;
for (int step = 0; step < n; step++) {
    if (seen.containsKey(state)) {
        int cycleStart = seen.get(state);
        int cycleLen = step - cycleStart;
        int remaining = (n - step) % cycleLen;
        for (int i = 0; i < remaining; i++) state = next(state);
        return state;
    }
    seen.put(state, step);
    state = next(state);
}

Constructive Algorithms

Build the answer directly by understanding its structure. Often involves bit-by-bit construction or greedy with insight.

Example: Minimize XOR (LC 2429)

Make result have same popcount as num2, minimizing XOR with num1. Insight: set high bits of num1 first (to cancel them), then fill low bits.
// Set bits greedily from MSB to minimize XOR with num1
int minimizeXor(int num1, int num2) {
    int target = Integer.bitCount(num2);
    int result = 0;
    // First: set bits that are in num1 (cancels XOR)
    for (int i = 30; i >= 0 && target > 0; i--)
        if ((num1 & (1 << i)) != 0) { result |= (1 << i); target--; }
    // Then: set remaining bits from LSB (adds to XOR)
    for (int i = 0; i <= 30 && target > 0; i++)
        if ((result & (1 << i)) == 0) { result |= (1 << i); target--; }
    return result;
}

Example: Maximum XOR Product (LC 2939)

Choose x (within n bits) to maximize (a XOR x) * (b XOR x).

Insight: maximize product = make the two values as close as possible (AM-GM). For each bit position i (from MSB to LSB): Let a_i = bit i of a, b_i = bit i of b. Case 1: a_i == b_i (both 0 or both 1) → Set x_i to flip both to 1 (if both are 0) or keep both 1 (if both are 1). → This adds 2^i to BOTH values. Always good. Case 2: a_i != b_i (one is 0, one is 1) → x_i can give the bit to either a or b (XOR flips the one that has 0). → Give it to whichever is currently SMALLER (balance the two values). → This is greedy: making them closer maximizes the product. After processing all bits: answer = (a XOR x) * (b XOR x) mod 10^9+7.
// LC 2939: greedy bit-by-bit, balance a and b
long maximumXorProduct(long a, long b, int n) {
    long MOD = 1_000_000_007;
    for (int i = n - 1; i >= 0; i--) {
        long bit = 1L << i;
        if (((a >> i) & 1) == ((b >> i) & 1)) {
            // same bit: set both to 1
            a |= bit; b |= bit;
        } else {
            // different: give the 1-bit to the smaller value
            if (a < b) { a |= bit; b &= ~bit; }
            else { b |= bit; a &= ~bit; }
        }
    }
    return (a % MOD) * (b % MOD) % MOD;
}

Binary Tree / Divide-by-2 Thinking

Visualize the problem as a binary tree where n splits into n/2. Often leads to O(log n) solutions.

Example: Kth Smallest Instructions (LC 1643)

Grid path from (0,0) to (r,c) using r 'V' and c 'H' moves. Think: at each step, how many paths start with 'H'? = C(remaining-1, r) If k <= that count, go 'H'. Otherwise go 'V', subtract.

Example: Broken Calculator (LC 991)

Start at X, target Y. Can *2 or -1. Forward thinking is hard. Think BACKWARD from Y: If Y is even: Y/2 (reverse of *2) If Y is odd: Y+1 (reverse of -1) Stop when Y <= X, then answer += X - Y
int brokenCalc(int x, int y) {
    int steps = 0;
    while (y > x) {
        if (y % 2 == 0) y /= 2;
        else y++;
        steps++;
    }
    return steps + (x - y);
}

Key Technique: Think Backwards

When forward exploration branches too much, work backwards from the target. Division/halving backwards becomes multiplication, which has less branching. Used in: Broken Calculator, Reaching Points, Race Car.

Meet in the Middle

Split input in half, enumerate all possibilities for each half, then combine. Reduces O(2^n) to O(2^(n/2)).

When to Use

n is around 30-40 (too large for full 2^n, but 2^20 ~ 1M is fine)
Need to find pairs/sums/counts across all subsets
Classic signals: "find subset with sum closest to target", "count subsets satisfying condition"

Template: Closest Subset Sum

// Split array into two halves, enumerate all subset sums for each
// Then for each sum in left half, binary search in right half
int closestSum(int[] nums, int target) {
    int n = nums.length, half = n / 2;

    // Generate all subset sums for left half
    List<Long> left = new ArrayList<>();
    for (int mask = 0; mask < (1 << half); mask++) {
        long sum = 0;
        for (int i = 0; i < half; i++)
            if ((mask & (1 << i)) != 0) sum += nums[i];
        left.add(sum);
    }
    Collections.sort(left);

    // For each right subset sum, binary search in left
    long best = Long.MAX_VALUE;
    int right = n - half;
    for (int mask = 0; mask < (1 << right); mask++) {
        long sum = 0;
        for (int i = 0; i < right; i++)
            if ((mask & (1 << i)) != 0) sum += nums[half + i];

        long need = target - sum;
        int idx = Collections.binarySearch(left, need);
        if (idx < 0) idx = -(idx + 1);
        if (idx < left.size())
            best = Math.min(best, Math.abs(target - sum - left.get(idx)));
        if (idx > 0)
            best = Math.min(best, Math.abs(target - sum - left.get(idx - 1)));
    }
    return (int) best;
}

Complexity

Time: O(2^(n/2) * log(2^(n/2))) = O(2^(n/2) * n)
Space: O(2^(n/2))
vs brute force: O(2^n) → O(2^(n/2)) — exponentially better for n=36+

Interview Strategy for Pattern Problems

Step-by-Step Communication

1. "Let me compute some small examples to see if there's a pattern."
2. Write out n=1,2,3,4,5 explicitly on the whiteboard.
3. "I notice that f(4) = 2*f(2), and f(5) = 2*f(2)+1... it seems like f(n) relates to f(n/2)."
4. "Let me verify with f(6) and f(7)..."
5. "This gives us an O(log n) recursive solution."

When You're Stuck

Try these lenses:
• Write answers in binary — often reveals XOR/shift patterns
• Think backwards from the answer — reverse operations are often simpler
• Split into odd/even cases — often different recurrences
• Look at differences between consecutive answers — is the delta pattern simpler?
• Check if it's a known sequence (Fibonacci, Catalan, powers of 2)

Common Traps

Don't assume the pattern holds without checking! Always verify at least 2 extra cases beyond what you used to form the hypothesis.
Watch for off-by-one: Is it 0-indexed or 1-indexed? Is n the count or the max value?
Edge cases: n=0, n=1 often need special handling.
