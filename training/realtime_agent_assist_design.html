What Is Agent Assist?

Agent Assist is an AI system that listens to live customer-agent conversations (voice or chat) and provides real-time guidance to the human agent. Think of it as an invisible senior colleague whispering suggestions into the agent's ear during every call.

Core Capabilities

  • Real-time transcription: Converts speech to text as the conversation unfolds
  • Intent detection: Identifies what the customer wants (billing dispute, cancellation, technical issue)
  • Smart suggestions: Recommends next-best-action, script snippets, or knowledge articles
  • Compliance monitoring: Flags when agents miss required disclosures or violate policies
  • Sentiment tracking: Alerts supervisors when customer frustration escalates

Why It Matters (Business Context)

  • Turns every agent into a top performer — new hires get AI-powered coaching from day one
  • Reduces average handle time — agents find answers instantly instead of searching knowledge bases
  • Ensures compliance — auto-detects missed disclosures (PCI, HIPAA, financial regulations)
  • Reduces training ramp — from 8 weeks to 3 weeks with AI-guided onboarding
  • Improves CSAT — faster, more accurate resolutions lead to happier customers

DESIGN PRINCIPLE

Event-driven, not request-response. Each stage emits events that the next stage consumes. This allows parallel processing (e.g., compliance checks can run simultaneously with suggestion generation) and makes the system resilient to individual component slowness.

Dual-Channel Audio: Why It Matters

CRITICAL DESIGN DECISION

You MUST separate agent audio from customer audio. Without channel separation, the system cannot determine who said what, making intent detection meaningless. "I want to cancel" means completely different things depending on the speaker.

  • Agent channel: Used to verify compliance ("Did the agent say the required disclosure?")
  • Customer channel: Used for intent detection and suggestion triggering
  • SIPREC gives you this for free — two separate RTP streams labeled by participant
  • Single-channel audio requires diarization (speaker separation ML) — adds 100ms+ latency and errors

Vendor Integration Reality

Industry Insight

Under 300ms is the threshold for natural conversation flow. Suggestions that arrive after 800ms feel like they're about the previous topic and degrade quality sharply. Your audio pipeline must be near-zero latency — just forwarding, not processing.

REAL-WORLD COMPLEXITY

Each CCaaS vendor (Cisco, Five9, Genesys, NICE, Talkdesk) fires different event sequences for the same actions. Call transfer, hold, conference — each vendor signals these differently. You need a separate state machine per vendor in the Audio Gateway.

Audio Gateway Responsibilities

Audio Gateway (Your Code)

  • Protocol normalization: Accept SIP/RTP/WebSocket/gRPC, output uniform audio frames
  • Channel separation: Tag each audio chunk with speaker role (agent/customer)
  • Codec transcoding: Convert Opus/G.711/G.729 to 16kHz 16-bit PCM (STT input format)
  • Buffering: Accumulate 20ms audio frames, forward in 100ms chunks to STT
  • Session registration: On call start, create session in Session Manager
  • Health monitoring: Detect audio silence (>30s), dead streams, reconnections

STT Integration (Black Box)

You send audio bytes in, you get transcript chunks back. The STT service is a black box — you don't design it, but you must handle its output correctly:

Output TypeLatencyAccuracyUse For
Partial (interim)~50msLower (may change)Quick intent signals, UI "typing" indicator
Final (stable)~500msHigher (stable text)Suggestion generation, compliance checks, archival
OPTIMIZATION

Use partials for speed, finals for accuracy. Trigger fast-path suggestions on partial transcripts (e.g., "cancel" keyword detected), but only commit to session context on final transcripts. This gives you sub-100ms reaction time for high-confidence intents.

// Handling STT output
type TranscriptEvent struct {
    SessionID   string
    Channel     string    // "agent" | "customer"
    Text        string
    IsFinal     bool
    Confidence  float64
    Timestamp   time.Time
    WordTimings []WordTiming  // for compliance: exactly when each word was spoken
}

// Fast path: react to partials for high-confidence keywords
func (p *Pipeline) OnTranscript(evt TranscriptEvent) {
    if !evt.IsFinal && containsHighConfidenceKeyword(evt.Text) {
        // Fast-path: trigger suggestion immediately
        p.fastPathSuggest(evt)
    }
    if evt.IsFinal {
        // Normal path: full NLP analysis
        p.sessionManager.AppendTranscript(evt)
        p.nlpPipeline.Analyze(evt)
    }
}
"The audio pipeline is the first 50ms of our latency budget. Audio arrives via SIPREC or WebSocket, the gateway normalizes it to 16kHz PCM, separates channels, and streams to the STT service. I use partial transcripts for fast-path keyword detection and final transcripts for full NLP analysis. The critical design decision is maintaining dual-channel separation so we always know who said what."
Session Manager Deep Dive
The one component you would actually build in this system. Maintains conversation state for all concurrent sessions.
WHY THIS IS THE STAR

In an interview, this is where you prove engineering depth. The ML models are black boxes, but the Session Manager is 100% your design. It touches every other component, manages concurrency, and its data model drives everything downstream.

Responsibilities

  • Lifecycle management: Create session on call start, maintain during call, archive on end
  • Transcript accumulation: Append utterances from both channels with ordering guarantees
  • Context computation: Maintain rolling context window for ML input
  • Suggestion tracking: Record what was shown to avoid duplicates
  • Concurrency control: Multiple services write simultaneously (STT, NLP, post-processor)
  • TTL and cleanup: Expire stale sessions, archive to persistent storage

// Redis operations for session management
func (sm *SessionManager) AppendTranscript(sessionID string, utt Utterance) error {
    pipe := sm.redis.Pipeline()

    // 1. Append utterance to transcript list
    data, _ := json.Marshal(utt)
    pipe.RPush(ctx, fmt.Sprintf("session:%s:transcript", sessionID), data)

    // 2. Trim to last 50 utterances (sliding window)
    pipe.LTrim(ctx, fmt.Sprintf("session:%s:transcript", sessionID), -50, -1)

    // 3. Update last_activity in session hash
    pipe.HSet(ctx, fmt.Sprintf("session:%s", sessionID), "last_activity", time.Now().Unix())

    // 4. Update score in active sessions sorted set
    pipe.ZAddXX(ctx, fmt.Sprintf("tenant:%s:active", tenantID),
        &redis.Z{Score: float64(time.Now().Unix()), Member: sessionID})

    // Execute atomically
    _, err := pipe.Exec(ctx)
    return err
}

Sliding Context Window

DESIGN DECISION

Keep the last 20 utterances as the "hot" context window for ML input. Full transcript is stored for archival, but ML models perform best with focused context (not 30 minutes of rambling conversation).

// Build context window for ML request
func (sm *SessionManager) GetContextWindow(sessionID string) (*MLContext, error) {
    // Get last 20 utterances from Redis list
    results, err := sm.redis.LRange(ctx,
        fmt.Sprintf("session:%s:transcript", sessionID), -20, -1).Result()

    // Get accumulated entities
    entities, _ := sm.redis.HGetAll(ctx,
        fmt.Sprintf("session:%s:entities", sessionID)).Result()

    // Get session metadata
    meta, _ := sm.redis.HGetAll(ctx,
        fmt.Sprintf("session:%s", sessionID)).Result()

    return &MLContext{
        RecentUtterances: parseUtterances(results),
        Entities:         entities,
        CurrentIntent:    meta["current_intent"],
        TenantID:         meta["tenant_id"],
        CallDuration:     time.Since(parseTime(meta["started_at"])),
    }, nil
}

Concurrency Control

RACE CONDITION

Multiple services write to the same session simultaneously: STT appends transcripts, NLP updates intents, post-processor records shown suggestions. Without coordination, you get lost updates and inconsistent state.

  • Redis Pipeline: Group related writes into atomic pipeline execution
  • Lua Scripts: For read-modify-write operations (e.g., "append only if session is active")
  • Field-level isolation: Each service writes to different Hash fields — no conflicts
  • Append-only transcript: RPUSH is inherently safe for concurrent appenders
-- Lua script: Append transcript only if session is active
-- KEYS[1] = session hash key, KEYS[2] = transcript list key
-- ARGV[1] = utterance JSON
local status = redis.call('HGET', KEYS[1], 'status')
if status == 'active' then
    redis.call('RPUSH', KEYS[2], ARGV[1])
    redis.call('LTRIM', KEYS[2], -50, -1)
    redis.call('HSET', KEYS[1], 'last_activity', ARGV[2])
    return 1
end
return 0  -- Session not active, discard

ML Request Assembly

// Assemble the request to the ML suggestion service
type SuggestionRequest struct {
    SessionID       string       `json:"session_id"`
    TenantID        string       `json:"tenant_id"`
    ModelVersion    string       `json:"model_version"`     // Per-tenant model routing
    Utterances      []Utterance  `json:"utterances"`        // Last 20
    CurrentIntent   string       `json:"current_intent"`
    Entities        map[string]string `json:"entities"`
    CallDuration    int          `json:"call_duration_sec"`
    AgentExperience string       `json:"agent_experience"` // "new" | "experienced"
    ExcludeIDs      []string     `json:"exclude_ids"`      // Already-shown suggestion IDs
}

func (p *NLPPipeline) BuildRequest(evt TranscriptEvent) (*SuggestionRequest, error) {
    ctx, err := p.sessionMgr.GetContextWindow(evt.SessionID)
    if err != nil {
        return nil, err
    }

    tenantCfg := p.tenantConfig.Get(ctx.TenantID)
    shownIDs := p.sessionMgr.GetRecentSuggestionIDs(evt.SessionID, 10)

    return &SuggestionRequest{
        SessionID:       evt.SessionID,
        TenantID:        ctx.TenantID,
        ModelVersion:    tenantCfg.ModelVersion,
        Utterances:      ctx.RecentUtterances,
        CurrentIntent:   ctx.CurrentIntent,
        Entities:        ctx.Entities,
        CallDuration:    int(ctx.CallDuration.Seconds()),
        AgentExperience: tenantCfg.AgentTier,
        ExcludeIDs:      shownIDs,
    }, nil
}

Handling ML Latency Spikes

COMMON FAILURE MODE

ML models have tail latency spikes (p99 can be 5x the p50). If you wait for slow responses, your entire pipeline stalls. You need a strict timeout with a useful fallback — not just an error.

func (p *NLPPipeline) GetSuggestion(req *SuggestionRequest) (*SuggestionResponse, error) {
    // Hard timeout: 50ms for ML call
    ctx, cancel := context.WithTimeout(context.Background(), 50*time.Millisecond)
    defer cancel()

    resp, err := p.mlClient.Suggest(ctx, req)
    if err != nil {
        // Fallback strategy:
        // 1. Check cache for this intent
        if cached := p.cache.Get(req.CurrentIntent, req.TenantID); cached != nil {
            return cached, nil  // Serve cached suggestion
        }
        // 2. Return rule-based fallback
        return p.ruleEngine.GetFallback(req.CurrentIntent), nil
    }

    // Cache successful response for this intent
    p.cache.Set(req.CurrentIntent, req.TenantID, resp, 60*time.Second)
    return resp, nil
}

WebSocket Delivery
Maintaining persistent connections to 10K+ agents and pushing suggestions with <20ms delivery latency.

Connection Model

  • One WebSocket per agent session: Agent's browser opens WS on call start, closes on call end
  • Server capacity: Each Go server handles ~5K concurrent WebSocket connections (goroutine-per-connection)
  • Horizontal scaling: 2-3 WebSocket servers behind a load balancer for 10K sessions
  • Sticky sessions: Load balancer uses session_id cookie for connection affinity

}

// When NLP pipeline needs to push a suggestion:
func (gw *WSGateway) DeliverSuggestion(sessionID string, msg []byte) error {
    // Look up which server holds this connection
    serverID, err := gw.redis.Get(ctx, fmt.Sprintf("ws:session:%s", sessionID)).Result()
    if err != nil {
        // Connection not found — queue for replay on reconnect
        return gw.queueForReplay(sessionID, msg)
    }

    if serverID == gw.selfID {
        // Local delivery — direct to goroutine
        return gw.localDeliver(sessionID, msg)
    }
    // Remote delivery — forward via internal pub/sub
    return gw.internalBus.Publish(serverID, sessionID, msg)
}

Fan-Out: Multiple Viewers

FAN-OUT PATTERN

One session might have multiple viewers: the agent, a supervisor monitoring the call, and a QA reviewer. Each viewer has their own WebSocket connection but subscribes to the same session's events. Use a pub/sub pattern internally.

// Session subscription model
type SessionSubscribers struct {
    SessionID   string
    Connections map[string]*WSConn  // connID -> connection
    // Roles determine what messages each viewer receives
    Roles       map[string]string   // connID -> "agent" | "supervisor" | "qa"
}

// Supervisor sees everything; agent sees suggestions only; QA sees transcript only
func (ss *SessionSubscribers) Broadcast(msg Message) {
    for connID, conn := range ss.Connections {
        role := ss.Roles[connID]
        if shouldReceive(role, msg.Type) {
            conn.Send(msg)
        }
    }
}

Graceful Degradation

DISCONNECTION HANDLING

Network blips happen constantly in call centers. If you drop suggestions during a 3-second disconnect, the agent misses critical guidance. Queue messages and replay on reconnect.

  • Message queue per session: On disconnect, buffer up to 50 messages in Redis list
  • Replay on reconnect: Client sends last-received message ID; server replays everything after it
  • Staleness check: Don't replay suggestions older than 30 seconds (conversation has moved on)
  • Supervisor fallback: If agent is disconnected >10s, escalate to supervisor dashboard
// Reconnection with message replay
func (gw *WSGateway) HandleReconnect(sessionID, lastMsgID string) {
    // 1. Restore connection mapping
    gw.RegisterConnection(sessionID, gw.selfID)

    // 2. Get queued messages since disconnect
    msgs, _ := gw.redis.LRange(ctx,
        fmt.Sprintf("ws:queue:%s", sessionID), 0, -1).Result()

    // 3. Filter: only replay messages after lastMsgID, within 30s
    cutoff := time.Now().Add(-30 * time.Second)
    for _, raw := range msgs {
        msg := parseMessage(raw)
        if msg.ID > lastMsgID && msg.Timestamp.After(cutoff) {
            gw.localDeliver(sessionID, []byte(raw))
        }
    }

    // 4. Clear the queue
    gw.redis.Del(ctx, fmt.Sprintf("ws:queue:%s", sessionID))
}
"WebSocket delivery is the last 20ms of our pipeline. Each Go server handles 5K connections using goroutines. I maintain a Redis mapping of session-to-server so any pipeline component can route messages to the right WebSocket server. For resilience, disconnected sessions get messages queued in Redis and replayed on reconnect — but I discard suggestions older than 30 seconds since the conversation context has moved on. Fan-out supports multiple viewers per session with role-based message filtering."
