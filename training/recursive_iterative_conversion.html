When Should You Convert?

Decision Tree

Interviewer asks: "Can you do it iteratively?" ├─ Is it a tree traversal? │ ├─ YES → Use standard iterative template (preorder/inorder/postorder) │ └─ NO → Is it a DFS on graph/grid? │ ├─ YES → Replace call stack with explicit Stack<int[]> holding state │ └─ NO → Is it DP with memoization? │ ├─ YES → Can you define a topological fill order? │ │ ├─ YES → Convert to bottom-up table │ │ └─ NO → Keep top-down (tree DP, sparse states) │ └─ NO → Mechanical stack simulation (see General Technique)

SituationKeep Recursive Because...
Tree DP (dp on children)Bottom-up requires explicit topo-sort of tree; recursive DFS IS the natural order
Memoization with heavy pruningTop-down only visits reachable states; bottom-up fills entire table wastefully

Mechanical Stack Simulation

Any recursion can be converted to iteration by simulating the call stack. Each "frame" on our explicit stack stores the same state the recursive call would have.

Template: Simulating Any Recursion

// RECURSIVE version
void solve(int node, int parent) {
    // PRE-ORDER work
    preWork(node);
    for (int child : adj[node]) {
        if (child != parent) {
            solve(child, node);
            // BETWEEN-CHILDREN work
            mergeChild(node, child);
        }
    }
    // POST-ORDER work
    postWork(node);
}

// ITERATIVE version — explicit stack of frames
void solveIterative(int root) {
    // Frame: [node, parent, childIndex]
    Deque<int[]> stack = new ArrayDeque<>();
    stack.push(new int[]{root, -1, 0});

    while (!stack.isEmpty()) {
        int[] frame = stack.peek();
        int node = frame[0], parent = frame[1], ci = frame[2];

        if (ci == 0) {
            preWork(node); // first visit
        }

        // Find next unvisited child
        List<Integer> children = adj[node];
        while (ci < children.size() && children.get(ci) == parent) ci++;

        if (ci < children.size()) {
            frame[2] = ci + 1; // advance child pointer
            stack.push(new int[]{children.get(ci), node, 0});
        } else {
            postWork(node); // all children done
            stack.pop();
            if (!stack.isEmpty()) {
                mergeChild(stack.peek()[0], node);
            }
        }
    }
}

Key insight: The child index ci in each frame is what "resumes" the function after returning from a child call. It's the iterative equivalent of the instruction pointer.

When This Pattern is Overkill

For simple DFS without post-order work, just use a basic stack:

// Simple DFS (pre-order only) — no need for frame simulation
Deque<Integer> stack = new ArrayDeque<>();
stack.push(start);
while (!stack.isEmpty()) {
    int node = stack.pop();
    if (visited[node]) continue;
    visited[node] = true;
    process(node);
    for (int next : adj[node]) stack.push(next);
}

DP: Top-down ↔ Bottom-up Conversion

The Core Idea

Top-down = recursive + memo. Bottom-up = iterative table-filling. They compute the same values, but differ in:

AspectTop-downBottom-up
States visitedOnly reachable statesAll states in table
Space optimizationNot possible (need full memo)Rolling array possible
Stack depthO(depth) — can overflowO(1) stack
Code clarityOften more naturalCan be tricky for complex recurrences
PruningNatural (don't call unreachable)Must manually skip

Conversion Recipe

  1. Identify state variables — what parameters does the recursive function take?
  2. Determine fill order — which states must be computed before which? (topological order of the dependency DAG)
  3. Allocate table — dp[state_var_1][state_var_2]...
  4. Set base cases — same as the recursion's base returns
  5. Fill in order — loop through states respecting dependencies
  6. Read answer — dp[original_call_params]

Example: Coin Change

// TOP-DOWN
int[] memo = new int[amount + 1];
Arrays.fill(memo, -1);
int solve(int remain) {
    if (remain == 0) return 0;
    if (remain < 0) return Integer.MAX_VALUE;
    if (memo[remain] != -1) return memo[remain];
    int res = Integer.MAX_VALUE;
    for (int c : coins)
        res = Math.min(res, solve(remain - c));
    return memo[remain] = (res == Integer.MAX_VALUE) ? res : res + 1;
}

// BOTTOM-UP (converted)
int[] dp = new int[amount + 1];
Arrays.fill(dp, Integer.MAX_VALUE);
dp[0] = 0;
for (int i = 1; i <= amount; i++)
    for (int c : coins)
        if (c <= i && dp[i - c] != Integer.MAX_VALUE)
            dp[i] = Math.min(dp[i], dp[i - c] + 1);

Fill order: increasing i (each state depends on smaller states).

Example: LCS (2D)

// TOP-DOWN
int lcs(int i, int j) {
    if (i == 0 || j == 0) return 0;
    if (memo[i][j] != -1) return memo[i][j];
    if (a[i-1] == b[j-1]) return memo[i][j] = lcs(i-1, j-1) + 1;
    return memo[i][j] = Math.max(lcs(i-1, j), lcs(i, j-1));
}

// BOTTOM-UP → enables space optimization to O(n)
int[] prev = new int[n + 1], curr = new int[n + 1];
for (int i = 1; i <= m; i++) {
    for (int j = 1; j <= n; j++)
        curr[j] = (a[i-1] == b[j-1]) ? prev[j-1] + 1 : Math.max(prev[j], curr[j-1]);
    int[] tmp = prev; prev = curr; curr = tmp;
    Arrays.fill(curr, 0);
}

Space win: Top-down needs O(m×n) memo. Bottom-up with rolling array: O(n). This is the #1 reason to convert.

Example: Knapsack 0/1 (Space Optimization)

// 2D bottom-up: dp[i][w] — O(n*W) space
// 1D optimization: iterate w BACKWARDS to avoid using updated values
int[] dp = new int[W + 1];
for (int i = 0; i < n; i++)
    for (int w = W; w >= weight[i]; w--)
        dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);

Trap: Unbounded knapsack iterates w FORWARD (reuse current item). 0/1 knapsack iterates BACKWARD (each item used once). Getting this wrong is a classic bug.

When Bottom-up is WORSE

Case 1: Sparse Reachable States

// Top-down visits only reachable states from (0, 0, initialMask)
// State space: dp[n][m][2^k] — but only ~1% of states are reachable
// Top-down with HashMap: visits O(reachable) states
// Bottom-up: allocates and fills ALL n*m*2^k entries → TLE
Map<Long, Integer> memo = new HashMap<>();
int solve(int i, int j, int mask) {
    long key = (((long)i * m) + j) * (1 << k) + mask;
    if (memo.containsKey(key)) return memo.get(key);
    // ... compute
}

Case 2: Top-down with Pruning

// Branch-and-bound style: if current cost already exceeds best known, prune
int solve(int state, int cost) {
    if (cost >= bestKnown) return INF; // PRUNE — bottom-up can't do this
    // ...
}

Case 3: Tree DP — Natural Recursion

// Tree DP: children must be computed before parent
// Recursive DFS naturally processes in correct order
// Bottom-up requires: (1) find leaves (2) BFS inward (3) awkward iteration
int treeDp(int v, int parent) {
    int res = 0;
    for (int u : adj[v]) if (u != parent)
        res += treeDp(u, v); // children first — the recursion IS the topo order
    return dp[v] = f(res);
}

Don't convert tree DP to bottom-up. The recursive DFS is already the correct traversal order. Converting adds complexity for zero benefit.

Tree DFS → Iterative Conversion

Preorder (easiest)

// Push right first, then left → left is processed first (LIFO)
Deque<TreeNode> stack = new ArrayDeque<>();
stack.push(root);
while (!stack.isEmpty()) {
    TreeNode node = stack.pop();
    result.add(node.val); // process
    if (node.right != null) stack.push(node.right);
    if (node.left != null) stack.push(node.left);
}

Inorder (go-left pattern)

// Go as far left as possible, then process, then go right
Deque<TreeNode> stack = new ArrayDeque<>();
TreeNode curr = root;
while (curr != null || !stack.isEmpty()) {
    while (curr != null) { stack.push(curr); curr = curr.left; }
    curr = stack.pop();
    result.add(curr.val); // process
    curr = curr.right;
}

Postorder (two approaches)

Approach 1: Modified preorder + reverse

// Preorder but visit right before left → reverse gives postorder
Deque<TreeNode> stack = new ArrayDeque<>();
List<Integer> result = new ArrayList<>();
stack.push(root);
while (!stack.isEmpty()) {
    TreeNode node = stack.pop();
    result.add(node.val);
    if (node.left != null) stack.push(node.left); // left first this time
    if (node.right != null) stack.push(node.right);
}
Collections.reverse(result);

Approach 2: True iterative (track last visited)

Deque<TreeNode> stack = new ArrayDeque<>();
TreeNode curr = root, lastVisited = null;
while (curr != null || !stack.isEmpty()) {
    while (curr != null) { stack.push(curr); curr = curr.left; }
    TreeNode peek = stack.peek();
    if (peek.right != null && peek.right != lastVisited) {
        curr = peek.right;
    } else {
        result.add(peek.val);
        lastVisited = stack.pop();
    }
}

Morris Traversal — O(1) Space

// Inorder traversal without stack or recursion
TreeNode curr = root;
while (curr != null) {
    if (curr.left == null) {
        result.add(curr.val);
        curr = curr.right;
    } else {
        // Find inorder predecessor
        TreeNode pred = curr.left;
        while (pred.right != null && pred.right != curr) pred = pred.right;
        if (pred.right == null) {
            pred.right = curr; // create thread
            curr = curr.left;
        } else {
            pred.right = null; // restore
            result.add(curr.val);
            curr = curr.right;
        }
    }
}

Morris: Temporarily modifies the tree (threading), so only use when you're allowed to modify (or restore). Doesn't work on immutable trees.

Generic DFS with Post-order Work

When you need to compute something AFTER all children are processed (subtree size, tree DP):

// Two-pass approach: push with flag
Deque<int[]> stack = new ArrayDeque<>();
stack.push(new int[]{root, 0}); // [node, processed]
while (!stack.isEmpty()) {
    int[] top = stack.peek();
    if (top[1] == 0) {
        top[1] = 1; // mark as "children pushed"
        for (int child : adj[top[0]]) if (child != parent[top[0]])
            stack.push(new int[]{child, 0});
    } else {
        stack.pop();
        // POST-ORDER: all children already processed
        for (int child : adj[top[0]])
            dp[top[0]] += dp[child]; // merge children results
    }
}

Complexity Traps When Converting

Trap 1: Bottom-up fills unreachable states

Example: Bitmask DP with dp[mask][last] where mask ranges over 2^n subsets but only O(n!) permutation-like paths are reachable from the start. Top-down with memo: O(n! × n). Bottom-up: O(2^n × n × n) — exponentially more states computed.

Rule: If reachable states << total state space, keep top-down.

Trap 2: Bottom-up loses pruning power

Example: TSP with branch-and-bound. Top-down prunes entire subtrees when partial cost exceeds best known. Bottom-up must compute ALL dp[mask][v] entries regardless — no early termination. Can be 10-100x slower in practice.

Trap 3: Wrong fill order causes O(N) blowup

// Interval DP: dp[i][j] depends on dp[i][k] and dp[k+1][j] for i ≤ k < j
// WRONG: filling row by row → dp[i][k] might not be ready
// RIGHT: filling by LENGTH (j - i)
for (int len = 2; len <= n; len++)       // ← by length!
    for (int i = 0; i + len - 1 < n; i++) {
        int j = i + len - 1;
        for (int k = i; k < j; k++)
            dp[i][j] = Math.min(dp[i][j], dp[i][k] + dp[k+1][j] + cost(i, j));
    }

Trap 4: Space optimization breaks path reconstruction

Problem: After rolling array optimization (2D→1D), you can no longer backtrack through the DP table to reconstruct the actual solution. If you need the solution (not just the optimal value), either:

  1. Keep the full table (no space optimization), OR
  2. Use Hirschberg's trick (divide-and-conquer on the DP) for O(n) space + reconstruction

Trap 5: Digit DP — keep top-down

Digit DP has states (position, tight, leading_zero, ...) where the "tight" constraint means many states are unreachable. Bottom-up requires iterating all combinations of these flags, most of which are invalid. Top-down naturally only visits valid states.

Google L5 Interview Tips

What they actually ask: "Can you convert this to iterative?" usually means one of:

  1. Tree traversal — know all 3 orders iteratively cold. Morris is bonus points.
  2. DFS on graph — simple stack replacement. They want to see you handle it smoothly.
  3. DP space optimization — "Can you do this in O(n) space?" = rolling array.
  4. Avoid StackOverflow — "What if n is 10^6?" = you must go iterative.

They almost NEVER ask you to convert tree DP or complex backtracking to iterative. If the recursion is naturally post-order, saying "I'd keep this recursive because the DFS order IS the topological order" shows judgment.
